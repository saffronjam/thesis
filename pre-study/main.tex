\section{Background}
Virtual machines (VMs) have long been the core part of IT infrastructure, allowing the underlying hardware to be split into multiple virtual computing units \cite{Li2023} . However, while VMs enables robust isolation, they also introduce overhead that could hinder performance of the application running inside of it. A popular alternative to VMs is containers, since they reduce the required overhead by sharing the underlying system. This, of course, by sacrificing many of the isolation properties. While there are some architectures that aims to unify the benefits of both VMs and containers, such as gVisor and Firecracker, it remains a fact that both containers and VMs have their individual use-cases. For this reason, VMs and containers have traditionally been managed separately; VMs in platforms such as OpenStack and OpenNebula, and containers using a platform such as Kubernetes. 

% Todo:
% - remove "of course" in the above paragraph

However, with the advent of container orchestration platforms like Kubernetes, there has been an ongoing convergence in how VM and containers are managed \cite{Li2023}. Specifically by integrating VM management in a container-centric environment. KubeVirt, an extension to Kubernetes, has thus emerged as a solution by enabling the running of VMs inside a Kubernetes environment alongside the containers creating a common environment \cite{kubevirt}. This approach, offers benefits such as reduced architectural complexity (as only Kubernetes is needed) and enhanced integration with the Kubernetes ecosystem \cite{kubevirt-cncf}. By managing the VMs they same way as any other Kubernetes resource, they benefit from the flexible and scalable Kubernetes infrastructure. 

Additionally, KubeVirt's benefits can be derived from the architectural changes that can be made when moving VMs into Kubernetes \cite{free-turtles}. Kubernetes nodes are commonly deployed in VMs to improve the flexibility of a cluster. However, with KubeVirt enabling running VMs inside Kubernetes, it opens the possibility of running Kubernetes directly on the bare metal servers. This approach could reduce overhead as it reduces nested virtualization, and thus could allow Kubernetes to function as a traditional VM management system.

\section{Literature Study}
In the domain of evaluating cloud computing alternatives, numerous studies have assessed the performance and usability of different cloud computing systems. As this project aims to evaluate Kubernetes with KubeVirt as a VM management system, it is important to consider what research has already been done in the field of comparing cloud alternatives. Additionally, since KubeVirt is a relatively new technology, it is important to consider research that evaluates KubeVirt itself and its practicality in a real-world scenario.

% PERFORMANCE
Regarding performance, several studies have evaluated popular systems that can manage VMs, such as OpenStack and CloudStack. A notable comparison between OpenStack and CloudStack was conducted by Paradowski et al. in 2014 \cite{cs-os-benchmark}. The comparison focused on performance and used several benchmarks such as deployment time, deletion time, CPU utilization during deployment and deletion, and revealed significant and quantifiable differences between the two platforms. However, the scale of the tests used in the comparison were relatively small, and did not explore how the systems manages VMs under higher load. In contrast, a 2022 study by Amaral M. benchmarked KubeVirt using a setup with a large amount of VMs relatively to the size of the nodes \cite{ibm-kubevirt-scale}, aiming to explore the limits of the KubeVirt's control plane (scheduler etc.). The findings highlighted a possibly large overhead when KubeVirt provisions a large amount of VMs. Additionally, in 2019 Kolahdouzan A. compared KubeVirt with OpenStack in terms of network traffic flow and network performance \cite{openstack-kubevirt-network}, identifying a large overhead in OpenStack for inter-node communication. The strategies used in to measure control-plane overhead and network communication overhead provide a foundational basis for when evaluating KubeVirt's performance.

% USABILITY
Sonninen O. investigates many of the possible benefits using a declarative approach in system architecture \cite{declarative-benefits}. A declarative approach is a paradigm that promotes usage of documents that describe a desired state of a system, such as with Kubernetes. The study explains how a declarative approach facilitates technologies such as GitOps, a way to manage infrastructure using files in Git, which enables simpler environment management, efficient resource management, easier rollbacks. Using Git as a source of truth for the desired state of the system has benefits, such as version control, audit and collaboration. The findings in this study are relevant to this project as Kubernetes, and thus KubeVirt, uses a declarative approach.

Trakulmaiphol S. and Piromsopa K. present some shortcomings when migrating from a traditional VM management system, such as CloudStack and OpenStack, to KubeVirt \cite{kubevirt-migrate-issues}. One of the issues identified is the management of of IP addresses, particularly when maintaining static IP addresses in a dynamic IP environment, such as Kubernetes. In the study they developed a framework to solve some of the issues on application level, such as DNS queries, and provided a set of guidelines related to KubeVirt's  unique features, such as the Multus Container Network Interface. These findings are especially relevant to this project as they highlight shortcomings that needs to be addressed when migrating a VM to KubeVirt.

\section{Methods}
\subsection{Performance Analysis}
As previously described, Paradowski A. presents a set of benchmarks used to evaluate CloudStack and OpenStack \cite{cs-os-benchmark} (This study will be referred to the \textit{the IaaS comparison} in the following paragraphs). To extend the comparison, KubeVirt will be compared to more than one IaaS platform: OpenStack, CloudStack, and OpenNebula. This systems were chosen as they are popular open-source IaaS platforms. 

The testing process will be conducted within Azure using a group of VMs that will act as \textit{hosting nodes}. This approach is chosen to ensure that the tests are not affected by the underlying hardware, and to facilitate reproducibility. A set of scripts will be created to automate the process of installing and configuring each VM management system, and to run the tests that are summarized in a table below. As the previously mentioned comparison suggests, each test case will be run 10 times to ensure that the results are consistent. After the tests are finished, all the VMs will be deleted. 

Each test case will use a setup of three nodes, one acting as the management node, and the two other acting as worker nodes. Having more than one worker node is crucial to be able to test live migration, as the tests will require the VMs to be moved between nodes. The nodes will configured to be large enough to facilitate broader range of tests, including tests with many small VMs and tests with a few large VMs. For this, Azure A4 VMs will be used, as they have 8 CPU cores and 14GB of RAM.

In the study by Paradowski, they used a number of preconfigured VM types in the tests, including \textit{tiny}, \textit{small}, \textit{medium}, and \textit{large}. As such, this project will use a similar approach. Furthermore, VMs will use the lightweight Alpine Linux image, as lightweight OS image is less likely to introduce noise in measurements \cite{linux-boot-time}. For instance, a larger OS image could introduce a larger overhead by requiring installing more packages that requires fetching more data from external sources, which is partly unrelated to the system's performance. Another reason to use a lightweight image is to reduces the time it takes to run the tests, as a lightweight image is usually faster to boot.

Below is a summary of the metrics that will be used to evaluate the performance of the VM management systems, and a list of the tests that will be performed. The metrics will be used for the affected node in the test, for instance, when live-migrating it is important to measure both the source and the destination node, while when a VM is created, it is only necessary to measure the node where the VM is created. However, the control-plane node will be measured in all tests, as it is the node that is responsible for scheduling and managing the VMs. 

\begin{itemize}
    \item Metrics
    \begin{itemize}
        \item CPU usage
        \item Memory usage
        \item Disk usage
        \item Duration
    \end{itemize}
\end{itemize}

Below is a summary of the tests that will be performed. They can be divided into three categories: basic, limit testing, and scaling. The basic tests are designed to test the basic functionality of the VM management systems, such as creating and deleting VMs. The limit testing tests are designed to evaluate how a VM management system handles a VMs when the maximum capacity of a worker node is reached. The scaling tests are designed to evaluate how a VM management system handles a scale-out and scale-in of worker nodes, both with and without VMs on it (that would need to be migrated).

\begin{itemize}
    \item Tests
    \begin{itemize}
        \item Create each type of VM
        \item Delete VM
        \item Create VM Snapshot
        \item Delete VM Snapshot
        \item Live Migrate VM
        \item Creating 10 Small VMs
        \item Creating 10 Large VMs
        \item Scale from 2 to 10 workers without VMs
        \item Scale down 10 to 2 workers without VMs
        \item Scale from 2 to 10 workers with VMs
        \item Scale down 10 to 2 workers with VMs
    \end{itemize}
\end{itemize}

\subsection{Usability Analysis}
To accurately measure the usability of KubeVirt, a set of tasks will be crafted. These tasks should be designed to be representative of the tasks that a system administrator would perform when managing VMs, including creation, deleting, and debugging. Crafting the tasks will be done after exploring KubeVirt and other VM management systems to understand their feature sets and what kind of tasks are relevant. However, the tasks will only performed on KubeVirt, since the system administrators are already expected to be familiar with the other systems.
\\~\\
Some examples of tasks that could be included are:
\begin{itemize}
    \item Create a VM
    \item Start a VM
    \item Delete a VM
    \item Stop a VM
    \item A VM is not starting, debug why
\end{itemize}

\subsection{System Management Analysis}
The last factor that will be considered in this project is the system management. This include exploring how the system is managed, and is mainly concerned with integrations with other technologies in order to understand how well KubeVirt integrates with other systems. This will be done by exploring feature sets and architectural principles of KubeVirt, and other VM management systems. For instance, depending if a system uses a declarative or imperative approach, some technologies or tools could be easier to integrate than others. A common example is that a declarative approach is easier to integrate with GitOps \cite{k8s-gitops}.
