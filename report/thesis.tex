\begin{abstract}
    \Glspl{vm} and containers both offer a computing environment for a wide range of workloads, providing their respective benefits such as strong isolation or being lightweight. However, they have traditionally been managed in separate management environments which has led to a fragmented infrastructure for organizations that use both \Glspl{vm} and containers. KubeVirt was created to unify the management of \Glspl{vm} and containers by integrating \Glspl{vm} into Kubernetes, a widely adopted platform originally designed for container orchestration. This project aims to evaluate Kubernetes with KubeVirt as a \Gls{vm} management system both by comparing it with traditional approaches and by investigating the implications of moving into the Kubernetes ecosystem. The project uses a mixed-methods approach including a literature study, a qualitative comparison in the form of a usability analysis and a quantitative comparison in the form of a performance analysis. The results show that managing \Glspl{vm} in Kubernetes using KubeVirt comes with a steeper learning curve than traditional approaches and that there is some additional overhead. However, the results also show that KubeVirt can provide performance and operational efficiencies that are not possible with traditional approaches, and as such, represent a more scalable and flexible approach to \Gls{vm} management.
\end{abstract}

\chapter{Introduction}
Virtualization is a technique that either splits or combines computing resources into one or more operating environments by creating a software layer that surrounds or underlies existing hardware or software. This enables the creation of virtualized resources that are decoupled from the underlying hardware. Virtualization can occur in various forms including the virtualization of storage, networks and servers. Virtualizing networks and storage is achieved by creating an abstraction layer on top of the physical network and storage resources, respectively, to provide a more flexible and scalable environment for other resources to interact with. Server virtualization, on the other hand, is used to create virtual computing environments that are decoupled from the underlying computing resources. By combining virtualized resources into a single operating environment, virtualization allows heterogeneous hardware resources to be used homogeneously and thus it improves resource utilization and flexibility\ \cite{virtualization-history, virtualization-basics, virtualization-trends}.

To enable server virtualization and manage virtual computing environments, a hypervisor is used. A hypervisor is a software layer that operates between the hardware and the \Gls{os} and is responsible for creating virtual computing environments called virtual machines (\Glspl{vm})\ \cite{virtualization-trends, virtualization-ibm, hypervisor-survey}. \Glspl{vm} are emulations of physical computers that are perceived as real computers by applications running inside them. They provide a complete and independent environment by virtualizing an \Gls{os}, rendering them suitable for a wide range of applications, including legacy applications and applications that require a high level of isolation. For this reason, \Glspl{vm} have long been the core part of IT infrastructure, serving as foundational components in system architecture\ \cite{virtualization-history, virtualization-basics}. 

In parallel with the evolution of \Glspl{vm}, containers have emerged as an alternative to \Glspl{vm}. Container virtualization is an \Gls{os}-level virtualization technique that allows for running multiple isolated applications on a single \Gls{os} instance. This means container virtualization does not require a hypervisor, and instead uses the features and capabilities of the \Gls{os} kernel to create and manage containers. As such, containers are more lightweight, require less overhead, and have faster startup times than \Glspl{vm}\ \cite{container-vm-overview}. This has made containers a popular choice for scenarios in which scaling and portability are key, such as in microservices, which are core components in modern cloud-native applications\ \cite{cloud-native}. Figure\ \ref{fig:container-vm-architecture} visualizes the differences between how \Glspl{vm} and containers are virtualized, and how the virtualization layer is placed on a lower level for \Glspl{vm} than for containers. 

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{img/vm_container_virtualization.pdf}
    \caption{Overview of \Glspl{vm} and containers. The virtualization layer is placed on a lower level for \Glspl{vm} than for containers, illustrating the difference in their respective virtualization techniques.}
    \label{fig:container-vm-architecture}
\end{figure}

However, while containers provide a lightweight and efficient alternative to \Glspl{vm}, they lack the deep isolation properties inherent to \Glspl{vm}. Sharing the \Gls{os} kernel with the host and other containers opens up a larger attack surface, which implies that containers are more prone to security vulnerabilities. While there exist technologies that aim to unify the isolation benefits of \Glspl{vm} and the lightweight benefits of containers, such as Kata Containers, gVisor and Firecracker, it remains a fact that both containers and \Glspl{vm} have their use cases\ \cite{container-runtimes-overview, container-as-vm-comparison}. 

Due to their different use cases, \Glspl{vm} and containers have traditionally been managed separately. \Glspl{vm} are often managed by a \Gls{vm} management system that provides a set of tools and processes to aid in the life-cycle management of \Glspl{vm}. This makes a \Gls{vm} management system act as a higher-level abstraction layer that unifies hypervisors, typically through an \Gls{api} such as a command line interface (\Gls{cli}) or a web-based user interface. Figure \ref{fig:vm-management-system} depicts a broad overview of the role of a \Gls{vm} management system where clients interact with an interface to manage \Glspl{vm} that are distributed across several hosts. Moreover, a \Gls{vm} management system is often part of a larger infrastructure, such as Infrastructure as a Service (\Gls{iaas}), that provides a wider range of virtualized resources including virtual networks and storage. Examples of such systems include OpenStack\ \cite{openstack-website}, CloudStack\ \cite{cloudstack-website}, and OpenNebula\ \cite{opennebula-website}. Containers, on the other hand, are often managed by a container orchestration platform. These platforms provide a similar set of tools and processes as a \Gls{vm} management system but for containers. Apart from creating and deploying containers, a container orchestration platform provides tools specifically designed to manage the life cycle of containers, such as scheduling and scaling. Examples of such platforms include Kubernetes\ \cite{k8s-website}, Nomad\ \cite{nomad-website}, and Docker Swarm\ \cite{docker-swarm-website}, with the former being the most widely adopted \cite{k8s-it-administration}. Furthermore, with the evolution and adaption of container orchestration platforms like Kubernetes, there has been an ongoing convergence in how \Glspl{vm} and containers are managed\ \cite{vm-cloud}. 

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{img/vm_management_system.pdf}
    \caption{Brief overview of a monolithic \Gls{vm} management system where clients communicate with a management node through a set of interfaces to manage \Glspl{vm} distributed across several hosts. The indirect communication allows a client to manage \Glspl{vm} without having to interact with the hosts directly.} 
    \label{fig:vm-management-system}
\end{figure}

Kubernetes, originally designed to manage containerized applications at scale, has recently been extended to accommodate \Glspl{vm} through the KubeVirt project. KubeVirt is an \Gls{api} extension for Kubernetes, that enables the running of \Glspl{vm} inside a Kubernetes environment alongside containers, creating a common environment for both virtualization technologies\ \cite{kubevirt}. Figure \ref{fig:container-vm-kubevirt-architecture} depicts a high-level overview of how \Glspl{vm} and containers coincide in a Kubernetes environment using KubeVirt, where traffic flows natively between the two types of resources. Further technical details of how KubeVirt achieves this will be discussed in \ref{sec:kubevirt-technical-overview}. 

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{img/container_vm_with_kubevirt.pdf}

    \label{fig:container-vm-kubevirt-architecture}
\end{figure}

The approach of incorporating \Glspl{vm} in Kubernetes not only benefits from the flexible, scalable and mature infrastructure that Kubernetes provides but also from tools and processes that are designed to integrate with or manage the life cycle of Kubernetes resources. This means by using KubeVirt, \Glspl{vm} can be managed similarly to containers, creating a more cloud-native approach to \Gls{vm} management\ \cite{cloud-native}. Furthermore, consolidating the management of \Glspl{vm} and containers into a single platform could offer benefits in terms of reduced architectural complexity and simpler hardware management, but also practical aspects such as reduced required skill set in an organization\ \cite{kubevirt-cncf, kubevirt-storware}. 

The integration of \Gls{vm} management into Kubernetes through KubeVirt represents a significant milestone toward a more agile, efficient, and unified IT infrastructure\ \cite{kubevirt-storware}. However, KubeVirt's practicality in a real-world scenario is yet to be fully assessed. As such, this project aims to evaluate Kubernetes with KubeVirt as a \Gls{vm} management system and compare it with traditional approaches to understand its practicality and limitations.

Due to the different use cases for \Glspl{vm} and containers, they have traditionally been managed separately. However, both \Glspl{vm} and containers are still part of the necessary infrastructure many organizations depend on. This has led to fragmented infrastructure with separate management systems and tools to manage \Glspl{vm} and containers with a direct implication of increased architectural complexity and a larger required skill set for operators. The purpose of this project is to evaluate if Kubernetes, a platform originally designed for container orchestration, is a suitable platform for managing \Glspl{vm} using KubeVirt. \section{Virtualization Fundamentals}
\label{virtualization}
Virtualization is a technology that enables partitioning or grouping computing resources into operating environments using methods such as partitioning, emulation, and time-sharing\ \cite{virtualization-basics}. Virtualization can occur in various forms including the virtualization of storage, networks and servers. Storage virtualization is achieved by creating an abstraction layer on top of physical storage resources to provide a single interface to storage resources that appear as a single storage device. This allows for a more flexible environment where storage resources can be added or removed without affecting the application layer. Similarly, network virtualization enables the creation of networks with the properties of a physical network, but with its appliances such as routers and switches being implemented in software. This allows for flexibility when managing resources in a network since common practices such as networking partitioning and isolation can be achieved without acquiring additional hardware. Server virtualization, on the other hand, is concerned with creating virtual computing environments that are decoupled from the underlying computing resources. It is often referred to as hypervisor-based virtualization, as it requires a software layer called a hypervisor that is responsible for creating and managing the virtual computing environments \ \cite{virtualization-history}.

A virtual computing environment, more often referred to as a virtual machine (\Gls{vm}), is an emulation of a physical computer and is perceived as a \textit{real} computer by the application running inside it even though the underlying resources are often shared with other \Glspl{vm}. To enable this, a hypervisor divides the hardware into virtualized resources, such as virtual CPUs and virtual memory, and provides an interface to manage these resources. For instance, an application running inside a \Gls{vm} is unaware that the resources it is using are virtualized and instead interacts with the \Gls{os} and hardware as if it were running on a physical machine. This is particularly useful for scenarios in which hardware is under-utilized, as it allows for the consolidation of workloads onto a smaller number of physical machines, which in turn reduces the need for acquiring additional hardware.\ \cite{virtualization-basics, virtualization-trends}.

Depending on the required level of abstraction, virtualization can be achieved in different layers, where the three major being hardware-level, \Gls{os}-level and application-level. For instance, storage and network virtualization both fall under the category of hardware virtualization, as they create an abstraction layer on top of the physical resources. However, in other scenarios, virtualization is not required to be as low-level, such as in application virtualization. Application virtualization is instead concerned with encapsulating an application in a lightweight virtualized environment, such that it is perceived as being run on a specific platform. A common example is the Java Virtual Machine (J\Gls{vm}) that enables programs written in the programming language Java to be run on multiple operating systems\ \cite{application-virtualization}. Figure \ref{fig:hardware-os-virtualization} presents an overview of the differences between the two more common layers of virtualization; hardware and \Gls{os} virtualization. While the figure does not include application-level virtualization, it is important to note that the virtualization layers are not mutually exclusive and that they can be used in combination. For instance, combining hardware-level and \Gls{os}-level virtualization creates a flexible environment where a guest process can run inside a guest \Gls{os}\ \cite{virtualization-basics}.

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/hardware_os_virtualization.pdf}
    \caption{Overview of the different layers of virtualization for hardware-level and \Gls{os}-level virtualization. The figure illustrates how hardware virtualization inhibits several internal \Glspl{os} from running on the same hardware, while \Gls{os} virtualization keeps a single \Gls{os} for each process.}
    \label{fig:hardware-os-virtualization}
\end{figure}

Furthermore, virtualization can be divided into two categories, full virtualization and paravirtualization. Full virtualization is the traditional approach where the entire set of hardware is emulated, and the \Gls{vm} is presented with an indistinguishable image of the underlying hardware. This allows for complete decoupling of the \Gls{vm} from the underlying resources and does not require any modification to the guest \Gls{os}\ \cite{virtualization-basics, xen-virtualization, hypervisor-xen}. Paravirtualization, on the other hand, is a subset of full virtualization that aims to improve \Gls{vm} performance by reducing the number of resources that are emulated. The \Gls{vm} is presented with an image that is similar, but not identical, to the underlying hardware. As such, the guest \Gls{os} is aware it is running inside a \Gls{vm}\ \cite{xen-virtualization, hypervisor-xen}.

\subsection*{Benefits and Challenges}
Many of the benefits of virtualization stem from the fact that there exists an abstraction layer above the physical hardware. The abstraction layer decouples software from hardware, allowing for the use of vendors with different I/O in the same environment. This creates a more flexible and scalable computing environment as a whole where hardware is turned into a simple pool of computing resources\ \cite{virtualization-trends}. As such, there are numerous benefits of virtualization, a few of which are listed below:
\begin{itemize}
    \item Resource utilization: Virtualization allows for better utilization of hardware resources since multiple \Glspl{vm} can run on the same physical hardware. This is achieved by sharing the underlying resources between the \Glspl{vm}, such as CPU and memory\ \cite{virtualization-basics}. 
    \item Isolation: \Glspl{vm} can isolate their entire state from one another, including the \Gls{os}, filesystem and application stack. This is particularly useful for certain applications with stricter security requirements, such as applications managing sensitive or confidential data. Isolation is also beneficial in scenarios where reproducibility is key, such as in testing and development environments\ \cite{vm-docker-reproducibility}.
    \item Flexibility: \Glspl{vm} are not tied to specific hardware, and can be moved between different physical resources. Flexibility is an especially useful factor when in scenarios where hardware is not fully utilized since \Glspl{vm} can be migrated to other physical machines to balance the load\ \cite{virtualization-basics}.
    \item Management: \Glspl{vm} can be created and destroyed on demand, and can be managed using software tools\ \cite{virtualization-basics}.
    \item Virtual hardware: A \Gls{vm} can be configured with hardware that does not exist in the physical environment, such as a virtual network card or a virtual hard drive. This allows for better support for legacy applications or certain applications that are dependent on a specific hardware configuration\ \cite{virtualization-basics}.
\end{itemize}

However, the portability made possible by virtualization comes at a cost of performance. Since a \Gls{vm} is a complete but emulated system, every instruction issued must be interpreted in software. This means that despite the level of abstraction layer used, there is an inescapable added overhead when using virtualization. A common hypervisor such as \Gls{kvm} (see\ \ref{hypervisors}) introduces an overhead of 5 to 10\%\ \cite{kvm-overhead}, and the overhead can be even larger when using a full virtualization approach.

\section{Understanding Hypervisors}
\label{hypervisors}
A hypervisor is a software layer that enables managing multiple \Glspl{vm} on a single physical machine by allowing multiple operating systems (called \textit{guests}) to share the same hardware\ \cite{virtualization-history, hypervisor-survey}. Hypervisors, often referred to as \Gls{vmm} are responsible for creating and managing \Glspl{vm} and for providing the necessary abstractions to the \Glspl{vm}.

Hypervisors can be categorized into two types, \textit{type 1} and \textit{type 2}, based on their operational context and how they interact with the underlying hardware.  While the goals and purpose of type 1 and type 2 hypervisors are the same, there are some core differences. Type 1 hypervisors, commonly referred to as \textit{bare-metal} hypervisors, run directly on top of the physical hardware and do not require an \Gls{os} underneath. A type 1 hypervisor is therefore scalable to terabytes of RAM and hundreds of CPU cores\ \cite{hypervisor-types-1-2}. However, without any other additional software, it is also responsible to schedule and allocate required system resources to the \Glspl{vm}. Type 2 hypervisors, also known as \textit{hosted hypervisors}, rely on an underlying \Gls{os} to manage the calls to the CPU and memory resources. The presence of an underlying \Gls{os} inevitably introduces latency, which is why type 2 hypervisors are often used for development and testing purposes\ \cite{hypervisor-survey}. Figure \ref{fig:hypervisor-types} presents a high-level overview of the difference between a type 1 and type 2 hypervisor\ \cite{hypervisor-survey, hypervisor-types-1-2}.

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/type1_type2_hypervisor.pdf}
    \caption{Overview of type 1 and type 2 hypervisors. The figure illustrates how a type 1 hypervisor does not use an underlying \Gls{os} and instead runs directly on the hardware, while a type 2 hypervisor relies on an intermediary \Gls{os} to manage the underlying hardware.}
    \label{fig:hypervisor-types}
\end{figure}

Furthermore, hypervisors can be further categorized into \textit{monolithic} and \textit{microkernelized} hypervisors\ \cite{hypervisor-comparion-2013}. Monolithic hypervisors are built as a single large software stack, managing all the \Glspl{vm} and their resources, and hosting driving for all hardware access. Having all drivers reside within the hypervisor means that the hypervisor will only run on selected hardware. However, it also means no operating system is required at all, as the hypervisor acts as an operating system by supporting all the operations of its \Glspl{vm}. Microkernelized hypervisors, on the other hand, do not require device drivers to be present in the hypervisor. Instead, the drivers are installed in a form of managing \Gls{vm}, often referred to as parent \Gls{vm}. Each \Gls{vm} created by such a hypervisor is thus referred to as a child \Gls{vm}, as it uses the parent \Gls{vm}'s drivers to access the hardware. This approach is more flexible, as it can allow the hypervisor to run on any hardware that supports the parent \Gls{vm}'s operating system. 

% security rings definition
In the realm of hypervisors, the concept of \textit{security rings} is often used to describe the level of privilege different part of a hypervisor has. A security ring represents a hierarchical protection domain, where the innermost ring (ring 0), usually reserved for the \Gls{os} kernel, has the highest privilege, and the outermost ring (ring 3), usually reserved for user applications, has the lowest privilege. The hypervisor itself is often placed in ring 0, its \Gls{vm} kernel space in ring 1, and the \Gls{vm} user space in ring 3. Security rings are thus used to define execution level to provide a system to limit the execution capabilities of \Glspl{vm} to regulate access to shared memory. This enabled a strong isolation layer between the \Glspl{vm}, and between the \Glspl{vm} and the hypervisor, thus a key part of a hypervisor's security model\ \cite{hypervisor-comparion-2013, microkernel-monolithic}. Hypervisor security rings are visualized in figure \ref{fig:hypervisor-security-rings}. 

\begin{figure}
    \centering
    \includegraphics[width=0.7\textwidth]{img/hypervisor_security_rings.pdf}
    \caption{Overview of security rings in a hypervisor. The figure illustrates how the inner rings have higher privilege than the outer rings.}
    \label{fig:hypervisor-security-rings}
\end{figure}

\subsection*{Key Hypervisors}
\label{key-hypervisors}
There are several hypervisors available, each with its own set of features and capabilities, some of which are listed below:

\begin{itemize}
    \item VMware ESXi: A lightweight type 1 monolithic hypervisor often used in enterprise environments. ESXi is known for its ultra-lightweight footprint stretching merely a few hundred megabytes, and its ability to run directly on the hardware without an underlying \Gls{os}\ \cite{hypervisor-comparion-2013}.
    \item Microsoft Hyper-V: A type 1 microkernelized hypervisor that is part of the Windows Server operating system. As a microkernelized hypervisor, Hyper-V differentiates between parent and child \Glspl{vm} and uses the parent \Gls{vm} to manage hardware access for the child \Glspl{vm}\ \cite{hypervisor-comparion-2013, hypervisor-hyper-v-esxi}.
    \item Citrix Hypervisor: Formerly known as XenServer, is a type 1 microkernelized hypervisor based on the Xen Project. It runs using a modified Linux kernel in a \Gls{vm}, called Dom0, that manages hardware I/O and interacts with the other \Glspl{vm}, called DomU guests\ \cite{hypervisor-xen, hypervisor-comparion-2013}. 
    \item \Gls{kvm}: A type 1 hypervisor that is part of the Linux kernel. As such, \Gls{kvm} uses most of the functionalities from the Linux kernel including the scheduler and memory management. \Gls{kvm} has become a popular alternative to proprietary hypervisors, such as VMware ESXi and Microsoft Hyper-V since it is open-source and free to use\ \cite{hypervisor-comparion-2013, hypervisor-kvm}. 
\end{itemize}

\section{VM Management Essentials}
A management system for \Glspl{vm} involves a set of tools and processes that are used to create, deploy, and manage \Glspl{vm}. The management system is responsible for managing the entire life cycle of a \Gls{vm}, including provisioning, deployment, monitoring, and scaling. Not be confused with a hypervisor, a \Gls{vm} management system operates on a higher level and aims to provide an abstraction layer in the form of an interface to allow interaction with the \Glspl{vm}, often enabling the support of a set of hypervisors\ \cite{cloudframeworks-and-hypervisors}. As such, a \Gls{vm} management system unifies a set of hypervisors by allowing them to be managed as if they were homogeneous, typically through a higher-level \Gls{api}, such as a \Gls{cli} or a web-based user interface. However, certain hypervisor features, such as live migration, could be implemented differently across hypervisors, and it is up to the \Gls{vm} management system to provide a unified interface to manage such features\ \cite{openstack-opennebula-comparison}.

\subsection*{Concepts and Features}
With a higher abstraction layer, a \Gls{vm} management system brings a set of concepts to simplify the management of \Glspl{vm}. Many such features and concepts exist and can be broadly grouped as clustering, availability, and scalability\ \cite{cloudframeworks-and-hypervisors}. 

Clustering involves grouping a set of resources, such as \Glspl{vm} or storage nodes, to provide an image that is perceived as a single resource. By allowing multiple nodes to cooperate and share resources, clustering can allow for a more scalable and performant system. \Gls{hpc} is a common example of clustering where computing resources are aggregated to gain performance greater than that of a single computer. This enables \Gls{hpc} clusters to process large datasets and complex calculations that would take an infeasible amount of time to process on a single computer. Moreover, \Gls{hpc} clusters are designed such that they allow for the distribution of computational tasks where larger tasks are divided into smaller subtasks. However, the splitting of tasks is often not transparent to the end user, and the user perceives the cluster as a single resource\ \cite{google-hpc-overview,oracle-hpc-overview}. 

Availability refers to the ability to keep a resource active or running, which is, more often than not, clustered resources. A common practice for cloud computing is managing availability zones (\Gls{az}s). An \Gls{az} groups resources such that each \Gls{az} is independent of the other \Gls{az}s while providing the same service. This allows for outages to be isolated to a single \Gls{az} while the other \Gls{az}s continue to provide the service. Thus, through redundancy and isolation, \Gls{az}s enable high availability and fault tolerance\ \cite{cloudframeworks-and-hypervisors, azure-availability-zones}. Figure\ \ref{fig:availability-zones} presents an overview of how 3 \Gls{az}s provide redundant access to the same service. 

Scalability involves technologies to ensure resources can manage an increase or decrease in computational loads. A \Gls{vm} management system often facilitates both scaling up and scaling out. Scaling up, also referred to as vertical scaling, involves increasing or decreasing the performance specification of a resource. For instance, a \Gls{vm} can be scaled up by increasing the number of CPUs or the amount of memory. Scaling out, also referred to as horizontal scaling, involves adding or removing entire resources, such as \Glspl{vm}. This means, that instead of upgrading a single \Gls{vm}, additional \Glspl{vm} are added to the system to handle the increased load. This approach is often regarded as more scalable than scaling up, as it allows for a more granular approach to resource management, and can be down without any downtime. Additionally, by avoiding downtime, the system appears as more available to the end user \cite{scale-up-out-case-study, scale-up-out-ibm}

\subsection{Comparison of VM Management Systems}
Managing \Glspl{vm} can be done in many ways, and there are numerous \Gls{vm} management systems available, each with its own set of features and capabilities. Many such systems are part of a larger platform, such as an \Gls{iaas} or even as large as a cloud platform. Some of the most prevalent open-source platforms that have been used in the industry are OpenStack, CloudStack, and OpenNebula, and some of the most popular public cloud platforms are Amazon EC2, Azure Virtual Machines, and Google Compute Engine.

\subsubsection*{OpenStack}
An open-source \Gls{iaas} platform that offers a large set of features to manage \Glspl{vm}, networks, and storage\ \cite{openstack-opennebula-comparison}. OpenStack's architecture is inherently modular, and it is designed to be deployed in a distributed manner\ \cite{openstack-overview}. The architecture includes three major parts:
\begin{itemize}
    \item \textit{Nova}: The compute service that manages \Glspl{vm} and provides an interface to manage \Glspl{vm}. Nova supports all operations regarding the life cycle of a \Gls{vm}, such as creating, deleting, and migrating \Glspl{vm}. Nova is itself divided into multiple components, each employing a \textit{shared-nothing} and \textit{messaging-based} policy, which means every component is independent and communicates with other components using a message queue. 
    \item \textit{Swift}: The object storage service that provides a scalable and redundant storage system for \Glspl{vm}. It is a long-term storage for static data, such as images and backups, and is comparable to Amazon S3.
    \item \textit{Glance}: The image service that provides a catalog of \Gls{vm} images that can be used to create new \Glspl{vm}. 
\end {itemize}

OpenStack supports features such as live migration, snapshot management, and scalability, and is often used in large-scale cloud environments\ \cite{openstack-opennebula-comparison}. However, these features are often hypervisor-specific and require both support from the hypervisor and specific configuration to work\ \cite{openstack-live-migration}.

\subsubsection*{CloudStack}
An open-source \Gls{iaas} platform and cloud computing software that manages computing, networking and storage resources, and is used to build and manage public and private cloud environments\ \cite{cs-overview}. On a high level, CloudStack is built in a monolithic fashion, encapsulating all the management features into two software packages called the management server and agent. The management server is responsible for managing the life cycle of \Glspl{vm}, including provisioning, deployment, and scaling, and is comparable to a control node. The agent, however, is installed on each node designated to running \Glspl{vm} and is responsible for low-level management tasks of the \Glspl{vm} on that node, such as interacting with the underlying hypervisor. While it is a monolithic system, CloudStack is designed to be horizontally scalable and can be deployed in a distributed manner with multiple management servers\ \cite{cs-overview}. 

CloudStack supports a large variety of hypervisors, such as VMware, \Gls{kvm}, and XenServer, and storage services, such as Amazon S3, allowing it to function as an abridgment layer for computing, network, and storage to unify a wide range of existing infrastructure\ \cite{cs-overview}. Features such as live migration, snapshot management, and scalability are supported but are often hypervisor-specific and require specific configuration to work\ \cite{cs-live-migration}.

\subsubsection*{OpenNebula}
An open-source \Gls{iaas} platform and toolkit used to provide open, scalable, and comprehensive management of virtualized data centers\ \cite{openstack-opennebula-comparison}. OpenStack employs a modular architecture divided into three layers\ \cite{openenbula-tele-teaching, iaas-comparisons}: 
\begin{itemize}
    \item \textit{Tools layer}: Provides a set of interfaces such as a \Gls{cli} and web-based user interface to interact with the system, but also high-level management tools such as scheduling.
    \item \textit{Core layer}: Provides the core functionalities of the system, such as host management, \Gls{vm} management, and virtual network management.
    \item \textit{Driver layer}: Provides an abstraction on top of the underlying infrastructure to enable the system to support a wide range of hardware, including networking, storage, and hypervisors. As such, the driver layer is responsible for allocation, deallocation and monitoring of resources, such as \Glspl{vm} and storage. 
\end{itemize}

By dividing the system into these layers, OpenNebula provides modularity that enables the system to be extended and customized to fit specific needs, such as integrating with other systems or using OpenNebula to build a private cloud.

\subsubsection*{Amazon EC2}
Amazon Elastic Compute Cloud (EC2) is a web service that provides on-demand compute resources over the cloud\ \cite{ec2-overview, ec2-overview-web}. EC2 is part of Amazon Web Services (AWS), is designed to be scalable and flexible, and is often used to host general-purpose applications, including web applications, databases and enterprise applications. While the EC2 platform uses Xen-based hypervisors (see\ \ref{key-hypervisors}) under the hood\ \cite{public-cloud-comparison}, it acts as a complete abstraction layer over the underlying hardware and hypervisors and instead provides a rich set of tools to manage the \Glspl{vm}.

This enables EC2 to provide a wide range of \Glspl{vm}, such as general-purpose, compute-optimized, memory-optimized, and storage-optimized \Glspl{vm}\ \cite{ec2-overview}. \subsubsection*{Azure Virtual Machines}
Azure Virtual Machines is a service provided by Azure to manage \Glspl{vm} in the cloud\ \cite{azure-overview}, and is designed to be a cost-effective alternative to on-premises hardware\ \cite{azure-cost-effective}. Azure Virtual Machine provides a set of preconfigured \Glspl{vm} that can be used to create \Glspl{vm} for specific purposes, such as general-purpose or \Gls{hpc}.

Behind the scenes Azure Virtual Machines runs on Hyper-V (see\ \ref{hypervisors}) and provides a set of features such as live migration, snapshot management, and auto-scaling\ \cite{azure-overview-web}.

Compute Engine uses \Gls{kvm} as its hypervisor (see\ \ref{hypervisors}), and provides many features such as live migration, snapshot management, and auto-scaling\ \cite{gce-overview-web, public-cloud-comparison}.

\section{Containers and Kubernetes: A Paradigm Shift}
\label{containers-kubernetes}
Besides the common cloud service models (\Gls{iaas}, \Gls{paas}, and \Gls{saas}), another service model has emerged, called \gls{caas} \cite{caas}. \gls{caas} is a container-based cloud service model that has gained significant traction due to the lightweight and portable nature of a container, allowing for simple application bundling that can run in a vast range of environments\ \cite{microservices-overview}. 

Containers are often used in conjunction with a container orchestration platform, such as Kubernetes, to manage the life cycle of the containers including deployment, scaling, and monitoring. As such, they are often used to build, test, and deploy applications consistently. A common use case for containers is to run microservices, which is a software architecture that employs a set of small, independent and loosely coupled applications\ \cite{microservices-overview}. 

\subsection{Containers Explained}
\label{containers}
Containers are a form of virtualization that offers a lightweight alternative to \Glspl{vm}. Unlike \Glspl{vm}, containers do not require a hypervisor to run and instead run on top of an \Gls{os} and share the \Gls{os} kernel with other containers. By not requiring a hypervisor, containers can run with less overhead than \Glspl{vm}, making their life cycle relatively fast and efficient compared to \Glspl{vm}\ \cite{container-vm-overview, container-k8s-benchmark}. The differences between container and hypervisor virtualization are visualized in figure \ref{fig:container-vm-architecture-2}.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{img/vm_container_virtualization.pdf}
    \caption{Overview of \Glspl{vm} and containers. The virtualization layer is placed on a lower level for \Glspl{vm} than for containers, illustrating the difference in their respective virtualization techniques.}
    \label{fig:container-vm-architecture-2}
\end{figure}

A container is created from a container image, which is a lightweight executable software package that can run a specific application including dependencies and data files. By encapsulating an application and its dependencies in a single package, the container image becomes portable and independent allowing for the container to run consistently across different environments. A container image is created by describing its parent image, such as an \Gls{os} image, and the steps to install and configure the application and its dependencies in a file, such as a Dockerfile. A parent image can itself have a parent image, creating a chain of dependencies. Furthermore, to enable caching and efficient storage, a container image can be divided into a series of steps referred to as layers. Each layer represents a change to the parent image, such as adding a file or installing a package. As such, layers act as checkpoints and allow for incremental builds where only the necessary layers are rebuilt when a change is made to the image\ \cite{container-docker-layers, container-dockerfile, container-docker-reproducibility}. Figure\ \ref{fig:container-image-layers} visualizes the concept of container image layers, where each operation represents a change to the parent image. 

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/container_image_layers.pdf}
    \caption{Overview of container image layers where each operation in a Dockerfile represents a layer that can be cached and reused. The layers are stacked on top of each other and appended to the parent image to form the final container image.}
    \label{fig:container-image-layers}
\end{figure}

Moreover, container images are often stored in container registries. A container registry provides central storage for container images and is used to distribute container images to environments that can run containers, such as a Kubernetes cluster. Common features for a container registry include versioning, access control, image scanning, and features to integrate with software development cycles, such as triggers to notify an external system when a new image is pushed to the registry\ \cite{container-registries, container-registries-redhat}.

The Open Container Initiative (OCI) is a standardization effort that was created to ensure that containers are portable and interoperable across different platforms\ \cite{oci}. The OCI standard defines a runtime specification, image specification, and distribution specification, allowing for a container to be consistent in terms of its runtime environment, image format, and distribution method. Moreover, a container runtime, such as containerd, is used to create, deploy, and manage containers on a node\ \cite{container-runtimes}. Most runtimes, including containerd, implement the OCI runtime specification and can consistently run containers. This suggests that the OCI can be viewed as an abstraction layer, where the OCI runtime specification acts as an intermediary between the container runtime and the \Gls{os} kernel, and the container image with the distribution specification acts as an intermediary between the container runtime and the container image registry.

\subsection{Orchestration Using Kubernetes}
\label{sec:kubernetes-orchestration}
Kubernetes is an open-source container orchestration platform that manages containers. It maintains a set of abstraction layers to manage the life cycle of containers and to provide a consistent environment for the containers to run in, facilitating tools for scheduling, scaling and networking\ \cite{container-vm-overview}.

Some of the abstractions used in Kubernetes to provide execution environments and communication include \textit{Pods}, \textit{Services} and \textit{Deployments}. A Pod is the smallest deployable unit in Kubernetes, which groups one or more containers into a single unit that can be run on nodes. This means that Kubernetes ensures that all containers in a Pod are run on the same node. Encapsulating several containers allows for the logical grouping of coupled containers that share the same resources, such as networking and storage. This is commonly used when a Pod contains a main application container and a secondary container that provides logging and monitoring, called a sidecar container\ \cite{k8s-docs-pods}. A Service represents a single-point-of-entry communication layer for a set of Pods and is used to provide a consistent way to access the Pods, such as load balancing. Services are especially useful in scenarios where Pods are fungible, meaning that the requesting client does not care which Pod it communicates with, as it creates an abstraction layer that hides the details of the Pods. This decouples frontends from backends and allows for a more scalable and resilient system. Furthermore, services are also useful for rolling updates, as they can be used to ensure that a certain part of the traffic is routed to the new version of the application\ \cite{k8s-docs-services}. Contrary to Pods and Services, a Deployment is a higher-level abstraction on top of Pods that is primarily used for replication control. A Deployment does not represent an executable resource but instead describes a desired state in which a set of Pods should be. This includes the number of replicas and update strategy. As such, a Deployment proves useful when a set of Pods should be managed as a single unit, such as when scaling up or down, or when deploying a new version of an application\ \cite{k8s-docs-deployments, k8s-overview}. 

Furthermore, to support stateful applications Kubernetes mainly uses \textit{PersistentVolumes}, \textit{PersistentVolumeClaims} and \textit{StatefulSets}. A PersistentVolume defines how and where persistent data is stored. Since there are many different ways to store data, such as local storage and network storage, a PersistentVolume is used to abstract the underlying storage. As such, each PersistentVolume is backed by a storage class, which defines the parameters and configuration to interact with the storage provider. A PersistentVolumeClaim, on the other hand, connects the PersistentVolume to a Pod. It acts as the bridge between the persistent storage and the execution environment effectively decoupling the Pod from the underlying storage. Contrary to PersistentVolumes and PersistentVolumeClaims, a StatefulSet is not used to manage the storage and is instead comparable to a stateful Deployment. A StatefulSet manages the life cycle of a set of Pods such that they are run predictably with their own unique identity. A common use case for a StatefulSet is scenarios that require each Pod to maintain a specific role, such as a primary or secondary database. A StatefulSet ensures that Pods maintain their role and that they are started in a predictable order, such as starting the primary database before the secondary database\ \cite{k8s-docs-statefulsets, k8s-overview}.

Figure \ref{fig:k8s-overview} presents a brief overview of the Kubernetes abstractions and how they interact with one another. 

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/kubernetes_overview.pdf}
    \caption{Brief overview of Kubernetes abstractions including many of the core components. The figure illustrates the communication between the different components and how storage is managed in Kubernetes.}
    \label{fig:k8s-overview}
\end{figure}
 

In certain scenarios, the built-in abstractions are not enough, and custom behavior is needed. For such scenarios, Kubernetes provides a way to extend its abstractions through Custom Resource Definitions (CRDs). A CRD is a form of extension for Kubernetes resources that allows for the creation of custom resources. It is used to extend the existing abstractions to provide additional features and capabilities that allow for a more feature-complete platform when custom behavior is needed. For instance, a CRD can be used to enforce certain fields to be specified to create the resource by applying custom validation rules\ \cite{k8s-crds, k8s-overview, k8s-crd-webiste}.

There are many other abstractions in Kubernetes, each with its own set of features and capabilities that together create a feature-complete platform for container orchestration, but presenting them all in this report falls out of the scope of this project.

Kubernetes is inherently declarative, meaning that instead of issuing commands to perform a specific task, such as starting a container, Kubernetes is given a desired state for a set of resources, and it is responsible for ensuring that the current state of the resources matches a desired state. For this reason, Kubernetes provides auto-restart, re-schedule, and replication control to ensure a resource stays in the desired state. By specifying the desired state, an entire application can be contained in a single configuration file, including its container, networking, storage and security requirements\ \cite{container-vm-overview, k8s-overview}. The entity responsible for ensuring the desired state is referred to as a controller, and there are many different types of controllers depending on the resources they manage. For instance, a Deployment controller is the entity responsible for ensuring that the number of Pod replicas in a Deployment matches the desired amount. Moreover, managing resources using a configuration file and a controller is referred to as the \textit{Controller pattern}, and is a common design pattern in Kubernetes as it provides a consistent and predictable way to manage resources \cite{k8s-docs-controllers}. 

Architecture-wise, Kubernetes uses a client-server approach. The server, often referred to as the control plane, is responsible for managing the state of the cluster. It is responsible for scheduling, scaling and networking by managing system-related services such as the \Gls{api} server, scheduler, and controller manager. The \Gls{api} server can be viewed as the primary service for a Kubernetes cluster since it represents the gateway for all management tasks for the cluster by receiving and processing requests from the client. The client, often referred to as the worker node, is responsible for running the containers. The worker node consists of two primary components: the kubelet and the kube-proxy service. The kubelet is a service that wraps a container runtime and runs containers according to requests by the control plane\ \cite{k8s-docs-kubelet, k8s-overview}. The kube-proxy service, on the other hand, manages the network communication for the containers. It can manage streams and forwarding rules for traffic using TCP, UDP or STCP\ \cite{k8s-overview, k8s-docs-kubeproxy}. 

To interact with Kubernetes from the perspective of a user, a client is used to send requests to the \Gls{api} server. The client can be a \Gls{cli}, such as kubectl, a web-based user interface, or a programmatic interface such as a REST \Gls{api}. The \Gls{cli} tool kubectl is the most common way to interact with Kubernetes and is used to create, delete, and manage resources in the cluster\ \cite{k8s-overview}. 

Through its abstractions, life cycle management and declarative client-server architecture, Kubernetes provides a consistent and feature-rich platform for container orchestration. This has led to wide adoption of Kubernetes, and as such, it is often used in large-scale cloud environments\ \cite{k8s-it-administration}. 

\section{Containers vs. Virtual Machines}
Virtualization has enabled the commoditization of computing resources due to the benefits of being able to map virtual hardware to physical hardware (see\ \ref{virtualization}) and has been key in enabling cloud computing. It has enabled the creation of flexible and scalable computing environments and has been used to build and manage large-scale cloud platforms. Virtualization has mainly brought two technologies: hypervisors and containers. Hypervisors are used to create \Glspl{vm}, including all associated aspects such as an \Gls{os}. Containers, on the other hand, are used to create a lightweight and portable environment for an application to run in and are often run inside \Glspl{vm}. As such, a container comes with less overhead when compared to a hypervisor. However, by its nature, a \Gls{vm} can provide a more isolated environment than a container, leading to a more secure environment. Thus, the benefits of containers do not come without a cost and there is a trade-off between the two technologies; one is lightweight, portable, and quick to start while the other is more independent and secure\ \cite{container-vm-overview, container-vm-comparison}. 

\subsection{Comparative Overview}
While containers and hypervisors are both technologies brought by virtualization, they operate on distinct levels of abstraction. Containers operate on a higher level than \Glspl{vm} using \Gls{os}-level virtualization, which leads to the \Gls{os} kernel with other containers. A hypervisor on the other hand can virtualize the entire hardware stack, including the entire \Gls{os}, depending on the type of hypervisor used (see\ \ref{hypervisors}). As such, the two technologies have different use cases and are often used in conjunction with each other\ \cite{container-vm-comparison}.

\label{container-vm-isolation}
The main difference between containers and \Glspl{vm} is the level of isolation they provide. Containers share the \Gls{os} kernel with other containers while \Glspl{vm} run on their \Gls{os}. While this means a container can share resources more efficiently, it also means that a container is subject to a larger attack surface, as a vulnerability in the \Gls{os} kernel can affect all containers running on the same host. \Glspl{vm}, on the other hand, reduce the attack surface by running more independently, meaning a compromised \Gls{vm} is less likely to affect other \Glspl{vm} running on the same host\ \cite{container-kata-security}.

Another aspect of isolation is how co-located containers and \Glspl{vm} can share resources. When sharing resources such as CPU and memory, containers are vulnerable to the noisy neighbor problem. The noisy neighbor problem occurs when a resource-sharing mechanism, such as a scheduler, is unable to provide a fair share of resources. Some scenarios include when a container can consume all available resources due to a heavily loaded application or when a container intentionally consumes all available resources to starve other containers. Solutions exist to mitigate the noisy neighbor problem, including limiting the number of resources a container can consume through \Gls{os}-level features such as \textit{cgroups}. A cgroup is a feature in the Linux \Gls{os} designed to limit, prioritize and isolate resource usage of a set of processes\ \cite{cgroups, vm-noisy-neighbor, container-vm-scale}. \Glspl{vm}, on the other hand, can provide a fair share of resources, as they operate in a more isolated environment, which leads to more resilient and predictable performance. However, the isolation does not mitigate the noisy neighbor problem for all resource types. For instance, a \Gls{vm} is still vulnerable to the noisy neighbor problem when sharing network-related resources such as network bandwidth and network-based storage\ \cite{container-vm-comparison}. 

Overcommitment is a technique used to improve packing efficiency, often referred to as \textit{consolidation ratio}. Containers use the overcommitment mechanisms built into the \Gls{os} itself, such as CPU overcommitment by multiplexing and memory overcommitment through virtual memory and memory swapping. Hypervisors, on the other hand, map virtual hardware devices to physical hardware, creating virtual computing resources that are not directly tied to the physical resources. This works well for the CPU by creating virtual CPU (vCPU) cores. However, memory overcommitment is a more complex issue in hypervisors than in containers due to the way hypervisors allocate memory. Unlike containers, hypervisors assign a static amount of memory to a \Gls{vm} when it is created, and only reclaim the memory when the \Gls{vm} is deleted. Methods such as memory ballooning exist to address the limitation, which is a technique that treats memory as a shared resource across \Glspl{vm} and allows the hypervisor to reclaim memory from \Glspl{vm} when it is needed elsewhere. It is implemented as a driver that reserves unused memory (deflation) on a \Gls{vm}, rendering it unavailable to other processes running in the \Gls{vm}. The memory pages on the physical memory mapped to these pages in the \Gls{vm} are allocated to other \Glspl{vm} (inflation). This allows for \Glspl{vm} to share unused memory\ \cite{vm-ballooning}. However, it is important to note that techniques like memory ballooning do not directly solve memory overcommitment for \Glspl{vm}, but instead reduce the need for it by managing the available memory more efficiently\ \cite{container-vm-comparison, vm-overcommitment}. Figure \ref{fig:memory-ballooning} visualizes the processes of deflating the memory of a \Gls{vm} and how it is used in another \Gls{vm}.

\label{container-vm-live-migration}
Live migration is a technique used to move a running resource from one host to another without interrupting the services provided by the resource. The technique is often used to balance the load or to perform maintenance on the hosting nodes. For \Glspl{vm}, it is a mature and widely technique for large data centers. Live migration works by copying memory pages from the source host to the destination host and then transferring the control of the \Gls{vm} from the source hypervisor to the destination hypervisor. Since \Gls{vm} disks are often stored on a shared storage, such as a Storage Attached Network (SAN), the \Gls{vm} disk is not transferred but rather reattached after the live migration is finished. For this reason, the \Gls{vm} can safely continue to run after its running state has been transferred to the destination host\ \cite{vm-live-migration}. Live migration for containers, on the other hand, is not as mature as for \Glspl{vm}. Unlike \Glspl{vm}, containers require \Gls{os}-level process migration, which is less reliable. However, several projects aim to provide live migration for containers, such as Checkpoint-restore in Userspace (CRUI). CRUI operates by checkpointing the state of a container, such as its memory and file descriptors, and then restoring the state on another host. While this is considered as container live migration, it is more comparable to snapshot and restore operations than a live migration operation, as CRUI freezes the container during the checkpointing, and converts the state of a container to a collection of files on disk, that can be used to restore the container on another host\ \cite{container-criu-overview, container-criu-linux, container-criu-web}. Another approach for live migrating containers involves terminating them on one host and restarting them on another. This strategy assumes the container is stateless or utilizes shared storage for its state, as is often used in Kubernetes since it does not provide container live migration. Instead, Kubernetes facilitates persistent storage through Persistent Volume Claims (PVCs), essentially allowing a user to map portions of the container's state to shared storage, and assumes the rest can be discarded when a container is terminated\ \cite{k8s-overview}.

\subsection{Decision Factors}
The choice between containers and \Glspl{vm} depends on the specific use case, and there are several factors to consider when deciding which technology to use. Some of the core factors are security, scalability, and legacy support\ \cite{container-vm-comparison}. 

Security is a key factor when deciding between containers and \Glspl{vm}. Using containers means sharing the \Gls{os} kernel, which implies that a vulnerability in the \Gls{os} kernel could affect all containers running on the same host. Containers are thus, from an attack-surface perspective, more prone to some vulnerabilities than \Glspl{vm}, making \Glspl{vm} more suitable in scenarios that require a higher level of isolation. For instance when an application manages sensitive or confidential data\ \cite{container-vm-decision-redhat}. Moreover, sandboxing is a security mechanism that is used to restrict an execution environment such that it can safely run untrusted code without affecting the rest of the system. For instance, when analyzing the effect of malware a sandbox environment ensures that the malware is not able to escape the restricted environment. Sandboxing is done in multiple layers, including limiting the resources a process can access, such as memory and CPU, and limiting the system calls a process can make. As such, a sandbox environment requires a high level of isolation, which makes \Glspl{vm} a natural fit\ \cite{vm-decision-kong, sandboxing}.

Configuration drift is another aspect to consider regarding security. It is a phenomenon that arises when the configuration of a system changes over time due to several factors including manual intervention and software updates, and can lead to an undefined state in a system that could create security holes\ \cite{configuration-drift-tripwire, configuration-drift-wiz}. Containers, as described in\ \ref{container-vm-live-migration}, are often run in a stateless manner using a single configuration file to describe the container, such as a Dockerfile. By being able to discard the state and recreate a container from a configuration file, the risk of configuration drift is reduced. As such, containers can provide a more consistent environment than \Glspl{vm}. Moreover, containers represent a more standardized and portable environment, using layers of dependencies, see figure \ref{fig:container-image-layers}. By standardizing the environment, external tools are more easily able to provide security features such as vulnerability scanning\ \cite{container-vm-decision}.  

When deciding between containers and \Glspl{vm} scalability is an important factor. One of the main benefits of containers is their ability to share the underlying \Gls{os}. However, as described in\ \ref{container-vm-isolation}, this also means that containers are more prone to the noisy neighbor problem, where co-located applications can cause performance interference. While this issue is related to isolation and security, it is more prominent in larger-scale environments, especially in multi-tenant environments\ \cite{container-vm-scale, container-vm-decision}.

Scalability is also related to cloud-native applications, which are applications that are designed to run in the cloud\ \cite{cloud-native}. A cloud-native application is characterized by being horizontally scalable, stateless, and resilient, and is often built using microservices, see\ \ref{containers-kubernetes}. Cloud-native applications thus require small and lightweight environments, making containers with an orchestration platform such as Kubernetes a natural fit.

Legacy support is another factor to consider when deciding between containers and \Glspl{vm}. Both \Glspl{vm} and containers can provide environments for legacy applications, such as applications that require a specific \Gls{os} version or a specific set of dependencies. However, certain legacy applications are not able to run in a container due to the way they are designed, such as applications that require a specific \Gls{os} kernel or a specific set of hardware. In such cases, a \Gls{vm} is often a better fit, as it enables virtualized hardware and a complete \Gls{os} to be run\ \cite{container-vm-decision,vm-decision-microsoft}.

\subsection{Towards Stronger Container Isolation}
Containers are often used in conjunction with \Glspl{vm} due to the lack of strong isolation properties out of the box, see\ \ref{container-vm-isolation}. However, several projects aim to provide the isolation properties of a \Gls{vm} and the lightweight properties of a container, such as Firecracker, Kata Containers, and gVisor.

Firecracker is a \Gls{kvm}-based hypervisor that creates lightweight \Glspl{vm}, called microVMs. Compared to a traditional \Gls{vm}, a microVM can start in a fraction of a second using only a fraction of the memory, comparable to that of containers. This is achieved through heavy optimization both in the hypervisor and in the guest functionality, such as reduced overhead through lightweight I/O and a stripped-down guest \Gls{os}\ \cite{container-runtimes-overview, container-firecracker}.

Kata Containers is a project that, similar to Firecracker, aims to provide lightweight \Glspl{vm} that perform like containers while also providing strong isolation properties. Unlike Firecracker, Kata Containers uses both \Glspl{vm} and containers and replaces the container runtime, such as containerd (see\ \ref{containers}) with the Kata runtime. The Kata runtime implements the OCI runtime specification to be interoperable with container ecosystems. However, instead of running the container directly on the host, the Kata runtime creates a \Gls{vm}, in which the container runs. This provides the isolation properties of a \Gls{vm} and the lightweight properties of a container. Kata Containers can achieve this by heavily optimizing the \Gls{vm} and its creation process towards running containers, as this is the only purpose of the \Gls{vm}\ \cite{container-runtimes-overview}. 

Another approach to strong container isolation is gVisor, a user-space kernel that provides a secure environment for the container. Unlike both Firecracker and Kata Containers, gVisor does not use \Glspl{vm} to enhance the isolation properties but instead operates as an extension around the container. gVisor consists of two core components, \textit{Sentry} and \textit{Gofer}. Sentry is a user-space kernel that intercepts and handles system calls made by the container and can provide a secure environment for the container. Gofer is a file access component that provides the container with file access. Separating the file access from the syscall access allows for a more secure environment. For instance, a compromised Sentry does not imply read or write access to any file\ \cite{container-gvisor-firecracker}.

In summary, Firecracker stands out among the three since it does not use container runtimes. As such, Kata Containers and gVisor are more comparable. Figure \ref{fig:secure-containers-overview} visualizes the differences between how Kata Containers and gVisor wrap a container to provide a more secure execution environment. However, due to the varying added overhead, they are yet to be widely adopted\ \cite{container-runtimes-overview}.

\section{Extending Kubernetes with KubeVirt}
\label{sec:kubevirt}
KubeVirt is a novel project that aims to provide a unified platform for managing both containers and \Glspl{vm} using Kubernetes. It is designed to naturally blend in with the Kubernetes ecosystem and provide a consistent interface to manage the life cycle of \Glspl{vm}, such as deployment, scaling, and monitoring. KubeVirt being built on top of Kubernetes means that it does not implement all the components found in a traditional \Gls{vm} management system, such as the scheduler, but instead uses the mature and feature-rich services the Kubernetes ecosystem provides\ \cite{k8s-scheduler}. By using CRDs (see\ \ref{containers-kubernetes}), KubeVirt can extend the Kubernetes \Gls{api} to provide a specification for \Glspl{vm} to support features commonly found in traditional \Gls{vm} management systems, such as live migration, scaling and snapshot management. Moreover, CRDs allow KubeVirt to enforce validation on the resources to prevent invalid configurations, such as a \Gls{vm} that references a non-existing storage disk\ \cite{kubevirt-architecture, kubevirt-validation}.

The design pattern chosen for KubeVirt's \Gls{vm} management is heavily inspired by other Kubernetes resources. A \Gls{vm} in KubeVirt is split into two resources called \textit{VirtualMachine} and \textit{VirtualMachineInstance}. The VirtualMachine resource is comparable to a StatefulSet and is used to describe the desired state of a \Gls{vm}, such as the \Gls{vm}'s desired running state, CPU and RAM requirements, as well as cloud-init configuration. The VirtualMachineInstance, on the other hand, is comparable to a Pod and is created by the VirtualMachine resource to represent the running state of the \Gls{vm}. As such, a non-existing VirtualMachineInstance represents a non-running \Gls{vm}, and an existing VirtualMachineInstance represents a running \Gls{vm}. The VirtualMachineInstance contains runtime information about the \Gls{vm} such as the \Gls{vm}'s IP address, running state and resource usage. 

Similar to other Kubernetes resources, such as Deployments and StatefulSets, KubeVirt uses controllers to manage the life cycle of the \Glspl{vm}. These controllers are together responsible for creating the environment in which a \Gls{vm} can be run, but also that the desired state of VirtualMachine resource matches the current state of its VirtualMachineInstance resource. For instance, when VirtualMachine is updated with requirements for CPU cores, the KubeVirt controllers are responsible for updating the VirtualMachineInstance to match the new specification\ \cite{k8s-docs-kubevirt, kubevirt-architecture}.

Live migration and snapshots are implemented as CRDs, allowing for a declarative interface to manage these features. This is in stark contrast to traditional \Gls{vm} management systems, where live migration and snapshot management are commonly implemented as imperative operations. For instance, by creating a LiveMigration resource, KubeVirt initiates a live migration operation for the referenced VirtualMachine resource\ \cite{kubevirt-snapshot,kubevirt-live-migration}. This is comparable to the binding operation process for a PersistentVolume and its corresponding PersistentVolumeClaim, where the binding takes place when a PVC is created\ \cite{k8s-persistent-volumes}

An additional benefit of KubeVirt being built on top of Kubernetes is that KubeVirt-related resources can be managed using the kubectl \Gls{cli} tool. This allows for the declarative creation, modification and deletion of \Glspl{vm} using the same interface as for other Kubernetes resources. However, certain operations are not possible to perform using kubectl. For instance, accessing serial or graphical consoles of a \Gls{vm} is not possible using kubectl. To enable these operations, KubeVirt provides a complementary \Gls{cli} tool called \textit{virtctl}. While virtctl primarily enables console access, it also provides convenience features such as live migration and snapshot management\ \cite{kubevirt-virtctl}.

\subsection{KubeVirt Technical Overview}
\label{sec:kubevirt-technical-overview}
On a technical level, KubeVirt's \Glspl{vm} are hosted inside Kubernetes Pods. As described in \ref{sec:kubernetes-orchestration}, a Pod is the smallest deployable unit in Kubernetes. KubeVirt does not create a new abstraction for an execution environment and instead encapsulates the \Gls{vm} within a Pod. This brings many benefits as it allows for seamless communication between the \Gls{vm} and the Kubernetes ecosystem since it complies with Container Network Interface (CNI). For instance, a \Gls{vm} can be exposed to the network using a Service or use PersistentVolumeClaims to provide persistent storage. 

KubeVirt consists of four main components to enable the running of \Glspl{vm} in Kubernetes: \textit{virt-controller}, \textit{virt-handler}, \textit{virt-launcher} and the \textit{libvirt daemon}. The virt-controller and virt-handler are responsible for managing the life cycle of the \Glspl{vm}, where the virt-controller is responsible for creating the Pod in which the \Gls{vm} will be hosted, and the the virt-handler is responsible for creating the \Gls{vm} itself. The virt-launcher, on the other hand, manages the runtime environment for a \Gls{vm}, including starting, stopping and monitoring the \Gls{vm}. Finally, the libvirt daemon (libvirtd) provides an interface to manage the \Glspl{vm} hosted by the underlying \Gls{kvm} hypervisor. The relationship between the three components is visualized in figure \ref{fig:kubevirt-components}, where the virt-controller operates closest to Kubernetes \Gls{api}-server, virt-handler operates as a DaemonSet on every node to communicate with the underlying hypervisor, and virt-launcher operates as a sidecar container in every Pod to manage the runtime environment of the \Gls{vm}\ \cite{kubevirt-architecture, k8s-docs-kubevirt}.

\subsection{KubeVirt and GitOps}
GitOps is a set of practices that enables using Git as a single source of truth for the desired state of the system, and it thus is heavily tied to the declarative nature of Kubernetes, as the desired state of the system is described in configuration files. By managing the infrastructure in Git, resources can be version controlled, and any changes to the system can be tracked and audited. This creates a developer-centric workflow, where developers can manage the infrastructure using the same tools and processes they use for managing application code\ \cite{gitops-evolution}. Moreover, since GitOps is used for systems to manage their resources declaratively, it is a natural fit for KubeVirt. As such, KubeVirt opens up the possibility of managing \Glspl{vm} using GitOps, meaning that \Gls{vm} specification and life cycle management can be version-controlled and audited\ \cite{gitops-kubevirt}. Being easily integrated with GitOps further indicates that KubeVirt can unify the management of both containers and \Glspl{vm} using Kubernetes.

\subsection{Comparison with Traditional Approaches}
KubeVirt does not represent a new virtualization or hypervisor technology but instead leverages QEMU and \Gls{kvm} to run the \Glspl{vm}, which means KubeVirt is fundamentally similar to traditional \Gls{vm} management systems, such as OpenStack, CloudStack, and OpenNebula\ \cite{kubevirt-white-paper-platform9}. Therefore, KubeVirt can provide the same level of isolation and security as traditional \Gls{vm} management systems, as it uses the same underlying hypervisor technology. However, KubeVirt is not as feature-rich, as it is not designed to be a complete replacement for traditional \Gls{vm} management systems. For instance, OpenStack not only manages \Glspl{vm} and their life cycle but also provides additional services, including host provisioning and data warehousing\ \cite{kubevirt-tech-comparison}. KubeVirt, on the other hand, is primarily designed to manage \Glspl{vm} and their life cycle such that it can provide a unified platform for managing both containers and \Glspl{vm}. As such, KubeVirt integrates well with Kubernetes to provide a seamless experience for \Gls{vm} management. For instance, \Glspl{vm} created with KubeVirt support CNI plugins and integrate with PersistentVolumeClaims to provide storage\ \cite{kubevirt-architecture, kubevirt-network-plugin, kubevirt-disks}.

\section{Related Work}
\label{literature-study}
\subsection{Current challenges}
The \Gls{cncf} presents in a report some of the challenges of managing \Glspl{vm} using a traditional \Gls{iaas} platform and uses OpenStack as an example\ \cite{kubevirt-cncf}. While OpenStack has been a popular choice for managing \Glspl{vm}, it has been criticized for being complex and difficult to manage, as well as having a steep learning curve. They argue that the complexity of OpenStack is not always needed and that it is possible to manage \Glspl{vm} using a simpler and more efficient platform, such as Kubernetes with KubeVirt. 

Additionally, the \Gls{cncf} report highlights the importance of having a unified platform for managing both \Glspl{vm} and containers, as it could reduce the complexity of the underlying technology stack and management costs. These claims are further supported by a report by Kubacki\ \cite{kubevirt-storware}. He argues that managing \Glspl{vm} using traditional \Gls{vm} management systems alongside Kubernetes leads to unnecessary costs, such as maintaining personnel and skill sets for two technologies. Moreover, by allowing the coexistence of \Glspl{vm} and containers in a single platform, KubeVirt could potentially reduce any barrier of entry around managing \Glspl{vm} for organizations that are already using Kubernetes for container orchestration, or act as a catalyst for organizations considering adopting Kubernetes. 

\subsection{KubeVirt challenges}
While KubeVirt is presented as a solution to the challenges of managing \Glspl{vm} using traditional \Gls{vm} management systems, it is important to consider the challenges of using KubeVirt itself. A study by Trakulmaiphol and Piromsopa presents some shortcomings when migrating from a traditional \Gls{vm} management system to KubeVirt\ \cite{kubevirt-migrate-issues}. One of the issues identified is the management of IP addresses, particularly when maintaining static IP addresses in a dynamic IP environment, such as Kubernetes. In the study, they developed a framework to solve some of the issues on the application level, such as DNS queries, and provided a set of guidelines related to KubeVirt's unique features, such as the Multus Container Network Interface. 

A common practice when managing Kubernetes clusters is to deploy Kubernetes nodes on \Glspl{vm} created by a \Gls{vm} management system. This means that when using KubeVirt, \Glspl{vm} are created inside other \Glspl{vm} leading to nested virtualization. Ye et al.\ argue in a study that nested virtualization raises security concerns due to the introduction of a larger trusted computing base (TCB)\ \cite{free-turtles}. By using a larger TCB, the size of the system's attack vector increases leading to an increased risk for vulnerabilities. For this reason, nested virtualization might not be suitable when strict security is required, such as applications managing sensitive or confidential data. This study is particularly relevant as it highlights a potential limitation if KubeVirt is simply seen as another Kubernetes resource, and not as a \Gls{vm} management system. 

\subsection{Usability of Kubernetes and KubeVirt}
Using Kubernetes as a \Gls{vm} management system is a relatively new concept, and there is limited research on the usability of KubeVirt. However, several studies have evaluated the usability of Kubernetes, which is relevant as KubeVirt is an extension built on top of Kubernetes.

Kubernetes \Gls{api} is inherently declarative, meaning an approach that promotes the usage of documents that describe a desired state of a system. A study by Sonninen investigates many of the possible benefits of using a declarative approach in system architecture\ \cite{declarative-benefits}. The study found that a declarative approach facilitates GitOps, a way to manage infrastructure using files in Git, which simplifies system and resource management and enables easier rollbacks. Using Git as a source of truth for the desired state of the system has the added benefits of version control, audit and collaboration. These benefits were exemplified when CERN migrated the internal CMS project from a \Gls{vm} cluster to Kubernetes, as described in the two studies they published. The first study conducted by Imran et al.\ describes how the Kubernetes migration enhanced the overall system management compared to using a \Gls{vm} cluster\ \cite{k8s-cern}. While many of their experienced benefits stem from adopting containers instead of \Glspl{vm}, the study still suggests Kubernetes provided many advantages, such as reduced operational costs. The other study conducted by Badran presents the practical benefits they experienced by using Kubernetes and is more focused on the adoption of a declarative approach for system management\ \cite{k8s-cern-techreport}. It describes how using Kubernetes instead of a \Gls{vm} cluster allowed them to craft reusable and composable configurations that enabled their entire infrastructure to be managed as code. 

The benefits of moving into Kubernetes are further exemplified in a study by Gelley\ \cite{k8s-cloud-foundry}. It presents a case study where an organization moved its business application from a \Gls{paas} provided by Cloud Foundry into Kubernetes. The study not only describes how Kubernetes provided tremendous flexibility in terms of configuration and deployment compared to the previous \Gls{paas} but also how the organization was able to leverage the Kubernetes ecosystem to provide additional features such as auto-scaling and monitoring. Moreover, by moving into Kubernetes, possible future migration to other platforms was made simpler due to how widely adopted Kubernetes is. 

User experience in Kubernetes was investigated in a study by Packard et al.\ that gathered several scenarios in which projects were built and managed using Kubernetes\ \cite{k8s-user-experience}. Many aspects were considered in the study, including administration such as managing persistent storage and limiting user resources with quotas, but also developer experience such as building and deploying applications. The study found that while system administrators face a learning curve when using Kubernetes, such as understanding the Kubernetes \Gls{api} and the ecosystem, users without prior knowledge of Kubernetes were able to successfully deploy applications. This finding is important to consider when evaluating KubeVirt, as it suggests that KubeVirt is a viable option even for users without prior knowledge of Kubernetes. Another study that investigated the user experience of Kubernetes was conducted by Boronin\ \cite{k8s-migration-king}. The study presents a case study where an organization faced a migration from a traditionally monolithic system architecture to a more cloud-native design. The study is relevant as it not only describes the challenges or benefits of Kubernetes itself but also considers the organizational impact of migrating to Kubernetes. This involves transferring skill sets among personnel, and the impact on the organization's culture.

In summary, these studies suggest that the benefits of using Kubernetes as a platform are not directly tied to what kind of resources are managed, but rather to the platform itself. Thus, when evaluating KubeVirt, it is important to not only consider benefits of KubeVirt itself but also the benefits that follow by adopting Kubernetes. 

\subsection{Performance Evaluations}
Regarding performance, several studies have evaluated popular systems that can manage \Glspl{vm}. For instance, open-source alternatives such as OpenStack and CloudStack, or proprietary systems such as Amazon EC2. 

A study by Kang et al.\ compared the performance of OpenStack and Amazon EC2 using a wide range of configurations and analyzed diverse aspects based on different hypervisor, storage and networking configurations\ \cite{openstack-ec2-comparison}. Apart from the performance comparisons, the study also investigated performance impact when using more advanced features, such as Infiniband networking instead of Ethernet. The study showed mixed results, where OpenStack outperforms EC2 in some scenarios such as when using Infiniband networking, but EC2 tends to outperform OpenStack in parallel processing. As such, the study concluded that component-wise, each system has strengths. This study brings with it relevant perspectives as it highlights many of the aspects that are important to consider when evaluating the performance of \Gls{vm} management systems.

Another study by Mullerikkal and Sastri compared OpenStack and CloudStack regarding implementation complexity, overall stability and performance\ \cite{cs-os-comparison-2009}. The performance tests were designed to measure disk- and network I/O, and not CPU or memory usage. The study found that while OpenStack had a more complex implementation, it was more stable. Furthermore, by using a set of benchmarks it found that OpenStack had a better overall performance compared to CloudStack. However, a few years later, a complementary comparison between OpenStack and CloudStack was conducted by Paradowski et al.\ \cite{cs-os-comparison-2013}. The comparison focused on performance and used several benchmarks such as deployment time, deletion time, and CPU utilization during deployment and deletion, and revealed significant and quantifiable differences between the two platforms. These tests were designed to represent user interactions with the system. However, the scale of the tests used in the comparison was relatively small and did not explore how the systems manage \Glspl{vm} under higher load. Analyzing the performance of a \Gls{vm} management system under higher load is important, as it can reveal how well a system scales when the ratio between \Glspl{vm} and nodes increases. 

In contrast, a later study by Amaral benchmarked KubeVirt uses a setup with a large number of \Glspl{vm} relative to the size of the nodes, aiming to explore the limits of KubeVirt's control plane (scheduler etc.)\ \cite{ibm-kubevirt-scale}. The study found that, while KubeVirt was able to create all \Glspl{vm}, the control plane suffered a heavy load, indicating a possible large overhead when KubeVirt provisions a large number of \Glspl{vm}. This finding is important to consider when evaluating KubeVirt, as it indicates that KubeVirt may not yet be suitable for managing a large number of \Glspl{vm}.

In terms of network performance, a study conducted by Kolahdouzan investigated the networking traffic pattern in KubeVirt and OpenStack by performing networking performance tests to evaluate potential \Gls{vm} overhead\ \cite{kubevirt-openstack-network}. The study found that the network overhead was significantly larger for scenarios in which \Glspl{vm} reside on separate nodes. However, while the study found overhead for both KubeVirt and OpenStack, it found that OpenStack had a considerably larger overhead. While this project will not focus on networking performance, it is important to consider the findings of this study when evaluating KubeVirt.

Hypervisors are a key component of \Gls{vm} management systems, and several studies have evaluated the performance of hypervisors. A study by Hwang et al.\ evaluated the performance of a range of hypervisors including \Gls{kvm}, Hyper-V and Xen\ \cite{hypervisor-comparion-2010}. The study found that applications and assigned resources had a significant impact on the overhead of the hypervisors. However, the study concluded that no single hypervisor was the silver bullet and that the choice of hypervisor should be based on the specific use case. Reviewing the performance of hypervisors is important when evaluating KubeVirt, as KubeVirt uses \Gls{kvm} as its hypervisor. 

\chapter{Performance Analysis}
As described in the literature study, Paradowski et al.\ presented a set of benchmarks used to evaluate CloudStack and OpenStack\ \cite{cs-os-comparison-2013}. This comparison was used as a foundational basis for the performance analysis in this project, employing many of its approaches for evaluating the performance of a \Gls{vm} management system. In this project, however, OpenNebula was compared with KubeVirt. Choosing OpenNebula as the second system was based on its popularity and the fact it is open-source, but also to ensure that the tests were relevant to Net Insight's current infrastructure. 

\section{Setup}
The testing process was conducted on Azure using a group of \Glspl{vm} that acted as hosting nodes. This approach was chosen to ensure that the tests are not affected by the underlying hardware and to facilitate reproducibility. 

A set of scripts was created to automate the entire testing process. The scripts automated node creation in Azure, as well as installation and configuration of the \Gls{vm} management system on the nodes. Once the tests were finished, the results were collected by the scripts and the nodes were deleted in Azure. The scripts were thus used to create a complete setup for each system in a consistent manner using a single command.

Payload-wise, this project used a similar approach to the study by Paradowski et al.\ with predefined types for \Glspl{vm} with CPU ranging from 1 to 4 CPU cores, and RAM ranging from 128 MB to 4 GB (see Table\ \ref{table:vm-specs} for a detailed overview). Moreover, \Glspl{vm} used the lightweight Cirros Linux image, since a lightweight \Gls{os} image is less likely to introduce noise in measurements\ \cite{linux-boot-time}. This also meant a \Gls{vm} disk could be smaller, which allowed for more \Glspl{vm} to be created per node and for tests to run quicker since a lightweight image is usually faster to boot than a larger image.

Each test case used a setup of three Azure nodes, one management node and two worker nodes. Having more than one worker node was crucial for certain tests to be able to run. For instance, a live migration test requires at least two worker nodes since it involves moving \Glspl{vm} between nodes. The Azure nodes were configured large enough to facilitate a broad range of tests, including tests with many small \Glspl{vm} and tests with a few large \Glspl{vm}. Azure offered many different \Gls{vm} types, many of which were suitable in terms of CPU and memory specification. However, only a few offered virtualization support, which was required when running nested \Glspl{vm}. This left the Azure Dv3-series as the most suitable \Gls{vm} type and, specifically, the D4v3 \Gls{vm} type that has 4 CPU cores and 16 GB of memory\ \cite{azure-dv3-series}. The price of the D4v3 \Gls{vm} type was 0.2 USD per hour, which was deemed acceptable for the project\ \cite{azure-pricing}. \section{Building a Framework}
Evaluating the performance of a \Gls{vm} management system is a complex task that requires a structured approach due to its many components and multi-node setup. For this reason, a framework to form standardized testing environments was created. The framework was designed to be \Gls{vm} management system agnostic, meaning that it can be used to evaluate any \Gls{vm} management system as long as the system supports the set of features defined in the framework. Consequently, this means that the framework can be extended to evaluate any other \Gls{vm} management system, such as the previously mentioned OpenStack and CloudStack, and not only OpenNebula and KubeVirt. However, it is important to note that the framework was designed to evaluate \Gls{vm} management systems that can be hosted on-premise since it quantifies the performance of a \Gls{vm} management system by measuring the utilization of nodes hosting the system. As such, the framework does not consider cloud-based \Gls{vm} management systems, such as Amazon EC2, Google Compute Engine or Azure Virtual Machines.

\subsection{Design}
\label{sec:framework-design}
The framework was designed to be modular and loosely coupled with external libraries. This design choice was made to ensure that the framework can be easily extended and modified and that it does not rely on external dependencies. To interact with a \Gls{vm} management system an interface was created that defines the methods that a \Gls{vm} management system must implement to be compatible. It defines methods for creating and deleting \Glspl{vm}, migrating \Glspl{vm} between nodes and scaling up and down the number of worker nodes, as well as methods for installing control and worker nodes. Each \Gls{vm} management system implements these methods in their respective packages. The full definition of the interface can be found in the appendix\ \ref{appendix:vmm-interface}. To host each \Gls{vm} management system, the framework created a set of nodes in Azure, which were completely installed and configured before any performance analysis was conducted to form a fair starting point for every \Gls{vm} management system. However, any dependencies required by the \Gls{vm} management system were intentionally left to the installation, as unnecessary dependencies could introduce noise in the measurements. 

Metrics were collected by running a daemon on each Azure node that collects utilization metrics, including CPU, memory and disk utilization. The daemon operated by fetching metrics every second and storing them in a JSON file locally. However, to only capture metrics when a test was running, the daemon only started collecting metrics when a test was initiated and stopped when the test was finished. This was controlled by placing an empty file called \textit{run} where the daemon was running. The daemon checked for the file's existence periodically and started collecting metrics when it noticed the file was present. As such, each node timestamped its metrics locally, which meant that the timestamp of the data points could vary between the nodes. However, this distributed approach was chosen to reduce the number of network requests to a minimum and to ensure that no skew was introduced in the measurements due to network latency. 

The utilization metrics were collected by a set of tools that were installed on each Azure node. Since the tools were required to be installed on every node, they were installed directly after creation. Table \ref{table:metric-tools} shows an overview of the tools used by the daemon. Another important aspect considered was asynchronous execution. Each of the tools used by the daemon was executed in parallel to minimize the time it took to collect the metrics for a given timestamp, essentially allowing shorter intervals between each data point. The structure of the output varied between the tools and was parsed using built-in tools in Linux such as \textit{awk} and \textit{sed} before being stored in a JSON file. The final output file was fetched over SSH using \textit{scp} and parsed by the framework to extract the metrics. The daemon was written in Bash to ensure that it could run on any Linux distribution and did not require any external dependencies. The code for the daemon can be found in the appendix\ \ref{appendix:metrics-collector} and the framework design is visualized in Figure\ \ref{fig:framework-design}.

The workflow of the framework was split into four phases to form a structured approach that allows for modularity and caching. Each phase was responsible for a specific part of the workflow and, in most cases, independent of each other. For instance, the installation phase can be omitted if the nodes are already installed and configured from a previous run. However, if the teardown phase was executed in one run, the next run after it must start with the setup phase. The design choice of dividing the workflow into phases was made to allow for more efficient testing, both when developing the scripts and when executing them. This was especially important considering that each test came with a cost in Azure. The entire workflow is illustrated in Figure \ref{fig:framework-workflow} and is described below:
\begin{enumerate}
    \item \textbf{Setup}: The setup phase is responsible for creating the nodes and networking in Azure. It is also responsible for installing base packages that every node requires, such as \textit{sysstat} to collect utilization metrics and \textit{jq} to parse JSON output.
    \item \textbf{Installation}: The installation phase is responsible for installing and configuring the \Gls{vm} management system on the Azure nodes. Each \Gls{vm} management system implements an installation method that is responsible for installing and configuring the system with one control node and two worker nodes such that it can run a \Gls{vm}. The installation method also installs several standby workers that will be used when scaling up and down the number of worker nodes. By having standby workers, the framework can independently measure the cost of scaling up and down without the cost of downloading dependencies. While larger dependencies can be seen as a metric on their own, it is important to note that production environments usually generate \Gls{os} images with prepared dependencies, rendering the cost of downloading dependencies irrelevant.
    \item \textbf{Benchmark}: The benchmark phase is responsible for running the performance tests. The tests are described in\ \ref{table:performance-tests} and are designed to evaluate the general performance of the \Gls{vm} management system. Each test is a combination of the methods the framework interface offers, such as creating \Glspl{vm} in different sizes, migrating \Glspl{vm} between nodes and scaling up and down the number of worker nodes. This phase generates and saves the results of the tests in a JSON file.
    \item \textbf{Teardown}: The teardown phase is responsible for deleting the Azure nodes and ensuring that no resources are left behind. Having this as a separate phase allows for tests to run sequentially without the time cost of creating and deleting nodes for each test. 
\end{enumerate}

\end{figure}

\subsection{Implementation}
The framework was implemented in the programming language Go, a language popular for its performance and concurrency features. Concurrency was a particularly important factor as several tests involved running multiple operations in parallel, such as creating or migrating several \Glspl{vm} at the same time. As such, settling for Go reduced the risk of introducing noise in the measurements and could run each test more efficiently. 

The most important part of the framework was arguably to provide fair and reproducible comparisons. As described previously, each \Gls{vm} management system implemented an interface that defined methods for creating and deleting \Glspl{vm}. While this provided uniform operations from the framework's point of view, it also made the method's implementation crucial: KubeVirt's implementation of creating a \Gls{vm} should be as close to OpenNebula's implementation as possible to ensure that the comparisons are fair and that no unnecessary overhead are introduced. One example of this was that any network-related configuration had to be omitted since network-related metrics were not considered in the project. Another example was that the scheduling configuration had to be edited to ensure that \Glspl{vm} were created in as similar a manner as possible. However, all features that a \Gls{vm} management system offered and that were beneficial for the specific operation were considered. For instance, OpenNebula did not allow a live migration action to be performed on a \Gls{vm} that is not running, while KubeVirt allowed the operation to be requested despite the state of a \Gls{vm}.\ This allowed KubeVirt to omit the waiting time for a \Gls{vm} to be in the correct state before requesting a migration action.

\chapter{Usability Analysis}
To accurately measure the usability of KubeVirt, a set of tasks was crafted in cooperation with a system administrator at Net Insight. The tasks were designed to be representative of the tasks that a system administrator would perform when managing \Glspl{vm}. The tasks were designed to be performed using the Kubernetes \Gls{cli} tool \textit{kubectl} and KubeVirt's complementary \Gls{cli} tool \textit{virtctl}. This approach was chosen since KubeVirt did not directly offer a \Gls{gui}, and using any third-party solution could risk introducing biases in the results. 

\section{Setup}
To host an environment that was easily accessible to the participants, an AKS cluster was set up. Alternatives included installing a Kubernetes cluster locally using tools such as \textit{Minicube} or \textit{k3d}, but this approach was deemed less suitable since it could be time-consuming and require more resources from the participants. Moreover, a cloud-based solution ensured that the participants could access the environment anywhere reducing possible barriers to participation. AKS specifically was chosen since it included preinstalled storage classes that enabled plug-and-play persistent storage functionality. KubeVirt's snapshot-restore \Gls{api} required a CSI storage class, which made AKS a suitable choice. Similar to the performance analysis, the cluster included one control-plane node and two worker nodes to ensure that the participants could perform the tasks without any issues. Having two worker nodes was crucial for the live migration task, as it required moving a \Gls{vm} between two nodes. Separating the control node from the worker nodes was not necessary for the tests, but it is a common practice since it allows workers to be recreated without affecting the cluster's control components.

\section{Task Design}
The analysis was designed as a step-by-step guide that would take the participants through a set of tasks that were divided into three groups: basic usage, maintenance and debugging. The focus of the analysis was not to evaluate the participant's ability to perform the tasks or their Kubernetes knowledge but rather to expose them to KubeVirt and its workflows. As such, the tasks were designed to be as straightforward as possible, except for a debugging task that was designed to evaluate finding and resolving runtime issues. Moreover, the order of the tasks was designed to gradually increase in complexity to ensure a structured approach and allow the participants to familiarize themselves with KubeVirt. Below is a description of each group of tasks:

\begin{itemize}
    \item\textbf{Basic usage}\\
    This task was designed to act as a starting point for the usability analysis. The participants were asked to create a single \Gls{vm} configured with a \textit{containerDisk}, connect to it using SSH and VNC and then delete it. The purpose of this task was to evaluate how well KubeVirt supports the most basic operations and whether the participants deem the process intuitive. 

    \item\textbf{Maintenance}\\
    The second task was designed to evaluate how well KubeVirt supports recurring tasks that a system administrator could face when managing \Glspl{vm} and the platform hosting them. The analysis divided this task into two parts: live migration and snapshotting. The participants were first asked to create a single \Gls{vm} and start it on a specific host, then migrate the \Gls{vm} to another host without any downtime. A lightweight \Gls{vm} configured with a \textit{containerDisk} was chosen for this task to keep the focus on the live migration operation. The second part of the task was to create a snapshot of a \Gls{vm} and then restore it. A \Gls{vm} configured with a \textit{DataVolume} disk was chosen for this task since a container disk cannot be snapshotted. Both live migration and snapshotting were included in the usability analysis since they are inherently different in KubeVirt compared to traditional \Gls{vm} management systems due to the declarative nature of Kubernetes, see \ref{sec:kubevirt}. 

    \item\textbf{Debugging}\\
    The third task was designed to evaluate how well KubeVirt supports debugging and how intuitive the process of resolving runtime issues is. While KubeVirt does enforce correct specification upon creation with \Gls{api} validation, see \ref{sec:kubevirt}, it is still possible that a \Gls{vm} does not start if the misconfiguration affects the runtime. For this reason, the participants were presented with a \Gls{vm} that was not accessible due to a cloud-init misconfiguration that prevents SSH access to the \Gls{vm} by having invalid password syntax and missing authorized SSH public keys. This means that the participants could solve the issue either by resolving the password formatting issue or supplying a valid SSH public key. The participants were presented with a basic problem statement and were encouraged to use the official documentation if needed. For this task, a lightweight \Gls{vm} configured with a containerDisk was used to keep the focus on the debugging operation.
\end{itemize}

After each group of tasks was finished, the participants were asked to reflect on the experience and provide feedback on the tasks. The participants were encouraged to use a free-form approach but could use a set of open-ended questions as a reference. 

\section{Conducting the Usability Analysis}
The usability analysis was performed together with a group of system administrators at Net Insight. The participants were presented with a set of tasks as described in the previous section. Each participant was given a document containing all the tasks as well as an introduction to KubeVirt and each of the operations they would perform. For each task, the participant was encouraged to use the official KubeVirt documentation to aid them in completing the tasks. Together with the document containing the tasks, each participant was given a \textit{kubeconfig} and the name for their Kubernetes namespace in which they would perform their tasks. They were given separate namespaces to ensure that none of the tasks would interfere with the results of the other participants. Once finished, the reflection and feedback were collected from each participant. They were given two weeks to complete the tasks, and while this was a relatively long time, it was chosen to ensure that the participants were given enough time to potentially explore features in KubeVirt that were not part of the given tasks. The entire usability analysis can be found in the appendix\ \ref{appendix:usability-tasks}.

\section{Usability Analysis}

The collected answers from the participants are summarized below. Some participants chose a free-hand approach when answering the questions, while other participants chose to answer the questions directly in a structured manner. The full answers from every participant can be found in the appendix\ \ref{appendix:usability-analysis-answers}.

\subsection*{Basic Usage}
The participants found the process of creating a \Gls{vm} in KubeVirt to be mostly straightforward.

\begin{itemize}
    \item Participant 1: \textit{The act of doing kubectl apply -f and kubectl delete is quite neat.}
    \item Participant 2: \textit{Very easy! I like that it is as simple as creating a YAML file and then
    applying it.}
\end{itemize}

However, participants expressed some confusion when connecting to the \Gls{vm} using SSH, as it was not clear which SSH public key was used to connect to the \Gls{vm}, and that conflicts with previously created \Glspl{vm} could occur. Moreover, participant 1, in particular, found the \Gls{vm} manifest definition complex, and that the process of creating it was not intuitive. 

\begin{itemize}
    \item Participant 1: \textit{It would be quite difficult for me to figure out I need to use bus: virtio without having someone else tell me I need to.}
\end{itemize}

KubeVirt's complementary \Gls{cli} tool \textit{virtctl} was found to be useful when connecting to the \Gls{vm}, as it provided a single tool for both VNC and SSH.

\begin{itemize}
    \item Participant 2: \textit{Very simple! Simply doing virtual ssh cirros@my-vm and then typing
    in the password and then I was in!}
\end{itemize}
\subsection*{Maintenance}

The maintenance tasks received a broad range of feedback. Some of the participants found the process of live migration to be simple, while other participants deemed the process to be overly complex and non-intuitive. Participant 1 in particular regarded many of the operations as flawed, such as the live migration object not being deleted after the migration was completed, but also that KubeVirt lacked proper documentation around the practical aspects of how to perform a live migration. Similar to the basic usage tasks, the participants experienced issues with connecting to \Glspl{vm} using SSH, as it was not clear which SSH public key was used to connect to the \Gls{vm}.

\begin{itemize}
    \item Participant 2: \textit{I had to delete the entry from /.ssh/kubevirt\_known\_hosts each time I restarted the \Gls{vm} because virtual ssh complained about a key mismatch.}
\end{itemize}

\subsection*{Debugging}

Compared to both the basic tasks and maintenance, the debugging tasks received the most negative feedback. The participants found the process of debugging to be complex and unclear. Finding the correct documentation was another issue some participants faced, as both the official documentation and the \textit{virtctl --help} command did not provide the necessary information to solve the issue. 

\begin{itemize}
    \item Participant 1: \textit{I did not manage to solve the key mismatch issue as there is very little in the way of debugging virtctl ssh.}

However, while debugging the issue presented in the task was found to be complex, participant 3 regarded the YAML files in general as a positive aspect of debugging.

\begin{itemize}

The feedback to KubeVirt was in general mixed. While the workflow was found to be intuitive due to its similarities with ordinary Kubernetes resources. This was particularly highlighted by Partcipant 4 who already mangaged other Kubernetes manifests in Git.

\chapter{Discussion}
The research questions involved investigating potential efficiencies that could be gained by integrating \Glspl{vm} in Kubernetes using KubeVirt. This project has evaluated KubeVirt using two different approaches: a performance analysis and a usability analysis. The discussion aims to explore deeper into the results of the project to provide insight into how feasible KubeVirt is as a \Gls{vm} management system.

\section{Limited Features and Result Intepretation}
The goal of the performance analysis was to find factual and provable claims regarding the performance of a \Gls{vm} management system. This included creating a framework that could be used to craft environments that were both reusable and fair among the systems being evaluated. The framework also provided a common interface for which an implementation was created per \Gls{vm} management system. By definition, such a system is only as good as the implementations are, which is why the implementations were designed to be as similar as possible to ensure a fair comparison. However, certain features and settings were not able to be tuned in the same way, such as the scheduling configuration where OpenNebula provides only a single option for scheduling: the number of \Glspl{vm} that can be dispatched concurrently. Not being able to accurately configure the systems identically in every aspect leaves the interpretation of the results open for discussion. This project has chosen the approach where if the system does not provide a specific feature to be tuned, the system is considered to be as optimized as possible. This approach was chosen as it favors the \Gls{vm} management system that provides the most features and flexibility, and as such, the results should be interpreted with this in mind. Having this aspect in mind is particularly important when interpreting the \textit{Create Many Type-1 \Glspl{vm}} tests, see figures\ \ref{fig:create-many-cpu},\ \ref{fig:create-many-memory},\ \ref{fig:create-many-disk} and\ \ref{fig:create-many-duration}. KubeVirt, using the scheduler built into Kubernetes, was able to create 20 type-1 \Glspl{vm} in 67 seconds, while it took OpenNebula 192 seconds on average. However, the results also suggest that KubeVirt is more resource-intensive, as the utilization of CPU, memory and disk was significantly higher than OpenNebula, which can viewed as a trade-off between the duration to complete a task and the resources required to complete it. This project has chosen the duration to be the more important metric, as it indicates that the system is capable of handling a burst of requests efficiently and using its allocated resources to its fullest potential. 

It is not always the case that KubeVirt's larger resource utilization is short-lived compared to OpenNebula, where some results suggest that KubeVirt has a significantly larger disk utilization than OpenNeubla for a similar period. This is particularly evident in figure\ \ref{fig:scale-cluster-disk} where KubeVirt and OpenNebula follow a similar characteristic when scaling the cluster but KubeVirt disk utilization is significantly more intense. For this specific test, the result suggests that KubeVirt does not provide a trade-off between the duration to complete the task and the resources required to complete it. This could be due to the way KubeVirt implements adding additional nodes to the cluster, which could be a resource-intensive operation. Another explanation could be that the specific Kubernetes distribution used in the tests is not optimized for scaling the cluster. However, exploring solutions for this is outside the scope of this project.

The final test used in this project included scaling the cluster with \Glspl{vm} running on each node. It involved scaling the cluster from 2 worker nodes up to 10 worker nodes, then creating a type-1 \Gls{vm} on each worker node before scaling it down to 2 worker nodes. This caused the \Gls{vm} management system to migrate the \Glspl{vm} to available nodes. This test posed a challenge for the scheduler of both systems since a \Gls{vm} might be scheduled to the wrong host if the scale-down operation was not handled immediately. The test was also an extension to the \textit{Scale Cluster} test that involved the same steps but without the \Glspl{vm}. The result shows that OpenNebula was not able to complete the test due to some \Glspl{vm} not being migrated to the correct host, and instead remained on the host that was requested to be removed. Since the test was designed to disconnect the worker and then wait for each host to be empty, the OpenNebula implementation hung indefinitely. A possible explanation for this could be that OpenNebula attempted to move a \Gls{vm} from a host that disconnected to another host that was just about to disconnect, but hadn't yet, and could cause the receiving host to end up in a bad state. Another explanation could be that OpenNebula requires additional control plane nodes to handle such bursts for migration operation since the test environment only provided a single control plane node to reduce the cost of the test since the nodes were hosted in Azure. However, this is purely speculative and exploring the cause of the issue is outside the scope of this project. The issue of \Glspl{vm} stuck on hosts could be resolved by manual intervention, but since this was deemed unfair to the other system, OpenNebula was disqualified from the scale-with-\Glspl{vm} test. As such, the result for scaling with \Glspl{vm} was omitted for OpenNebula and only KubeVirt was evaluated for this test.

\section{Kubernetes Overhead and Induced Load}
The results of the performance analysis indicate that both KubeVirt and OpenNebula have a consistent overhead and that it differs between the systems. Having a larger overhead implies the system has more ongoing computational work in the background while not performing a requested operation. This is particularly evident in the tests creating different \Gls{vm}-types, see figures\ \ref{fig:create-tiny-kubevirt},\ \ref{fig:create-tiny-opennebula},\ \ref{fig:create-large-kubevirt} and\ \ref{fig:create-large-opennebula}. KubeVirt CPU utilization hovers around 4\% with a memory utilization of just below 12\%. OpenNebula, on the other hand, has a CPU utilization just above 2\% and a RAM utilization of 8\%. This suggests that OpenNebula has a lower required overhead to maintain the system compared to KubeVirt. The overhead is also evident in the tests including live migration, see\ \ref{fig:migrate-many-memory}. OpenNebula starts the test utilizing just above 8\% of the available memory while KubeVirt uses over 13\%. However, while this would suggest that KubeVirt is less resource-efficient than OpenNebula, it is important to first consider the surrounding factors. Firstly, since
Kubernetes was originally designed to manage containers, it has a different set of requirements compared to a traditional \Gls{vm} management system such as OpenNebula. Certain operations enabling container orchestration might not be necessary for a \Gls{vm} management system, which could manifest as an overhead. Secondly, the overhead shown in the results is not necessarily connected to KubeVirts implementation but rather Kubernetes itself, which could in turn be tied to the specific Kubernetes distribution used in the tests. However, exploring possible optimizations to reduce such overhead or the source of the overhead is outside the scope of this project. Lastly, as described in\ \ref{sec:kubevirt-technical-overview}, KubeVirt achieves \Gls{vm} hosting in Kubernetes by using custom resource and controllers that wraps a \Gls{vm} instance inside a Pod. While this approach allows for seamless integration of \Glspl{vm} in Kubernetes, it also introduces an unavoidable overhead. In conclusion, the overhead is a trade-off for the flexibility and features that KubeVirt provides, and the results suggest that KubeVirt is more resource-intensive than OpenNebula when not performing operations.

The flip side of this is that many of the results suggest the opposite regarding induced load. Induced load is the added load on the system when performing a specific operation and is calculated as the difference between the total utilization and the base utilization. This can be difficult to interpret since a large induced load over a short period is not necessarily worse than a small induced load over a longer period. However, this project has chosen a short-lived burst of utilization to be favorable over a longer period of low to medium utilization since it suggests that the system is capable of using its allocated resources to its fullest extent. The results suggest that KubeVirt has a lower induced load in general than OpenNebula, where in many cases the induced load is more than double for OpenNebula compared to KubeVirt. This is particularly evident in figure \ref{fig:create-many-cpu} where OpenNebula both take twice as long to complete the test and have twice as large induced load from the operation, around 6\% compared to KubeVirt's 3\%. The difference in induced load is further illustrated in figure\ \ref{fig:migrate-many-cpu}. Not only does it show that OpenNebula reaches an induced load of 76\% while KubeVirt reaches just above 50\%, but also that induced load is more prominent in the total utilization for both systems. This suggests that induced load is a more significant factor when evaluating the performance of \Gls{vm} management systems and, as such, should be considered when interpreting the results. Figure\ \ref{fig:migrate-many-memory-induced} further illustrates this point where for OpenNebula, the induced load is a significant part of the total utilization. Having a smaller induced load suggests that the system is better at scale while having a smaller base overhead suggests that the system is more efficient when the system is idle or performing few operations. However, whether this is true for all operations a \Gls{vm} management system offers remains to be investigated.

\section{Jagged Disk Utilization}
Throughout the performance analysis results, there is a recurring and notable pattern specifically tied to disk utilization. The disk utilization graphs are jagged and, in many cases, do not follow the expected pattern of the specific operation performed in the test. This is particularly evident in the \textit{Create \Gls{vm} Type 4} tests, see figures\ \ref{fig:create-large-kubevirt} and\ \ref{fig:create-large-opennebula}. The disk utilization for both Kubevirt and OpenNebula is initially larger than expected and does not return to the same state after the test is finished. Since the tests were run multiple times in no particular order, this likely means that noise was introduced in the measurement that caused each test to generate data points in different ranges. When the induced disk utilization was low for a particular test, background disk utilization from sources such as \Gls{os}-level operations, was more prominent in the results. By then applying a rolling mean smoothing function to the dataset, the noise was reduced, but due to the variability, some jagged features remained. This is further supported by the fact that the disk utilization is less jagged for tests with a larger expected disk utilization, such as the \textit{Create Many Type-1 \Glspl{vm}} tests since these tests generate a larger induced load. While the jagged disk utilization is not a problem in itself, it is important to consider when interpreting the results, especially those where disk utilization is small, as it could introduce bias unrelated to the operation performed.

\section{Mixed Opinions Using KubeVirt}
The results of the usability analysis broadly indicate that the participants experienced KubeVirt differently. Some participants found KubeVirt intuitive and acknowledged many of the benefits discussed throughout this report, while other participants made strong claims against the technology as a whole. To understand this discrepancy, it is important to consider what the positive or negative feedback targeted specifically to understand if the opinions were contradictory or if they were based on different aspects of the technology. The positive feedback was mostly tied to the idea of using a well-known \Gls{cli} tool, \textit{kubectl}, and using declarative configuration files to define the \Glspl{vm}. This formed a foundation where the participants were comfortable and appeared to find similarities with their existing knowledge of Kubernetes and, as such, basic tasks received well. Mixed opinions appeared once the tasks became more complex. For instance, the live migration task was found, contradictorily, to be both intuitive and confusing. This could be due to the different workflow KubeVirt presents compared to OpenNebula, where KubeVirt uses a declarative approach to define the live migration operation, while OpenNebula uses an imperative action-based approach. This is supported by the feedback from participants 1 and 2, where Participant 2 found the process of creating a resource for the live migration to be straightforward, while Participant 1 expressed confusion and argued that the process was unclear regarding clean-up and when the operation was completed. Structuring a live migration operation as a resource, where the action is performed once the resource is created is a common pattern in Kubernetes, but the the results of the usability analysis suggest this approach could be confusing.

Other negative feedback was more concerned with exploring KubeVirt outside the scope of the given tasks. This is a particularly prevalent issue for one participant who deemed KubeVirt's official documentation to be lacking and missing practical examples of how to get started. Since exploring documentation was not part of the usability analysis, and the participants were not given any guidance on how to find the necessary information, it is difficult to determine if the participants were unable to find the information due to the lack of documentation or guidance. Either way, having issues with finding correct documentation is a significant problem for a technology that involves many user interactions and operations.

The mixed responses in the usability analysis indicate both possible biases among the participants regarding previous knowledge and experience with Kubernetes, but also that KubeVirt could be a controversial approach to managing resources. Kubernetes is known for its steep learning curve due to its declarative approach to resource management and verbose configuration files, and since KubeVirt follows the same idea, it is not surprising that the participants found the technology complex. Biases could thus have been caused by previous \Gls{vm} management systems being more approachable than KubeVirt, which could have affected the participant's opinions on KubeVirt. Exploring whether previous knowledge of a \Gls{vm} management system affects the perceived usability of KubeVirt is a possible future research direction but it is outside the scope of this project. 

\section{Concluding Research Questions}
The results from the performance analysis indicate both strengths and weaknesses for KubeVirt. As discussed in previous sections, KubeVirt inhibits a larger base utilization without performing any operation requested by a user compared to OpenNebula. However, the induced load is in general significantly lower for KubeVirt than for OpenNebula. This suggests that KubeVirt is more efficient when performing operations and thus could be more scalable than OpenNebula. Furthermore, the usability analysis indicates that KubeVirt in many cases benefits from the declarative approach inherited by Kubernetes. Together the previous discussions the first research can be answered.

\subsubsection*{What performance, scalability, and operational efficiencies can be gained by integrating \Glspl{vm} in Kubernetes?}
KubeVirt provides an efficient way to manage \Glspl{vm} in Kubernetes with a lower induced load than OpenNebula. KubeVirt appears to be more scalable than OpenNebula, as evidenced by the lower induced load and the shorter duration to complete tasks. However, KubeVirt has a larger base utilization than OpenNebula, which argues for the consideration of each specific use case. Operational efficiencies were acknowledged by some participants, but argued against by others. This suggests that KubeVirt could provide operational efficiencies, but that it requires a strong familiarity with Kubernetes and its concepts to be fully utilized. In summary, KubeVirt provides performance and scalability efficiencies when managing many \Glspl{vm}, but might be a more resource-intensive solution when idle or performing few operations. Operational efficiencies are gained when the system administrators are familiar with KubeVirt, but it could lead to confusion regarding the workflow and the technology itself.

\subsubsection*{What are the current challenges and limitations in managing \Glspl{vm} and containerized applications in separate environments?}
The literature study in Chapter 2 showed that \Glspl{vm} and containers share many aspects including what they provide for a user, such as an environment that allows for computational workloads, and are thus often used in the same context. However, the literature study also described how both \Glspl{vm} and containers have their own use cases due to, for example, the different levels of isolation they provide. Having \Glspl{vm} and containers in separate environments not only implies that the system administrators are required to have a broader skillset to manage both environments but also that benefits from one environment cannot be utilized in the other. This is particularly evident in the performance analysis where KubeVirt appears to benefit from the mature and efficient resource scheduler built into Kubernetes. Furthermore, Kubernetes provides a vast ecosystem of tools and resources that could be utilized by KubeVirt, including monitoring, logging and networking, as well as third-party integrations for storage, security and observability. In summary, this project not only presents the benefits of integrating \Glspl{vm} and containers into a unified environment but also the benefits of integrating \Glspl{vm} into Kubernetes. 

\subsubsection*{How can system administration be simplified by running \Glspl{vm} in Kubernetes using KubeVirt?}
The usability analysis highlights several aspects of how a system administrator can harness the benefits of moving into a more declarative approach to managing resources. This is particularly evident by one participant who found that the configuration files were easy to debug and "googleable" due to how contained the information was. This suggests that error handling and debugging could be simplified by being less time-consuming. Furthermore, the literature study presents other benefits of managing \Glspl{vm} in KubeVirt. Since the resources are managed declaratively it opens up the possibility of managing the resources using GitOps, which could provide a more secure and reliable way of managing resources since it removes the burden of version control and auditing from the system administrator. However, it is important to note that the results of usability analysis suggest that KubeVirt comes with a steep learning curve and that the technology could be perceived as complex. As such, the benefits found in the usability analysis and literature study might not be fully utilized if the system administrators are not familiar with Kubernetes and its concepts.

\chapter{Conclusions and Future Work}
This project has successfully evaluated Kubernetes as a \Gls{vm} management system by benchmarking the performance of KubeVirt against a traditional \Gls{vm} management system and by performing a usability analysis with system administrators. The results of the performance analysis suggest that the performance of both \Gls{vm} management systems are similar, but that KubeVirt is more efficient in large-scale environments due to its larger base overhead but lower induced load. Limitations among the \Gls{vm} management system, including available configuration and tuning options and the performance analysis framework itself were discussed, and the results were interpreted with these in mind. The usability analysis suggests that KubeVirt could provide operational efficiencies such as debugging, but that the technology could be perceived as complex and require familiarity with Kubernetes, its concepts and design philosophy. Potential biases among the participants were discussed, including whether the previous \Gls{vm} management system used affects the usability of KubeVirt and future research directions were proposed.

Further work could involve exploring the source of the overhead discrepancy found in the performance analysis, including why OpenNebula inhibits a significantly lower overhead than KubeVirt. It could also involve exploring if KubeVirt's apparent lower induced load is a result of the Kubernetes scheduler, the Kubernetes distribution or KubeVirt's implementation. Additionally, further work could explore the source of the potential biases found in the usability analysis and answer if the adoption of KubeVirt differs depending on the previous \Gls{vm} management systems used, as this project has only concluded that previous Kubernetes experience affects the usability of KubeVirt positively. Lastly, further work could investigate KubeVirt's applicability in environments with strict security requirements and whether Kubernetes offers the necessary tools and resources to meet these requirements, as this could further support the claims made in this project regarding the benefits of Kubernetes integration.

In summary, the project has shown that while OpenNebula outperforms KubeVirt in some scenarios due to its simplicity and low overhead, KubeVirt appears more scalable in larger-scale deployments. KubeVirt also provides operational efficiencies given that the system administrators are familiar with both the core concepts of Kubernetes and KubeVirt itself.

\section{Metrics Collector}
\label{appendix:metrics-collector}
\begin{lstlisting}[language=bash]
#!/bin/bash

interval=1
OUTPUT_FILE="./output.json"

cpu_tmp_file=$(mktemp)
ram_tmp_file=$(mktemp)
disk_tmp_file=$(mktemp)

while true; do
    if [ ! -f "./run" ]; then
        sleep $interval
        continue
    fi

    (mpstat -P ALL "$interval" 1 | awk '/^[0-9]/ {if (NR>1) print $3/100}' | jq -R -s -c 'split("\n")[1:-1]' | jq '[.[] | gsub(","; ".") | tonumber]' > "$cpu_tmp_file") &

    (free | awk '/Mem:/ {print $3/$2}' | sed 's/,/./' > "$ram_tmp_file") &

    (iostat -dx "$disk" 1 2 | awk '/Device/{flag=1; next} flag && /^[^ ]/ {util=$NF} END{print util/100}' | sed 's/,/./' > "$disk_tmp_file") &
    timestamp=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

    wait

    name=$(hostname)
    cpu_core_utilization=$(cat "$cpu_tmp_file")
    cpu_utilization_avg=$(echo "$cpu_core_utilization" | jq 'add/length')
    ram_utilization=$(cat "$ram_tmp_file")
    disk_utilization=$(cat "$disk_tmp_file")

    # Create the JSON object for the current datapoint
    datapoint=$(jq -n \
                    --argjson cpuCores "$cpu_core_utilization" \
                    --argjson cpuAverage "$cpu_utilization_avg" \
                    --arg name "$name" \
                    --arg ramUtilization "$ram_utilization" \
                    --arg diskUtilization "$disk_utilization" \
                    --arg timestamp "$timestamp" \
                    '{name: $name, cpu: {cores: $cpuCores, average: $cpuAverage}, ram: ($ramUtilization | tonumber), disk: ($diskUtilization | tonumber), timestamp: $timestamp}')

    if [ ! -f "$OUTPUT_FILE" ]; then
        echo "[]" > "$OUTPUT_FILE"
    fi

    jq --argjson newData "$datapoint" '. += [$newData]' "$OUTPUT_FILE" > tmp.$$.json && mv tmp.$$.json "$OUTPUT_FILE"

    sleep $interval
done
\end{lstlisting}

\section*{Introduction}
This is a usability analysis of KubeVirt that involves performing a set of tasks and then reflecting on the experience, where the goal is to evaluate how well KubeVirt works in practice. By participating in the usability analysis you provide valuable feedback and insight that can be used to improve KubeVirt. Your answers will be included as-is in the appendix of the project report. Upon request, the answers can be anonymized.

The following is a brief introduction to KubeVirt, its most relevant features and how it compares to previous \Gls{vm} management systems. If you are already familiar with KubeVirt and its machinery, you can skip this section.
\\~\\
KubeVirt is an extension to Kubernetes that allows for the creation of virtual machines (\Glspl{vm}) and is designed to provide a similar experience to managing any other resource in Kubernetes, such as Pods or Deployments. This means that \Glspl{vm} are defined in YAML files and integrate with the command line tool \textit{kubectl}. KubeVirt defines its resources as a \textit{Custom Resource Definition} (CRD) in Kubernetes, where the most prominent resources are \textit{VirtualMachine} and \textit{VirtualMachineInstance}. The relationship between them is comparable to that of a Deployment and a Pod, where the Deployment defines a desired state and a Pod represents the running state. See References below for more information. 

Certain \Gls{vm} operations are not possible using kubectl, such as accessing the \Gls{vm} using SSH or VNC. For this reason, KubeVirt provides a complementary \Gls{cli} tool called \textit{virtctl}.

Being built on top of Kubernetes allows the \Glspl{vm} to integrate with other resources, including the Service and PersistentVolume resources. For instance, a \Gls{vm} can be exposed to the network using a LoadBalancer Service or manage storage dynamically using a persistent storage provisioner. However, since a \Gls{vm} requires a disk image (such as a QCOW2 image), it means that a PersistentVolume must point to a folder with a disk image in it. For this reason, manually creating a PersistentVolume for a \Gls{vm} can be tedious. Instead, KubeVirt primarily provides two alternatives for managing disks that are presented below. See the References section for more information.
\begin{itemize}
    \item \textbf{containerDisk}: A disk that is created at every boot. It is not persistent and is similar to a container.
    \item \textbf{DataVolume}: A disk that is persistent and initialized when the \Gls{vm} is created. It uses PersistentVolumeClaim and a PersistentVolume under the hood where it injects a disk image. 
\end{itemize}

\section*{Maintenance (Task 2/3)}
The second task is related to maintenance, where you will create a snapshot of a \Gls{vm} and then restore it. You will also perform a live migration on a \Gls{vm} to move it from one node to another without shutting it down. 

Both snapshots and live migrations are treated as a resource by KubeVirt, meaning that the operation will take place in the background after a VirtualMachineSnapshot resource or a VirtalMachineInstanceMigration resource is created. See the References section for more information.

A live migration resource in KubeVirt does not move resources itself and instead uses the Kubernetes scheduler to decide where to place the \Gls{vm}. This means that a LiveMigration resource only tells the scheduler that the \Gls{vm} can be moved but not where to move it. See the References section for more information. The Kubernetes scheduler is sophisticated and supports a wide range of scheduling strategies, but for this task, we can force a move between two nodes by using the \textit{spec.template.spec.nodeSelector} in the \Gls{vm} and editing the labels of the worker nodes in the cluster. To not interfere with other participants, you should create a unique label for you. 

A snapshot in KubeVirt works similarly to a snapshot in other \Gls{vm} management systems, where it captures the state of the \Gls{vm} at a specific point in time. However, the snapshot uses Kubernetes VolumeSnapshot, which is a Kubernetes \Gls{api} for taking snapshots of PersistentVolumes. This means that a \Gls{vm} with a containerDisk cannot be snapshotted. Instead, you should use a \Gls{vm} with a DataVolume. See the References section for more information.
