\section{Virtualization Fundamentals}
\label{virtualization}
Virtualization is a technology \cite{virtualization-basics} that enables partitioning or grouping computing resources into operating environments using methods such as partitioning, emulation, and time-sharing. Virtualization could occur in various forms, such as network virtualization, storage virtualization, and server virtualization. More commonly used, though, is hypervisor based virtualization \cite{virtualization-history}, which is a software layer that creates virtual operating environments, or more commonly referred to as virtual machines (VMs).

A VM is a emulation of a physical computer \cite{virtualization-trends} and is perceived as \textit{real} computer by the application running inside it even though underlying resources are, more often than not, shared with other VMs. For this reason, virtualization is often used to improve resource utilization and to isolate applications from each other. Such a setup improves flexibility of the computing environment, but it implies that the underlying mechanism is formally different from that of a physical machine. For instance, hardware abstraction is used to decouple the VM's perceived hardware capabilities from the physical hardware. Abstracting the underlying hardware allows for \textit{virtual} computing resources, such as virtual CPUs and virtual memory, either by grouping or partitioning the physical resources.

Furthermore, the abstraction layer can exist in multiple different layers depending on the virtualization technology used, ranging from the hardware level to the application level, some of which are listed below:
\begin{itemize}
    \item Full virtualization (FV): Provides a complete emulation of the underlying physical hardware and creates a complete virtual environment. FV enables fully decoupling the VM from the underlying resource, and does not require any modification \cite{virtualization-basics}\cite{xen-virtualization}\cite{hypervisor-xen}.
    \item Paravirtualization (PV): A subset of FV aiming to improve the VM performance by reducing the amount of resource that are emulated. The VM is presented with an image that is similar, but not identical, to the underlying hardware. As such, the guest OS is aware it is running inside a VM \cite{xen-virtualization}\cite{hypervisor-xen}.
    \item Application virtualization (AV): A method where an application is encapsulated in a virtualized lightweight environment, such it perceived it is being run on a specific platform. Application virtualized using AV act as a portable and self-sufficient unit, thus often used to package and distribute applications, such as the JVM \cite{virtualization-basics}\cite{application-virtualization}.
\end{itemize}

Define virtualization and discuss its significance in modern IT infrastructures, including the evolution from physical to virtual resources.
\subsection*{Benefits and Challenges}
Many of benefits with virtualization stems from the fact that there exists an abstraction layer above the physical hardware. The abstraction layer decouples software from hardware, e.g. vendors with different I/O devices can be used in the same environment, allowing for a more flexible and scalable computing environment as whole where hardware is turned into simple pool of computing resources \cite{virtualization-trends}. As such, there are numerous benefits with virtualization, a few of which are listed below:
\begin{itemize}
    \item Resource utilization: Virtualization allows for better utilization of hardware resources, as multiple VMs can run on the same physical hardware. This is achieved by sharing the underlying resources, such as CPU and memory, between the VMs \cite{virtualization-basics}. 
    \item Isolation: VMs are able to isolate their entire state from one another, including operating system (OS), filesystem and application stack. Isolation is also beneficial in scenarios where reproducibility is key, such as in testing and development environments \cite{vm-docker-reproducibility}.
    \item Flexibility: VMs not tied to a specific hardware, and can be moved between different physical resources.
    \item Management: VMs can be created and destroyed on demand, and can be managed using software tools \cite{virtualization-basics}.
    \item Virtual hardware: A VM can be configured with hardware that does not exist in the physical environment, such as a virtual network card or a virtual hard drive \cite{virtualization-basics}.
\end{itemize}

However, the portability made possible by virtualization comes at a cost of performance. Since, at the core, a VM is a complete but emulated system, it means the every instructions issued must be interpreted in software. As such, despite what level of abstraction layer is used, there is an inescapable added overhead when using virtualization. A common hypervisor such as KVM (see \ref{hypervisors}) introduces an overhead of 5-10\% \cite{kvm-overhead}, and the overhead can be even larger when using a full virtualization approach.

\section{Understanding Hypervisors}
\label{hypervisors}
A hypervisor is a software layer that enables managing multiple VMs on a single physical machine by allowing multiple operating systems (termed as \textit{guests}) to share the same hardware \cite{virtualization-history}\cite{hypervisor-survey}. Hypervisors, often referred to as virtual machine monitors (VMMs), are responsible for creating and managing VMs, and for providing the necessary abstractions to the VMs.

Hypervisors can be categorized into two types, \textit{type 1} and \textit{Type 2}, based on their operational context and how they interact with the underlying hardware \cite{hypervisor-survey}\cite{hypervisor-types-1-2}. While the goals and purpose of type 1 and Type 2 hypervisors is the same, there are some core differences. type 1 hypervisors, commonly referred to as \textit{bare-metal} hypervisors, run directly on top of the physical hardware and do not require an OS underneath. A type 1 hypervisor is therefore scalable to terabytes of RAM and hundreds of CPU cores \cite{hypervisor-types-1-2}. However, without any other additional software, it is also itself responsible to schedule and allocate required system resources to the VMs. Type 2 hypervisors, also known as \textit{hosted hypervisors}, relies on an underlying OS to manage the calls to the CPU and memory resources. The presence of an underlying OS inevitably introduces latency, which is why Type 2 hypervisors are often used for development and testing purposes \cite{hypervisor-survey}.

Furthermore, hypervisors can be further categorized into \textit{monolithic} and \textit{microkernelized} hypervisors \cite{hypervisor-comparion-2013}. Monolithic hypervisors are built as a single large software stack, managing all the VMs and their resources, and hosting drivings for all hardware access. Having all drivers reside within the hypervisor means that the such as hypervisor will only run on selected hardware. However, it also means no operating system is required at all, as the hypervisor acts as a operating system by supporting all the operations of its VMs. Microkernelized hypervisors, on the other hand, does not require devices drivers to be present in the hypervisor. Instead, the drivers are installed in a form of managing VM, often referred to as parent VM. Each VM created by such a hypervisor is thus referred to as a child VM, as it uses the parent VM's drivers to access the hardware. This approach is more flexible, as it can allow the hypervisor to run on any hardware that supports the parent VM's operating system. 

% security rings definition
In the realm of hypervisors, the concept of \textit{security rings} is often used to describe the level of privilege different part of a hypervisor has. A security ring represents a hierarchical protection domain, where the innermost ring (ring 0), usually reserved for the OS kernel, has the highest privilege, and the outermost ring (ring 3), usually reserved for user applications, has the lowest privilege. The hypervisor itself is often placed in ring 0, its VM kernel space in ring 1, and the VM user space in ring 3. Security rings are thus used to define execution level to provide a system to limit execution capabilities of VMs in order to regulate access to shared memory. This enabled a strong isolation layer between the VMs, and between the VMs and the hypervisor, thus a key part of a hypervisor's security model \cite{hypervisor-comparion-2013}\cite{microkernel-monolithic}.

\subsection*{Key Hypervisors}
\label{hypervisors}
There are several hypervisors available, each with its own set of features and capabilities, some of which are listed below:

\begin{itemize}
    \item VMware ESXi: A lightweight type 1 monolithic hypervisor often used in enterprise environments. ESXi is known for its ultra lightweight footprint stretching merely a few hundred megabytes, and its ability to run directly on the hardware without an underlying OS \cite{hypervisor-comparion-2013}.
    \item Microsoft Hyper-V: A type 1 microkernelized hypervisor that is part of the Windows Server operating system. As a microkernelized hypervisor, Hyper-V differentiates between parent and child VMs, and uses the parent VM to manage hardware access for the child VMs \cite{hypervisor-comparion-2013}\cite{hypervisor-hyper-v-esxi}.
    \item Citrix Hypervisor: Formerly known as XenServer, is a type 1 microkernelized hypervisor based on the Xen Project. It runs using a modified Linux kernel in a VM, called Dom0, that manges hardware I/O and interacts with the other virtual machines, called DomU guests \cite{hypervisor-xen}\cite{hypervisor-comparion-2013}. 
    \item KVM: A type 1 hypervisor that is part of the Linux kernel. As such, KVM uses most of the functionalities from the Linux kernel, such as the scheduler and memory management. KVM has become a popular alternatives to proprietary hypervisors, such as VMware ESXi and Microsoft Hyper-V, as it is open-source and free to use \cite{hypervisor-comparion-2013}\cite{hypervisor-kvm}. 
\end{itemize}

\section{VM Management Essentials}
A management system for VMs involves a set of tools and processes that are used to create, deploy, and manage VMs \cite{openstack-opennebula-comparison}. The management system is responsible for managing the entire life cycle of a VM, including provisioning, deployment, monitoring, and scaling. Not be confused with a hypervisor, a VM management system lives on a higher-level and aims to provide an abstraction layer in the form of an interface to allow interaction with the VMs, often enabling the support of a set of hypervisors \cite{cloudframeworks-and-hypervisors}. As such, a VM management system unifies hypervisors, typically through an higher-level API, such as a CLI or a web-based user interface.

\subsection*{Concepts and Features}
With a higher abstraction layer, a VM management system brings a set of concepts to simplify managing VMs, such as clustering, availability, and scalability \cite{cloudframeworks-and-hypervisors}. Clustering includes grouping a set of resources, such as VMs or storage nodes, to provide a image that is perceived as a single resource. Availability refers to the ability to keep a resource active or running, which is, more often than not, clustered resources. Scalability involves technologies to ensure resources are able to manage an increase or decrease in computational load, such as increasing the amount of virtual cores in a VM if it's overloaded.  

\subsection*{Infrastructure as a Service}
A VM management system is often part of a larger cloud computing infrastructure, such as Infrastructure as a Service (IaaS). IaaS is a lower-level cloud computing model that aims to eliminate the need to install and manage physical hardware by providing virtualized computing resources, such as VMs, virtual network and storage.

\subsection{Comparison of VM Management Systems}
Managing VMs can be done in many ways, and there are many VM management systems available, each with its own set of features and capabilities. Many such systems are a part of a larger platform, such as an IaaS or even as large as a cloud platform. Some of the most prevalent platforms that has been used in the industry are OpenStack, CloudStack, and OpenNebula, and some of the most popular public cloud platforms are Amazon EC2, Azure Virtual Machines, and Google Compute Engine.

\subsubsection*{OpenStack}
An open-source IaaS platform that offers a large set of features to manage VMs, network, and storage \cite{openstack-opennebula-comparison}. OpenStack's architecture is inherently modular, and it is designed to be deployed in a distributed manner \cite{openstack-overview}. The architecture includes three major parts:
\begin{itemize}
    \item \textit{Nova}: The compute service that manages VMs and provides an interface to manage VMs. Nova support all operation regarding the life cycle of a VM, such as creating, deleting, and migrating VMs. Nova is itself divided into multiple copmonents, each employing a \textit{shared-nothing} and \textit{messaging-based} policy, which means every component is independent and communicates with other components using a message queue. 
    \item \textit{Swift}: The object storage service that provides a scalable and redundant storage system for VMs. It is a long-term storage for static data, such as images and backups, and is comparable to Amazon S3.
    \item \textit{Glance}: The image service that provides a catalog of VM images that can be used to create new VMs. 
\end {itemize}

OpenStack supports features such as live migration, snapshot management, and scalability, and is often used in large-scale cloud environments \cite{openstack-opennebula-comparison}. However, these features are often hypervisor-specific, and requires both support from the hypervisor and specific configuration to work \cite{openstack-live-migration}.

\subsubsection*{CloudStack}
An open-source IaaS platform and cloud computing software that manages computing, networking and storage resources, and is used to build and manage public and private cloud environments \cite{cs-overview}. On a high level, CloudStack is built in a monolithic fashion, encapsulating all the management features into two software packages called the management server and agent. The management server is responsible for managing the life cycle of VMs, including provisioning, deployment, and scaling, and is comparable to a control node. The agent, however, is installed on each node designated to running VMs and is responsible for low-level management task of the VMs on that node, such as interacting with the underlying hypervisor. While it is a monolithic system, CloudStack is designed to be horizontally scalable, and can be deployed in a distributed manner with multiple management servers \cite{cs-overview}. 

CloudStack supports a large variety of hypervisors, such as VMware, KVM, and XenServer, and storage services, such as Amazon S3, allowing it to function as an abridgment layer for compute, network, and storage to unify a wide range of existing infrastructure \cite{cs-overview}. Features such as live migration, snapshot management, and scalability are supported, but are often hypervisor-specific and requires specific configuration to work \cite{cs-live-migration}.

\subsubsection*{OpenNebula}
An open-source IaaS platform and toolkit used to provide open, scalable, and comprehensive management of virtualized data centers \cite{openstack-opennebula-comparison}. OpenStack employs a modular architecture divided in three layers \cite{openenbula-tele-teaching}\cite{iaas-comparisons}: 
\begin{itemize}
    \item \textit{Tools layer}: Provides a set of interfaces such as a CLI and web-based user interface to interact with the system, but also high-level management tools such as scheduling.
    \item \textit{Core layer}: Provides the core functionalities of the system, such as host management, VM management, and virtual network management.
    \item \textit{Driver layer}: Provides an abstraction on top of the underlying infrastructure to enable the system to support a wide range of hardware, including networking, storage, and hypervisors. As such, the driver layer is responsible for allocation, deallocation and monitoring of resources, such as VMs and storage. 
\end{itemize}

By dividing the system into these layers, OpenNebula provides modularity that enables the system to be extended and customized to fit specific needs, such as integrating with other systems or using OpenNebula to build a private cloud.

Amazon Elastic Compute Cloud (EC2) is a web service that provides on-demand compute resources over the cloud \cite{ec2-overview}\cite{ec2-overview-web}. EC2 is part of Amazon Web Services (AWS), and is designed be scalable and flexible, and is often used to host general-purpose applications, including web applications, databases and enterprise applications. While EC2 platform uses Xen-based hypervisors (see \ref{hypervisors}) under the hood \cite{public-cloud-comparison}, it acts as a complete abstraction layer over the underlying hardware and hypervisors, and instead provides a rich set of tools to manage the VMs.

This enables EC2 to provide a wide range of VMs, such as general-purpose, compute-optimized, memory-optimized, and storage-optimized VMs \cite{ec2-overview}. \subsubsection{Comparison}
\label{vm-management-comparison}

OpenStack, OpenNebula, and CloudStack are all examples of software that can be used to build both private and public cloud platforms \cite{public-private-cloud}\cite{hybrid-cloud}. A private cloud is an on-premise cloud that hosted within an organization and is, more often than not, dedicated for  internal use. A private cloud on the other hand, is a cloud platform meant to facilitate external users. A common denominator for these systems is that they are open-source, and that they are highly customizable and extendable. As such, much of the configuration and management of the system is done by the user, and the system can be integrated with other systems to fit specific needs. This means that many of the conveniences such as preconfigured VMs and features such as live migration often requires specific configuration and setup. 

Amazon EC2, Azure Virtual Machines and Google Compute Engine are examples of VM management systems in public cloud platforms. They designed to be more user-friendly and to provide a more complete abstraction layer over the underlying hardware and hypervisors \cite{public-private-cloud}. In comparison, these technologies can been seen as complete setups of a the open-source platforms mentioned previously, with all the configuration and management done by a provider. As such, many of the conveniences and features are provided out of the box, and are perceived as hypervisor- and hardware-agnostic. This means that the user does not need to worry about the underlying hardware and hypervisors, and can instead focus on managing the VMs. This creates a trade-off where the user has less control and customization, but instead gains a more user-friendly and complete system.

Both private and public cloud solutions for VM management are designed to be scalable and flexible, and support a wide range of use cases (although private cloud solutions scales worse in practice \cite{hybrid-cloud}). Which solution to use depends on the specific needs of the organization, and the level of control, customization needs, and the required security level of the system.

\section{Containers and Kubernetes: A Paradigm Shift}
\label{containers-kubernetes}
Besides the common cloud service models (IaaS, PaaS, and SaaS), another service models has emerged, called \textit{Container as a Service (CaaS)} \cite{caas}. CaaS is a container based cloud service model that has gained significant traction due to the lightweight and portable nature of a container, allowing for simple application bundling that is able to run in a vast range of environments. 

Containers are often used in conjunction with a container orchestration platform, such as Kubernetes, to manage the life cycle of the containers including deployment, scaling, and monitoring. As such, they are often used to build, test, and deploy applications in a consistent manner. A common use case for containers is to run microservices, which is a software architecture that employs a set of small, independent and loosely coupled applications \cite{microservices-overview}. 

\subsection{Containers Explained}
\label{containers}
Containers are a form of virtualization that offers a horizontally scalable and lightweight alternative to VMs \cite{container-k8s-benchmark}\cite{container-vm-overview}. Unlike VMs, containers do not require a hypervisor to run, and instead run on top of an OS and shares the OS kernel with other containers. By not requiring a hypervisor, containers are able to run with less overhead than VMs, making their life cycle relatively fast and efficient compared to VMs. 

A container is created from a container image, which is a lightweight executable software package that can run a specific application including dependencies and data files. By encapsulating an application and its dependencies in single package, the container image becomes portable and independent, allowing for the container to run in a consistent manner across different environments. A container image is created by describing its parent image, such as an OS image, and the steps to install and configure the application and its dependencies in a file, such as a Dockerfile. A parent image can itself have a parent image, creating a chain of dependencies referred to as layers. Managing the container image as a series of layers allows structured and transparent dependency management and efficient caching \cite{container-docker-layers}\cite{container-dockerfile}\cite{container-docker-reproducibility}. 

Container images are often stored in a container registry. A container registry provides a central storage for container images, and is used to distribute containers images to environments that are able to run containers, such as a Kubernetes cluster. Common features for a container registry includes versioning, access control, image scanning, and features to integrate with software development cycles, such as triggers to notify an external system when a new image is pushed to the registry \cite{container-registries}\cite{container-registries-redhat}.



The Open Container Initiative (OCI) is a standardization effort that was created to ensure that containers are portable and interoperable across different platforms \cite{oci}. The OCI standard defines a runtime specification, image specification, and distribution specification, allowing for a container to be consistent in terms of its runtime environment, image format, and distribution method. Moreover, a container runtime, such as containerd, is used to create, deploy, and manage containers on a node \cite{container-runtimes}. Most runtimes, including containerd, implement the OCI runtime specification, and are able to run containers in a consistent manner. This suggests that the OCI can be viewed as an abstraction layers, where the OCI runtime specification acts as an intermediary between the container runtime and the OS kernel and the container image, and the distribution specification acts as an intermediary between the container runtime and the container image registry.

\subsection{Orchestration Using Kubernetes}
Kubernetes is an open-source container orchestration platform that manages containers. It maintains a set of abstraction layers to manage the life cycle of containers and to provide a consistent environment for the containers to run in, such as scheduling, scaling, and networking \cite{container-vm-overview}.

Kubernetes is inherently declarative, meaning that instead of issuing commands to perform a specific task, such as starting a container, Kubernetes is given a desired state for a set of resources, and it is responsible for ensuring that the current state of the resources matches a desired state \cite{container-vm-overview}\cite{k8s-overview}. For this reason, Kubernetes provides auto-restart, re-schedule, and replication control to ensure a resource stays in the desired state. By specifying the desired state, an entire application is able to be contained in a single configuration file, including its container, networking, storage, and security requirements. 

Some of the abstractions used in Kubernetes includes \textit{Pods}, \textit{Services}, and \textit{Deployments} \cite{k8s-overview}. A Pod is the smallest deployable unit in Kubernetes, and represents a group of one or more containers that are scheduled to run on a node. A Service represents a single-point-of-entry communication layer for a set of Pods, and is used to provide a consistent way to access the Pods, such as load balancing. A Deployment is a higher-level abstraction on top of Pods that is used to manage the life cycle of the Pods, such as scaling and rolling updates. A Deployment does not represent an executable resource, but instead describes a desired state in which a set of Pods should be in. There are many other abstractions in Kubernetes, each with its own set of features and capabilities that together creates a feature complete platform for container orchestration.

A Custom Resource Definition (CRDs) is a form of extension for Kubernetes resources that allows for the creation of custom resources \cite{k8s-overview}\cite{k8s-crd-webiste}. CRDs are used to extend the Kubernetes existing abstractions to provide additional features and capabilities to allow for a more feature complete platform when custom behavior is needed. Kubernetes uses a client-server architecture, where the server, often referred to as the control plane, is responsible for managing the state of the cluster, and the client, often referred to as the worker node, is responsible for running containers \cite{k8s-overview}. The control plane runs system-related services, such as the API server, scheduler, and controller manager, and is thus responsible for managing the state of the cluster. The API server can be viewed as the primary service for a Kubernetes cluster as it is the main management point for the cluster, receiving and processing request from the client. The worker node, on the other hand, is responsible for running the containers. The worker node consist of two primary components: the kubelet and the kube-proxy. The kubelet is a service that includes a container runtime, which runs containers in accordance to requests from the control plane. Meanwhile, the kube-proxy service manages the network communication for the containers. To interact with Kubernetes from the perspective of a user, a client is used to send requests to the API server. The client can be a CLI, such as kubectl, a web-based user interface, or a programmatic interface such as a REST API. The CLI tool kubectl is the most common way to interact with Kubernetes, and is used to create, delete, and manage resources in the cluster \cite{k8s-overview}.

\section{Containers vs. Virtual Machines}

Virtualization has enabled the commoditization of computing resources due to the benefits of being able to map virtual hardware to physical hardware (see \ref{virtualization}), and has been key in enabling cloud computing \cite{container-vm-overview}\cite{container-vm-comparison}. It has enabled the creation of flexible and scalable computing environments, and has been used to build and manage large-scale cloud platforms. Virtualization has mainly brought two technologies, hypervisors and containers. Hypervisors are used to create VMs, including all associated aspects such as an OS. Containers, on the other hand, are used to create a lightweight and portable environment for an application to run in, and are often run inside VMs. As such, a container comes with less overhead when compared to a hypervisor. However, by its nature, a VM is able to provide a more isolated environment than a container, leading to a more secure environment. Thus, the benefits of containers does not come without a cost, and there is a trade-off between the two technologies; one is lightweight, portable, and quick to start, while the other is more independent and secure. 

\subsection{Comparative Overview}
While containers and hypervisors are both technologies brought by virtualization, they operate on distinct levels of abstraction \cite{container-vm-comparison}. Containers operate on the application level using OS-level virtualization, which leads to sharing parts the OS kernel with other containers. A hypervisor on the other hand is able to virtualize the entire hardware stack, including the entire OS, depending on the type of hypervisor used (see \ref{hypervisors}). As such, the two technologies have different use cases, and are often used in conjunction with each other.

\subsubsection*{Isolation}
\label{container-vm-isolation}
The main difference between containers and VMs is the level of isolation they provide. Containers share the OS kernel with other containers while VMs run on their own OS. While this means a container is able to share resources more efficiently, it also means that a container is subject to a larger attack surface, as a vulnerability in the OS kernel can affect all containers running on the same host. VMs, on the other hand, reduces the attack surface by running in a more independent fashion, meaning a compromised VM is less likely to affect other VMs running on the same host \cite{container-kata-security}.

Another aspect of isolation is the how co-located containers and VMs are able to share resources. 
When sharing resources, such as CPU and memory, containers are vulnerable to the noisy neighbor problem. The noisy neighbor problem occurs when a resource sharing mechanism, such as a scheduler, is unable to provide a fair share of resources. Some scenarios include when a container is able to consume all available resources due to a heavily loaded application, or when a container intentionally consumes all available resources to starve other containers. Solutions exists to mitigate the noisy neighbor problem, such as limiting the amount of resources a container is able to consume through OS-level features such as \textit{cgroups}\cite{cgroups}. However, this solution brings a trade-off where system resources might not be fully utilized if there is not enough demand \cite{vm-noisy-neighbor}\cite{container-vm-scale}.  

VMs, on the other hand, are able to provide a more fair share of resources, as they provide a more isolated environment, leading to more resilient and predictable performance. However, the isolation does not mitigate the noisy neighbor problem for all resource types. For instance, a VM is still vulnerable to the noisy neighbor problem when sharing network related resources, such as network bandwidth and network-based storage \cite{container-vm-comparison}. 

\subsubsection*{Overcommitment}
Overcommitment is a technique used to improve packing efficiency, referred to as \textit{consolidation ratio}, and it is often necessary to overcommit physical resources \cite{container-vm-comparison}\cite{vm-overcommitment}. Containers use the overcommitment mechanisms built into the OS itself, such as CPU overcommitment by multiplexing and memory overcommitment through virtual memory and memory swapping. Hypervisors, on the other hand, provide overcommitment by mapping virtual hardware devices to physical hardware, creating virtual computing resources that are not directly tied to the physical resources. Memory overcommitment is a more complex issues in hypervisors than in containers due to the way hypervisors allocate memory. Unlike containers, hypervisors assign a static amount of memory to a VM when it is created, and only reclaims the memory when the VM is deleted. However, methods such as memory ballooning address this limitation \cite{vm-ballooning}. Memory ballooning is a technique used to treat memory as a shared resource across VMs, and allow the hypervisor to reclaim memory from VMs when it is needed elsewhere, thereby reducing the need for memory overcommitment.

\subsubsection*{Live Migration}
\label{container-vm-live-migration}
Live migration is a technique used to move a running resource from one host to another without interrupting the services provided by the resource \cite{vm-live-migration}. The technique is often used to balance load or to perform maintenance on the hosting nodes. 

Live migration of VMs is mature and wildely used in large data centers, and works by copying memory pages from the source host to the destination host, then transferring the control of the VM from the source hypervisor to the destination hypervisor. Since VM disks are often stored on a shared storage, such as a Storage Attached Network (SAN), the VM disk is not transferred but rather reattached after the live migration is finished. For this reason, the VM can safely continue to run after its running state has been transferred to the destination host. 

Live migration for containers, on the other hand, is not as mature as for VMs. Unlike VMs, containers require OS-level process migration, which is not as reliable. However, there are several projects that aim to provide live migration for containers, such as Checkpoint-restore in Userspace (CRUI) \cite{container-criu-overview}\cite{container-criu-linux}\cite{container-criu-web}. CRUI operates by checkpointing the state of a container, such as its memory and file descriptors, and then restoring the state on another host. While this is considered as container live migration, it is more comparable to snapshot and restore operations than a live migration operation, as CRUI freezes the container during the checkpointing, and converts the state of a container to a collection of files on disk, that can be used to restore the container on another host. Another approach for live migrating containers involves terminating them on one host and restarting them on another. This strategy assumes the container is stateless or utilizes shared storage for its state, as is often used in Kubernetes since it does not provide container live migration. Instead, Kubernetes facilitates persistent storage through Persistent Volume Claims (PVCs), essentially allowing a user to map portions of the container's state to shared storage, and assumes the rest can be discarded when a container is terminated \cite{k8s-overview}.

\subsection{Decision Factors}
The choice between containers and VMs depends on the specific use case, and there are several factors to consider when deciding which technology to use \cite{container-vm-comparison}. Some of the core factors are security, scalability, and legacy support. 

Security is a key factor when deciding between containers and VMs. Using containers means sharing the OS kernel, which implies that a vulnerability in the OS kernel could affect all containers running on the same host. Containers are thus, from an attack-surface perspective, more prone to some vulnerabilities than VMs, making VMs more suitable in scenarios that require higher level of isolation. For instance when an application manages sensitive or confidential data \cite{container-vm-decision-redhat}. Moreover, sandboxing is a security mechanism that is used to restrict an execution environment such that it is able to safely run untrusted code without affecting the rest of the system. For instance, when analyzing the effect of malware a sandbox environment ensures that the malware is not able to escape the restricted environment. Sandboxing is done in multiple layers, including limiting the resources a process is able to access, such as memory and CPU, and limiting the system calls a process is able to make. As such, a sandbox environment requires a high level of isolation, which makes VMs a natural fit \cite{vm-decision-kong}\cite{sandboxing}.

Configuration drift is another aspect to consider regarding security. It is a phenomenon that arises when the configuration of a system changes over time due to several factors including manual intervention and software updates, and can lead to a undefined state in a system that could create security holes \cite{configuration-drift-tripwire}\cite{configuration-drift-wiz}. Containers, as described in \ref{container-vm-live-migration}, are often run in a stateless manner using a single configuration file to describe the container, such as a Dockerfile. By being able to discard state and recreate a container from a configuration file, the risk of configuration drift is reduced. As such, containers are able to provide a more consistent environment than VMs. Moreover, containers represents a more standardized and portable environment, using layers of dependencies, see fig. By standardizing the environment, external tools are more easily able to provide security features such as vulnerability scanning \cite{container-vm-decision}.

When deciding between containers and VMs scalability is an important factor. One of the main benefits of containers is their ability to share the underlying OS. However, as described in \ref{container-vm-isolation}, this also means that containers are more prone to the noisy neighbor problem, where co-located applications can cause performance interference. While this is issue is related to isolation and security, it is more prominent in larger scale environments, especially in multi-tenant environments \cite{container-vm-decision}\cite{container-vm-scale}.

Scalability is also related cloud-native applications, which are applications that are designed to run in the cloud \cite{cloud-native}. A cloud-native application is characterized by being horizontally scalable, stateless, and resilient, and if often built using microservices, see \ref{containers-kubernetes}. Cloud-native applications thus requires small and lightweight environments, making containers with an orchestration platform such as Kubernetes a natural fit.

Legacy support is another factor to consider when deciding between containers and VMs. Both VMs and containers are able to provide environments for legacy application, such as applications that require a specific OS version or a specific set of dependencies. However, certain legacy applications are not able to run in a container due to the way they are designed, such as applications that require a specific OS kernel or a specific set of hardware. In such cases, a VM is often a better fit, as it enables virtualized hardware and a complete OS to be run \cite{container-vm-decision}\cite{vm-decision-microsoft}.

\subsection{Towards Stronger Container Isolation}
Containers are often used in conjunction with VMs due to the lack of strong isolation properties, see \ref{container-vm-isolation}, out of the box. However, there are several projects that aims to provide the isolation properties of a VM and the lightweight properties of a container, such as Firecracker, Kata Containers, gVisor.

Firecracker is a KVM-based hypervisor that creates lightweight VMs, called microVMs \cite{container-runtimes-overview}. Compared to a traditional VM, a microVM is able to start in a fraction of a second using only a fraction of the memory, comparable to that of containers. This is achieved through heavy optimization both in the hypervisor and in the guest functionality, such as reduced overhead through lightweight I/O and a stripped down guest OS \cite{container-firecracker}.

Kata Containers is a project that, similar to Firecracker, aims to provide lightweight VMs that performs like containers while also providing strong isolation properties. Unlike Firecracker, Kata Containers uses both VMs and containers, and replaces the container runtime, such as containerd (see \ref{containers}) with the Kata-runtime. The Kata-runtime implements the OCI runtime specification to be interoperable with container ecosystems. However, instead of running the container directly on the host, the Kata-runtime create a VM, in which the container runs. This provides the isolation properties of a VM and the lightweight properties of a container. Kata Containers is able to achieve this through heavily optimizing the VM and its creation process towards running containers, as this is the only purpose of the VM \cite{container-runtimes-overview}. 

Another approach to strong container isolation is gVisor, a user-space kernel that provides a secure environment for container. Unlike both Firecracker and Kata Containers, gVisor does not use VMs to enhance the isolation properties, but instead operates as an extension around the container. gVisor consists of two core components, \textit{Sentry} and \textit{Gofer}. Sentry is a user-space kernel that intercepts and handles system calls made by the container, and is able to provide a secure environment for the container. Gofer is a file access component that provides the container with file access. Separating the file access from the syscall access allows for a more secure environment; a compromised Sentry does not imply read or write access to any file \cite{container-gvisor-firecracker}.

All three technologies aim to unify the main benefits of containers and VMs by providing a environments that are both lightweight and secure. However, due to the varying added overhead, they are yet to be widely adopted \cite{container-runtimes-overview}.
