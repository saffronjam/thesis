\chapter{Introduction}
\section{Background}
Virtualization is a technique that either splits or combines computing resources into one or more operating environments by creating a software layer that surrounds or underlies existing hardware or software. This enables the creation of virtualized resources that are decoupled from the underlying hardware. Virtualization can occur in various forms including the virtualization of storage, networks and servers. Virtualizing networks and storage is achieved by creating an abstraction layer on top of the physical network and storage resources, respectively, to provide a more flexible and scalable environment for other resources to interact with. Server virtualization, on the other hand, is used to create virtual computing environments that are decoupled from the underlying computing resources. By combining virtualized resources into a single operating environment, virtualization allows heterogeneous hardware resources to be used homogeneously and thus it improves resource utilization and flexibility\ \cite{virtualization-history, virtualization-basics, virtualization-trends}. 

To enable server virtualization and manage virtual computing environments, a hypervisor is used. A hypervisor is a software layer that operates between the hardware and the operating system (OS) and is responsible for creating virtual computing environments called virtual machines (VMs)\ \cite{virtualization-trends, virtualization-ibm, hypervisor-survey}. VMs are emulations of physical computers that are perceived as real computers by applications running inside them. They provide a complete and independent environment by virtualizing an OS, rendering them suitable for a wide range of applications, including legacy applications and applications that require a high level of isolation. For this reason, VMs have long been the core part of IT infrastructure, serving as foundational components in system architecture\ \cite{virtualization-history, virtualization-basics}. 

In parallel with the evolution of VMs, containers have emerged as an alternative to VMs. Container virtualization is an OS-level virtualization technique that allows for running multiple isolated applications on a single OS instance. This means container virtualization does not require a hypervisor, and instead uses the features and capabilities of the OS kernel to create and manage containers. As such, containers are more lightweight, require less overhead, and have faster startup times than VMs\ \cite{container-vm-overview}. This has made containers a popular choice for scenarios in which scaling and portability are key, such as in microservices, which are core components in modern cloud-native applications\ \cite{cloud-native}. Figure\ \ref{fig:container-vm-architecture} visualizes the differences between how VMs and containers are virtualized, and how the virtualization layer is placed on a lower level for VMs than for containers. 

\begin{figure}[h]
    \centering
    \includegraphics[width=1\textwidth]{img/vm_container_virtualization.png}
    \caption{Overview of VMs and containers}
    \label{fig:container-vm-architecture}
\end{figure}

However, while containers provide a lightweight and efficient alternative to VMs, they lack the deep isolation properties inherent to VMs. Sharing the OS kernel with the host and other containers opens up a larger attack surface, which implies that containers are more prone to security vulnerabilities. While there exist technologies that aim to unify the isolation benefits of VMs and the lightweight benefits of containers, such as Kata Containers, gVisor and Firecracker, it remains a fact that both containers and VMs have their use cases\ \cite{container-runtimes-overview, container-as-vm-comparison}. 

Due to their different use cases, VMs and containers have traditionally been managed separately. VMs are often managed by a VM management system that provides a set of tools and processes to aid in the life-cycle management of VMs. This makes a VM management system act as a higher-level abstraction layer that unifies hypervisors, typically through an application programming interface (API) such as a command line interface (CLI) or a web-based user interface. Figure \ref{fig:vm-management-system} depicts a broad overview of the role of a VM management system where clients interact with an interface to manage VMs that are distributed across several hosts. Moreover, a VM management system is often part of a larger infrastructure, such as Infrastructure as a Service (IaaS), that provides a wider range of virtualized resources including virtual networks and storage. Examples of such systems include OpenStack\ \cite{openstack-website}, CloudStack\ \cite{cloudstack-website}, and OpenNebula\ \cite{opennebula-website}. Containers, on the other hand, are often managed by a container orchestration platform. These platforms provide a similar set of tools and processes as a VM management system but for containers. Apart from creating and deploying containers, a container orchestration platform provides tools specifically designed to manage the life cycle of containers, such as scheduling and scaling. Examples of such platforms include Kubernetes\ \cite{k8s-website}, Nomad\ \cite{nomad-website}, and Docker Swarm\ \cite{docker-swarm-website}, with the former being the most widely adopted \cite{k8s-it-administration}. Furthermore, with the evolution and adaption of container orchestration platforms like Kubernetes, there has been an ongoing convergence in how VMs and containers are managed\ \cite{vm-cloud}. 

\begin{figure}[h]
    \centering
    \includegraphics[width=1\textwidth]{img/vm_management_system.png}
    \caption{Overview of a VM management system}
    \label{fig:vm-management-system}
\end{figure}

Kubernetes, originally designed to manage containerized applications at scale, has recently been extended to accommodate VMs through the KubeVirt project. KubeVirt is an API extension for Kubernetes, that enables the running of VMs inside a Kubernetes environment alongside containers, creating a common environment for both virtualization technologies\ \cite{kubevirt}. Figure \ref{fig:container-vm-kubevirt-architecture} depicts a high-level overview of how VMs and containers coincide in a Kubernetes environment using KubeVirt, where traffic flows natively between the two types of resources. Further technical details of how KubeVirt achieves this will be discussed in \ref{sec:kubevirt-technical-overview}. 

\begin{figure}[h]
    \centering
    \includegraphics[width=1\textwidth]{img/container_vm_with_kubevirt.png}
    \caption{Overview of VMs and containers managed by using KubeVirt}
    \label{fig:container-vm-kubevirt-architecture}
\end{figure}

The approach of incorporating VMs in Kubernetes not only benefits from the flexible, scalable and mature infrastructure that Kubernetes provides but also from tools and processes that are designed to integrate with or manage the life cycle of Kubernetes resources. This means by using KubeVirt, VMs can be managed similarly to containers, creating a more cloud-native approach to VM management\ \cite{cloud-native}. Furthermore, consolidating the management of VMs and containers into a single platform could offer benefits in terms of reduced architectural complexity and simpler hardware management, but also practical aspects such as reduced required skill set in an organization\ \cite{kubevirt-cncf, kubevirt-storware}. 

The integration of VM management into Kubernetes through KubeVirt represents a significant milestone toward a more agile, efficient, and unified IT infrastructure\ \cite{kubevirt-storware}. However, KubeVirt's practicality in a real-world scenario is yet to be fully assessed. As such, this project aims to evaluate Kubernetes with KubeVirt as a VM management system and compare it with traditional approaches to understand its practicality and limitations.

\section{Problem}
Due to the different use cases for VMs and containers, they have traditionally been managed separately. However, both VMs and containers are still part of the necessary infrastructure many organizations depend on. This has led to fragmented infrastructure with separate management systems and tools to manage VMs and containers with a direct implication of increased architectural complexity and a larger required skill set for operators. Furthermore, many cloud-native technologies are designed for Kubernetes, and as such, by running VMs outside of Kubernetes, these benefits might not be utilized.   

\section{Purpose}
The purpose of this project is to evaluate if Kubernetes, a platform originally designed for container orchestration, is a suitable platform for managing VMs using KubeVirt. This evaluation is important in the field of cloud computing, as it could provide insights into how the management of VMs and containers could be both simplified and unified, but also how the benefits of Kubernetes can be utilized for VMs. 

\section{Goals}
This project aims to explore the integration of VMs into Kubernetes environments using KubeVirt. By comparing how VMs are managed using KubeVirt against traditional VM management systems, this project aims to address possible inefficiencies and complexities that arise when managing VMs outside of Kubernetes. The project will investigate the implications of moving into a KubeVirt-based system, such as the benefits and limitations of using KubeVirt, and how it compares to traditional VM management systems.

\section{Research Method}
This project will use a mixed-methods approach to investigate the research questions. Initially, a literature review will be carried out to understand the current state of the art in VM management systems and the current challenges faced and limitations in managing VMs and containers in separate environments. Following this, both a qualitative and quantitative comparison will be conducted to understand the practical implications of moving into a KubeVirt-based system. The qualitative comparison will be conducted to understand the practical implications of moving into a KubeVirt-based system. It will include collecting feedback by interviewing system administrators and operators with experience in both traditional VM management systems and KubeVirt. The quantitative comparison will be conducted to evaluate measurable performance, scalability, and operational efficiencies that could arise when using Kubernetes with KubeVirt, and will include a set of benchmarks. 

\section{Limitations}
\begin{enumerate}
    \item This project will not investigate any solutions for shortcomings in KubeVirt itself, such as finding alternatives for features that exist in existing virtual machine management systems but not in KubeVirt. As such, this project acknowledges that KubeVirt might not be a complete replacement for traditional virtual machine management tools since it might not offer all the features of an IaaS system. Therefore, only features that exist in KubeVirt and the compared system will be considered when testing. 
    \item This project will primarily focus on the practical implications of moving into a KubeVirt-based system. For this, there are mainly two scenarios that will be considered:
    \begin{itemize}
        \item \textbf{Replacement}: A system uses only a traditional virtual management system. By using KubeVirt, the system will be replaced by a Kubernetes-based system.
        \item \textbf{Unifying}: A system uses both traditional management and Kubernetes. By using KubeVirt, the system will be unified into a single platform for both VMs and containers.
    \end{itemize}
\end{enumerate}

\section{Virtualization Fundamentals}
\label{virtualization}
Virtualization is a technology that enables partitioning or grouping computing resources into operating environments using methods such as partitioning, emulation, and time-sharing\ \cite{virtualization-basics}. Virtualization can occur in various forms including the virtualization of storage, networks and servers. Storage virtualization is achieved by creating an abstraction layer on top of physical storage resources to provide a single interface to storage resources that appear as a single storage device. This allows for a more flexible environment where storage resources can be added or removed without affecting the application layer. Similarly, network virtualization enables the creation of networks with the properties of a physical network, but with its appliances such as routers and switches being implemented in software. This allows for flexibility when managing resources in a network since common practices such as networking partitioning and isolation can be achieved without acquiring additional hardware. Server virtualization, on the other hand, is concerned with creating virtual computing environments that are decoupled from the underlying computing resources. It is often referred to as hypervisor-based virtualization, as it requires a software layer called a hypervisor that is responsible for creating and managing the virtual computing environments \ \cite{virtualization-history}.

A virtual computing environment, more often referred to as a virtual machine (VM), is an emulation of a physical computer and is perceived as a \textit{real} computer by the application running inside it even though the underlying resources are often shared with other VMs. To enable this, a hypervisor divides the hardware into virtualized resources, such as virtual CPUs and virtual memory, and provides an interface to manage these resources. For instance, an application running inside a VM is unaware that the resources it is using are virtualized, and instead interacts with the OS and hardware as if it were running on a physical machine. This is particularly useful for scenarios in which hardware is under-utilized, as it allows for the consolidation of workloads onto a smaller number of physical machines, which in turn reduces the need for acquiring additional hardware.\ \cite{virtualization-basics, virtualization-trends}.

Depending on the required level of abstraction, virtualization can be achieved in different layers, where the three major being hardware-level, OS-level and application-level. For instance, storage and network virtualization both fall under the category of hardware virtualization, as they create an abstraction layer on top of the physical resources. However, in other scenarios, virtualization is not required to be as low-level, such as in application virtualization. Application virtualization is instead concerned with encapsulating an application in a lightweight virtualized environment, such that it is perceived as being run on a specific platform. A common example is the Java Virtual Machine (JVM) that enables programs written in the programming language Java to be run on multiple operating systems\ \cite{application-virtualization}. Figure \ref{fig:hardware-os-virtualization} presents an overview of the differences between the two more common layers of virtualization; hardware and OS virtualization. While the figure does not include application-level virtualization, it is important to note that the virtualization layers are not mutually exclusive and that they can be used in combination. For instance, combining hardware-level and OS-level virtualization creates a flexible environment where a guest process can run inside a guest OS\ \cite{virtualization-basics}.

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/hardware_os_virtualization.png}
    \caption{Overview of the different layers of virtualization}
    \label{fig:hardware-os-virtualization}
\end{figure}

Furthermore, virtualization can be divided into two categories, full virtualization and paravirtualization. Full virtualization is the traditional approach where the entire set of hardware is emulated, and the VM is presented with an indistinguishable image of the underlying hardware. This allows for complete decoupling of the VM from the underlying resources and does not require any modification to the guest OS\ \cite{virtualization-basics, xen-virtualization, hypervisor-xen}. Paravirtualization, on the other hand, is a subset of full virtualization that aims to improve VM performance by reducing the number of resources that are emulated. The VM is presented with an image that is similar, but not identical, to the underlying hardware. As such, the guest OS is aware it is running inside a VM\ \cite{xen-virtualization, hypervisor-xen}.

\subsection*{Benefits and Challenges}
Many of the benefits of virtualization stem from the fact that there exists an abstraction layer above the physical hardware. The abstraction layer decouples software from hardware, allowing for the use of vendors with different I/O in the same environment. This creates a more flexible and scalable computing environment as a whole where hardware is turned into a simple pool of computing resources\ \cite{virtualization-trends}. As such, there are numerous benefits of virtualization, a few of which are listed below:
\begin{itemize}
    \item Resource utilization: Virtualization allows for better utilization of hardware resources since multiple VMs can run on the same physical hardware. This is achieved by sharing the underlying resources between the VMs, such as CPU and memory\ \cite{virtualization-basics}. 
    \item Isolation: VMs can isolate their entire state from one another, including the operating system (OS), filesystem and application stack. This is particularly useful for certain applications with stricter security requirements, such as applications managing sensitive or confidential data. Isolation is also beneficial in scenarios where reproducibility is key, such as in testing and development environments\ \cite{vm-docker-reproducibility}.
    \item Flexibility: VMs are not tied to specific hardware, and can be moved between different physical resources. Flexibility is an especially useful factor when in scenarios where hardware is not fully utilized since VMs can be migrated to other physical machines to balance the load\ \cite{virtualization-basics}.
    \item Management: VMs can be created and destroyed on demand, and can be managed using software tools\ \cite{virtualization-basics}.
    \item Virtual hardware: A VM can be configured with hardware that does not exist in the physical environment, such as a virtual network card or a virtual hard drive. This allows for better support for legacy applications or certain applications that are dependent on a specific hardware configuration\ \cite{virtualization-basics}.
\end{itemize}

However, the portability made possible by virtualization comes at a cost of performance. Since a VM is a complete but emulated system, every instruction issued must be interpreted in software. This means that despite the level of abstraction layer used, there is an inescapable added overhead when using virtualization. A common hypervisor such as KVM (see\ \ref{hypervisors}) introduces an overhead of 5 to 10\%\ \cite{kvm-overhead}, and the overhead can be even larger when using a full virtualization approach.

\section{Understanding Hypervisors}
\label{hypervisors}
A hypervisor is a software layer that enables managing multiple VMs on a single physical machine by allowing multiple operating systems (called \textit{guests}) to share the same hardware\ \cite{virtualization-history, hypervisor-survey}. Hypervisors, often referred to as virtual machine monitors (VMMs) are responsible for creating and managing VMs and for providing the necessary abstractions to the VMs.

Hypervisors can be categorized into two types, \textit{type 1} and \textit{type 2}, based on their operational context and how they interact with the underlying hardware.  While the goals and purpose of type 1 and type 2 hypervisors are the same, there are some core differences. Type 1 hypervisors, commonly referred to as \textit{bare-metal} hypervisors, run directly on top of the physical hardware and do not require an OS underneath. A type 1 hypervisor is therefore scalable to terabytes of RAM and hundreds of CPU cores\ \cite{hypervisor-types-1-2}. However, without any other additional software, it is also responsible to schedule and allocate required system resources to the VMs. Type 2 hypervisors, also known as \textit{hosted hypervisors}, rely on an underlying OS to manage the calls to the CPU and memory resources. The presence of an underlying OS inevitably introduces latency, which is why type 2 hypervisors are often used for development and testing purposes\ \cite{hypervisor-survey}. Figure \ref{fig:hypervisor-types} presents a high-level overview of the difference between a type 1 and type 2 hypervisor\ \cite{hypervisor-survey, hypervisor-types-1-2}.

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/type1_type2_hypervisor.png}
    \caption{Overview of type 1 and type 2 hypervisors}
    \label{fig:hypervisor-types}
\end{figure}

Furthermore, hypervisors can be further categorized into \textit{monolithic} and \textit{microkernelized} hypervisors\ \cite{hypervisor-comparion-2013}. Monolithic hypervisors are built as a single large software stack, managing all the VMs and their resources, and hosting driving for all hardware access. Having all drivers reside within the hypervisor means that the hypervisor will only run on selected hardware. However, it also means no operating system is required at all, as the hypervisor acts as an operating system by supporting all the operations of its VMs. Microkernelized hypervisors, on the other hand, do not require device drivers to be present in the hypervisor. Instead, the drivers are installed in a form of managing VM, often referred to as parent VM. Each VM created by such a hypervisor is thus referred to as a child VM, as it uses the parent VM's drivers to access the hardware. This approach is more flexible, as it can allow the hypervisor to run on any hardware that supports the parent VM's operating system. 

% security rings definition
In the realm of hypervisors, the concept of \textit{security rings} is often used to describe the level of privilege different part of a hypervisor has. A security ring represents a hierarchical protection domain, where the innermost ring (ring 0), usually reserved for the OS kernel, has the highest privilege, and the outermost ring (ring 3), usually reserved for user applications, has the lowest privilege. The hypervisor itself is often placed in ring 0, its VM kernel space in ring 1, and the VM user space in ring 3. Security rings are thus used to define execution level to provide a system to limit the execution capabilities of VMs to regulate access to shared memory. This enabled a strong isolation layer between the VMs, and between the VMs and the hypervisor, thus a key part of a hypervisor's security model\ \cite{hypervisor-comparion-2013, microkernel-monolithic}. Hypervisor security rings are visualized in figure \ref{fig:hypervisor-security-rings}.

    \caption{Overview of security rings in a hypervisor}

\end{figure}

\subsection*{Key Hypervisors}
\label{key-hypervisors}
There are several hypervisors available, each with its own set of features and capabilities, some of which are listed below:

\begin{itemize}
    \item VMware ESXi: A lightweight type 1 monolithic hypervisor often used in enterprise environments. ESXi is known for its ultra-lightweight footprint stretching merely a few hundred megabytes, and its ability to run directly on the hardware without an underlying OS\ \cite{hypervisor-comparion-2013}.
    \item Microsoft Hyper-V: A type 1 microkernelized hypervisor that is part of the Windows Server operating system. As a microkernelized hypervisor, Hyper-V differentiates between parent and child VMs and uses the parent VM to manage hardware access for the child VMs\ \cite{hypervisor-comparion-2013, hypervisor-hyper-v-esxi}.
    \item Citrix Hypervisor: Formerly known as XenServer, is a type 1 microkernelized hypervisor based on the Xen Project. It runs using a modified Linux kernel in a VM, called Dom0, that manages hardware I/O and interacts with the other virtual machines, called DomU guests\ \cite{hypervisor-xen, hypervisor-comparion-2013}. 
    \item KVM: A type 1 hypervisor that is part of the Linux kernel. As such, KVM uses most of the functionalities from the Linux kernel including the scheduler and memory management. KVM has become a popular alternative to proprietary hypervisors, such as VMware ESXi and Microsoft Hyper-V since it is open-source and free to use\ \cite{hypervisor-comparion-2013, hypervisor-kvm}. 
\end{itemize}

\section{VM Management Essentials}
A management system for VMs involves a set of tools and processes that are used to create, deploy, and manage VMs. The management system is responsible for managing the entire life cycle of a VM, including provisioning, deployment, monitoring, and scaling. Not be confused with a hypervisor, a VM management system operates on a higher level and aims to provide an abstraction layer in the form of an interface to allow interaction with the VMs, often enabling the support of a set of hypervisors\ \cite{cloudframeworks-and-hypervisors}. As such, a VM management system unifies a set of hypervisors by allowing them to be managed as if they were homogeneous, typically through a higher-level API, such as a CLI or a web-based user interface. However, certain hypervisor features, such as live migration, could be implemented differently across hypervisors, and it is up to the VM management system to provide a unified interface to manage such features\ \cite{openstack-opennebula-comparison}.

\subsection*{Concepts and Features}
With a higher abstraction layer, a VM management system brings a set of concepts to simplify the management of VMs. Many such features and concepts exist and can be broadly grouped as clustering, availability, and scalability\ \cite{cloudframeworks-and-hypervisors}. 

Clustering involves grouping a set of resources, such as VMs or storage nodes, to provide an image that is perceived as a single resource. By allowing multiple nodes to cooperate and share resources, clustering can allow for a more scalable and performant system. High-performance computing (HPC) is a common example of clustering where computing resources are aggregated to gain performance greater than that of a single computer. This enables HPC clusters to process large datasets and complex calculations that would take an infeasible amount of time to process on a single computer. Moreover, HPC clusters are designed such that they allow for the distribution of computational tasks where larger tasks are divided into smaller subtasks. However, the splitting of tasks is often not transparent to the end user, and the user perceives the cluster as a single resource\ \cite{google-hpc-overview,oracle-hpc-overview}. 

Availability refers to the ability to keep a resource active or running, which is, more often than not, clustered resources. A common practice for cloud computing is managing availability zones (AZs). An AZ groups resources such that each AZ is independent of the other AZs while providing the same service. This allows for outages to be isolated to a single AZ while the other AZs continue to provide the service. Thus, through redundancy and isolation, AZs enable high availability and fault tolerance\ \cite{cloudframeworks-and-hypervisors, azure-availability-zones}. Figure\ \ref{fig:availability-zones} presents an overview of how 3 AZs provide redundant access to the same service. 

Scalability involves technologies to ensure resources can manage an increase or decrease in computational loads. A VM management system often facilitates both scaling up and scaling out. Scaling up, also referred to as vertical scaling, involves increasing or decreasing the performance specification of a resource. For instance, a VM can be scaled up by increasing the number of CPUs or the amount of memory. Scaling out, also referred to as horizontal scaling, involves adding or removing entire resources, such as VMs. This means, that instead of upgrading a single VM, additional VMs are added to the system to handle the increased load. This approach is often regarded as more scalable than scaling up, as it allows for a more granular approach to resource management, and can be down without any downtime. Additionally, by avoiding downtime, the system appears as more available to the end user \cite{scale-up-out-case-study, scale-up-out-ibm}

\subsection*{Cloud Service Models}
A VM management system is often part of a larger cloud computing infrastructure, such as Infrastructure as a Service (IaaS). IaaS is a lower-level cloud service model that aims to eliminate the need to install and manage physical hardware by providing virtualized computing resources, such as VMs, virtual networks and storage. As such, an IaaS platform provides a single interface to manage a wide range of virtualized resources that acts as a foundation for building and managing cloud environments \cite{iaas-overview}.

Other cloud service models include Platform as a Service (PaaS) and Software as a Service (SaaS). They provide a higher level of abstraction than IaaS forming a hierarchical relationship with each other. This means a SaaS service can be built on top of a PaaS service, which in turn can be built on top of an IaaS service. \subsection{Comparison of VM Management Systems}
Managing VMs can be done in many ways, and there are numerous VM management systems available, each with its own set of features and capabilities. Many such systems are a part of a larger platform, such as an IaaS or even as large as a cloud platform. Some of the most prevalent open-source platforms that have been used in the industry are OpenStack, CloudStack, and OpenNebula, and some of the most popular public cloud platforms are Amazon EC2, Azure Virtual Machines, and Google Compute Engine.

\subsubsection*{OpenStack}
An open-source IaaS platform that offers a large set of features to manage VMs, networks, and storage\ \cite{openstack-opennebula-comparison}. OpenStack's architecture is inherently modular, and it is designed to be deployed in a distributed manner\ \cite{openstack-overview}. The architecture includes three major parts:
\begin{itemize}
    \item \textit{Nova}: The compute service that manages VMs and provides an interface to manage VMs. Nova supports all operations regarding the life cycle of a VM, such as creating, deleting, and migrating VMs. Nova is itself divided into multiple components, each employing a \textit{shared-nothing} and \textit{messaging-based} policy, which means every component is independent and communicates with other components using a message queue. 
    \item \textit{Swift}: The object storage service that provides a scalable and redundant storage system for VMs. It is a long-term storage for static data, such as images and backups, and is comparable to Amazon S3.
    \item \textit{Glance}: The image service that provides a catalog of VM images that can be used to create new VMs. 
\end {itemize}

OpenStack supports features such as live migration, snapshot management, and scalability, and is often used in large-scale cloud environments\ \cite{openstack-opennebula-comparison}. However, these features are often hypervisor-specific and require both support from the hypervisor and specific configuration to work\ \cite{openstack-live-migration}.

\subsubsection*{CloudStack}
An open-source IaaS platform and cloud computing software that manages computing, networking and storage resources, and is used to build and manage public and private cloud environments\ \cite{cs-overview}. On a high level, CloudStack is built in a monolithic fashion, encapsulating all the management features into two software packages called the management server and agent. The management server is responsible for managing the life cycle of VMs, including provisioning, deployment, and scaling, and is comparable to a control node. The agent, however, is installed on each node designated to running VMs and is responsible for low-level management tasks of the VMs on that node, such as interacting with the underlying hypervisor. While it is a monolithic system, CloudStack is designed to be horizontally scalable and can be deployed in a distributed manner with multiple management servers\ \cite{cs-overview}. 

CloudStack supports a large variety of hypervisors, such as VMware, KVM, and XenServer, and storage services, such as Amazon S3, allowing it to function as an abridgment layer for computing, network, and storage to unify a wide range of existing infrastructure\ \cite{cs-overview}. Features such as live migration, snapshot management, and scalability are supported but are often hypervisor-specific and require specific configuration to work\ \cite{cs-live-migration}.

\subsubsection*{OpenNebula}
An open-source IaaS platform and toolkit used to provide open, scalable, and comprehensive management of virtualized data centers\ \cite{openstack-opennebula-comparison}. OpenStack employs a modular architecture divided into three layers\ \cite{openenbula-tele-teaching, iaas-comparisons}: 
\begin{itemize}
    \item \textit{Tools layer}: Provides a set of interfaces such as a CLI and web-based user interface to interact with the system, but also high-level management tools such as scheduling.
    \item \textit{Core layer}: Provides the core functionalities of the system, such as host management, VM management, and virtual network management.
    \item \textit{Driver layer}: Provides an abstraction on top of the underlying infrastructure to enable the system to support a wide range of hardware, including networking, storage, and hypervisors. As such, the driver layer is responsible for allocation, deallocation and monitoring of resources, such as VMs and storage. 
\end{itemize}

By dividing the system into these layers, OpenNebula provides modularity that enables the system to be extended and customized to fit specific needs, such as integrating with other systems or using OpenNebula to build a private cloud.

\subsubsection*{Amazon EC2}
Amazon Elastic Compute Cloud (EC2) is a web service that provides on-demand compute resources over the cloud\ \cite{ec2-overview, ec2-overview-web}. EC2 is part of Amazon Web Services (AWS), is designed to be scalable and flexible, and is often used to host general-purpose applications, including web applications, databases and enterprise applications. While the EC2 platform uses Xen-based hypervisors (see\ \ref{key-hypervisors}) under the hood\ \cite{public-cloud-comparison}, it acts as a complete abstraction layer over the underlying hardware and hypervisors and instead provides a rich set of tools to manage the VMs.

This enables EC2 to provide a wide range of VMs, such as general-purpose, compute-optimized, memory-optimized, and storage-optimized VMs\ \cite{ec2-overview}. \subsubsection*{Azure Virtual Machines}
Azure Virtual Machines is a service provided by Azure to manage VMs in the cloud\ \cite{azure-overview}, and is designed to be a cost-effective alternative to on-premises hardware\ \cite{azure-cost-effective}. Azure Virtual Machine provides a set of preconfigured VMs that can be used to create VMs for specific purposes, such as general-purpose or high-performance computing (HPC).

\subsubsection*{Google Compute Engine}
Google Compute Engine is a service provided by Google Cloud to manage computing resources in the cloud\ \cite{gce-overview-web}. \subsubsection{Comparison}

OpenStack, OpenNebula, and CloudStack are all examples of software that can be used to build both private and public cloud platforms. A private cloud, on the other hand, is a cloud platform meant to facilitate external users. This means that the user does not need to worry about the underlying hardware and hypervisors, and can instead focus on managing the VMs. This creates a trade-off where the user has less control and customization but instead gains a more user-friendly and complete system\ \cite{public-private-cloud}.

Both private and public cloud solutions for VM management are designed to be scalable and flexible and support a wide range of use cases (although private cloud solutions scale worse in practice. \section{Containers and Kubernetes: A Paradigm Shift}
\label{containers-kubernetes}
Besides the common cloud service models (IaaS, PaaS, and SaaS), another service model has emerged, called \textit{Container as a Service (CaaS)} \cite{caas}. CaaS is a container-based cloud service model that has gained significant traction due to the lightweight and portable nature of a container, allowing for simple application bundling that can run in a vast range of environments\ \cite{microservices-overview}. 

Containers are often used in conjunction with a container orchestration platform, such as Kubernetes, to manage the life cycle of the containers including deployment, scaling, and monitoring. As such, they are often used to build, test, and deploy applications consistently. A common use case for containers is to run microservices, which is a software architecture that employs a set of small, independent and loosely coupled applications\ \cite{microservices-overview}. 

\subsection{Containers Explained}
\label{containers}
Containers are a form of virtualization that offers a lightweight alternative to VMs. Unlike VMs, containers do not require a hypervisor to run and instead run on top of an OS and share the OS kernel with other containers. By not requiring a hypervisor, containers can run with less overhead than VMs, making their life cycle relatively fast and efficient compared to VMs\ \cite{container-vm-overview, container-k8s-benchmark}. The differences between container and hypervisor virtualization are visualized in figure \ref{fig:container-vm-difference}.

    \label{fig:container-vm-difference}

A container is created from a container image, which is a lightweight executable software package that can run a specific application including dependencies and data files. By encapsulating an application and its dependencies in a single package, the container image becomes portable and independent allowing for the container to run consistently across different environments. A container image is created by describing its parent image, such as an OS image, and the steps to install and configure the application and its dependencies in a file, such as a Dockerfile. A parent image can itself have a parent image, creating a chain of dependencies. Furthermore, to enable caching and efficient storage, a container image can be divided into a series of steps referred to as layers. Each layer represents a change to the parent image, such as adding a file or installing a package. As such, layers act as checkpoints and allow for incremental builds where only the necessary layers are rebuilt when a change is made to the image\ \cite{container-docker-layers, container-dockerfile, container-docker-reproducibility}. Figure\ \ref{fig:container-image-layers} visualizes the concept of container image layers, where each operation represents a change to the parent image.

    \caption{Overview of container image layers}
    \label{fig:container-image-layers}
\end{figure}

Moreover, container images are often stored in container registries. A container registry provides central storage for container images and is used to distribute container images to environments that can run containers, such as a Kubernetes cluster. Common features for a container registry include versioning, access control, image scanning, and features to integrate with software development cycles, such as triggers to notify an external system when a new image is pushed to the registry\ \cite{container-registries, container-registries-redhat}.

The Open Container Initiative (OCI) is a standardization effort that was created to ensure that containers are portable and interoperable across different platforms\ \cite{oci}. The OCI standard defines a runtime specification, image specification, and distribution specification, allowing for a container to be consistent in terms of its runtime environment, image format, and distribution method. Moreover, a container runtime, such as containerd, is used to create, deploy, and manage containers on a node\ \cite{container-runtimes}. Most runtimes, including containerd, implement the OCI runtime specification and can consistently run containers. This suggests that the OCI can be viewed as an abstraction layer, where the OCI runtime specification acts as an intermediary between the container runtime and the OS kernel, and the container image with the distribution specification acts as an intermediary between the container runtime and the container image registry.

\subsection{Orchestration Using Kubernetes}
\label{sec:kubernetes-orchestration}
Kubernetes is an open-source container orchestration platform that manages containers. It maintains a set of abstraction layers to manage the life cycle of containers and to provide a consistent environment for the containers to run in, facilitating tools for scheduling, scaling and networking\ \cite{container-vm-overview}.

Some of the abstractions used in Kubernetes to provide execution environments and communication include \textit{Pods}, \textit{Services} and \textit{Deployments}. A Pod is the smallest deployable unit in Kubernetes, which groups one or more containers into a single unit that can be run on nodes. This means that Kubernetes ensures that all containers in a Pod are run on the same node. Encapsulating several containers allows for the logical grouping of coupled containers that share the same resources, such as networking and storage. This is commonly used when a Pod contains a main application container and a secondary container that provides logging and monitoring, called a sidecar container\ \cite{k8s-docs-pods}. A Service represents a single-point-of-entry communication layer for a set of Pods and is used to provide a consistent way to access the Pods, such as load balancing. Services are especially useful in scenarios where Pods are fungible, meaning that the requesting client does not care which Pod it communicates with, as it creates an abstraction layer that hides the details of the Pods. This decouples frontends from backends and allows for a more scalable and resilient system. Furthermore, services are also useful for rolling updates, as they can be used to ensure that a certain part of the traffic is routed to the new version of the application\ \cite{k8s-docs-services}. Contrary to Pods and Services, a Deployment is a higher-level abstraction on top of Pods that is primarily used for replication control. A Deployment does not represent an executable resource but instead describes a desired state in which a set of Pods should be. This includes the number of replicas and update strategy. As such, a Deployment proves useful when a set of Pods should be managed as a single unit, such as when scaling up or down, or when deploying a new version of an application\ \cite{k8s-docs-deployments, k8s-overview}. 

Furthermore, to support stateful applications Kubernetes mainly uses \textit{PersistentVolumes}, \textit{PersistentVolumeClaims} and \textit{StatefulSets}. A PersistentVolume defines how and where persistent data is stored. Since there are many different ways to store data, such as local storage and network storage, a PersistentVolume is used to abstract the underlying storage. As such, each PersistentVolume is backed by a storage class, which defines the parameters and configuration to interact with the storage provider. A PersistentVolumeClaim, on the other hand, connects the PersistentVolume to a Pod. It acts as the bridge between the persistent storage and the execution environment effectively decoupling the Pod from the underlying storage. Contrary to PersistentVolumes and PersistentVolumeClaims, a StatefulSet is not used to manage the storage and is instead comparable to a stateful Deployment. A StatefulSet manages the life cycle of a set of Pods such that they are run predictably with their own unique identity. A common use case for a StatefulSet is scenarios that require each Pod to maintain a specific role, such as a primary or secondary database. A StatefulSet ensures that Pods maintain their role and that they are started in a predictable order, such as starting the primary database before the secondary database\ \cite{k8s-docs-statefulsets, k8s-overview}.

Figure \ref{fig:k8s-overview} presents a brief overview of the Kubernetes abstractions and how they interact with one another. 

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/kubernetes_overview.png}
    \caption{Overview of Kubernetes abstractions}
    \label{fig:k8s-overview}
\end{figure}
 

In certain scenarios, the built-in abstractions are not enough, and custom behavior is needed. For such scenarios, Kubernetes provides a way to extend its abstractions through Custom Resource Definitions (CRDs). A CRD is a form of extension for Kubernetes resources that allows for the creation of custom resources. It is used to extend the existing abstractions to provide additional features and capabilities that allow for a more feature-complete platform when custom behavior is needed. For instance, a CRD can be used to enforce certain fields to be specified to create the resource by applying custom validation rules\ \cite{k8s-crds, k8s-overview, k8s-crd-webiste}.

There are many other abstractions in Kubernetes, each with its own set of features and capabilities that together create a feature-complete platform for container orchestration, but presenting them all in this report falls out of the scope of this project.

Kubernetes is inherently declarative, meaning that instead of issuing commands to perform a specific task, such as starting a container, Kubernetes is given a desired state for a set of resources, and it is responsible for ensuring that the current state of the resources matches a desired state. For this reason, Kubernetes provides auto-restart, re-schedule, and replication control to ensure a resource stays in the desired state. By specifying the desired state, an entire application can be contained in a single configuration file, including its container, networking, storage and security requirements\ \cite{container-vm-overview, k8s-overview}. The entity responsible for ensuring the desired state is referred to as a controller, and there are many different types of controllers depending on the resources they manage. For instance, a Deployment controller is the entity responsible for ensuring that the number of Pod replicas in a Deployment matches the desired amount. Moreover, managing resources using a configuration file and a controller is referred to as the \textit{Controller pattern}, and is a common design pattern in Kubernetes as it provides a consistent and predictable way to manage resources \cite{k8s-docs-controllers}. 

Architecture-wise, Kubernetes uses a client-server approach. The server, often referred to as the control plane, is responsible for managing the state of the cluster. It is responsible for scheduling, scaling and networking by managing system-related services such as the API server, scheduler, and controller manager. The API server can be viewed as the primary service for a Kubernetes cluster since it represents the gateway for all management tasks for the cluster by receiving and processing requests from the client. The client, often referred to as the worker node, is responsible for running the containers. The worker node consists of two primary components: the kubelet and the kube-proxy service. The kubelet is a service that wraps a container runtime and runs containers according to requests by the control plane\ \cite{k8s-docs-kubelet, k8s-overview}. The kube-proxy service, on the other hand, manages the network communication for the containers. It can manage streams and forwarding rules for traffic using TCP, UDP or STCP\ \cite{k8s-overview, k8s-docs-kubeproxy}. 

To interact with Kubernetes from the perspective of a user, a client is used to send requests to the API server. The client can be a CLI, such as kubectl, a web-based user interface, or a programmatic interface such as a REST API. The CLI tool kubectl is the most common way to interact with Kubernetes and is used to create, delete, and manage resources in the cluster\ \cite{k8s-overview}. 

Through its abstractions, life cycle management and declarative client-server architecture, Kubernetes provides a consistent and feature-rich platform for container orchestration. This has led to wide adoption of Kubernetes, and as such, it is often used in large-scale cloud environments\ \cite{k8s-it-administration}. 

\section{Containers vs. Virtual Machines}
Virtualization has enabled the commoditization of computing resources due to the benefits of being able to map virtual hardware to physical hardware (see\ \ref{virtualization}) and has been key in enabling cloud computing. It has enabled the creation of flexible and scalable computing environments and has been used to build and manage large-scale cloud platforms. Virtualization has mainly brought two technologies: hypervisors and containers. Hypervisors are used to create VMs, including all associated aspects such as an OS. Containers, on the other hand, are used to create a lightweight and portable environment for an application to run in and are often run inside VMs. As such, a container comes with less overhead when compared to a hypervisor. However, by its nature, a VM can provide a more isolated environment than a container, leading to a more secure environment. Thus, the benefits of containers do not come without a cost and there is a trade-off between the two technologies; one is lightweight, portable, and quick to start while the other is more independent and secure\ \cite{container-vm-overview, container-vm-comparison}. 

\subsection{Comparative Overview}
While containers and hypervisors are both technologies brought by virtualization, they operate on distinct levels of abstraction. Containers operate on a higher level than VMs using OS-level virtualization, which leads to the OS kernel with other containers. A hypervisor on the other hand can virtualize the entire hardware stack, including the entire OS, depending on the type of hypervisor used (see\ \ref{hypervisors}). As such, the two technologies have different use cases and are often used in conjunction with each other\ \cite{container-vm-comparison}.

\label{container-vm-isolation}
The main difference between containers and VMs is the level of isolation they provide. Containers share the OS kernel with other containers while VMs run on their OS. While this means a container can share resources more efficiently, it also means that a container is subject to a larger attack surface, as a vulnerability in the OS kernel can affect all containers running on the same host. VMs, on the other hand, reduce the attack surface by running more independently, meaning a compromised VM is less likely to affect other VMs running on the same host\ \cite{container-kata-security}.

Another aspect of isolation is how co-located containers and VMs can share resources. When sharing resources such as CPU and memory, containers are vulnerable to the noisy neighbor problem. The noisy neighbor problem occurs when a resource-sharing mechanism, such as a scheduler, is unable to provide a fair share of resources. Some scenarios include when a container can consume all available resources due to a heavily loaded application or when a container intentionally consumes all available resources to starve other containers. Solutions exist to mitigate the noisy neighbor problem, including limiting the number of resources a container can consume through OS-level features such as \textit{cgroups}. A cgroup is a feature in the Linux OS designed to limit, prioritize and isolate resource usage of a set of processes\ \cite{cgroups, vm-noisy-neighbor, container-vm-scale}. VMs, on the other hand, can provide a fair share of resources, as they operate in a more isolated environment, which leads to more resilient and predictable performance. However, the isolation does not mitigate the noisy neighbor problem for all resource types. For instance, a VM is still vulnerable to the noisy neighbor problem when sharing network-related resources such as network bandwidth and network-based storage\ \cite{container-vm-comparison}. 

Overcommitment is a technique used to improve packing efficiency, often referred to as \textit{consolidation ratio}. Containers use the overcommitment mechanisms built into the OS itself, such as CPU overcommitment by multiplexing and memory overcommitment through virtual memory and memory swapping. Hypervisors, on the other hand, map virtual hardware devices to physical hardware, creating virtual computing resources that are not directly tied to the physical resources. This works well for the CPU by creating virtual CPU (vCPU) cores. However, memory overcommitment is a more complex issue in hypervisors than in containers due to the way hypervisors allocate memory. Unlike containers, hypervisors assign a static amount of memory to a VM when it is created, and only reclaim the memory when the VM is deleted. Methods such as memory ballooning exist to address the limitation, which is a technique that treats memory as a shared resource across VMs and allows the hypervisor to reclaim memory from VMs when it is needed elsewhere. It is implemented as a driver that reserves unused memory (deflation) on a VM, rendering it unavailable to other processes running in the VM. The memory pages on the physical memory mapped to these pages in the VM are allocated to other VMs (inflation). This allows for VMs to share unused memory\ \cite{vm-ballooning}. However, it is important to note that techniques like memory ballooning do not directly solve memory overcommitment for VMs, but instead reduce the need for it by managing the available memory more efficiently\ \cite{container-vm-comparison, vm-overcommitment}. Figure \ref{fig:memory-ballooning} visualizes the processes of deflating the memory of a VM and how it is used in another VM. 

\begin{figure}
    \centering
    \includegraphics[width=0.9\textwidth]{img/memory_ballooning.png}
    \caption{Overview of memory ballooning}
    \label{fig:memory-ballooning}
\end{figure}

\label{container-vm-live-migration}
Live migration is a technique used to move a running resource from one host to another without interrupting the services provided by the resource. The technique is often used to balance the load or to perform maintenance on the hosting nodes. For VMs, it is a mature and widely technique for large data centers. Live migration works by copying memory pages from the source host to the destination host and then transferring the control of the VM from the source hypervisor to the destination hypervisor. Since VM disks are often stored on a shared storage, such as a Storage Attached Network (SAN), the VM disk is not transferred but rather reattached after the live migration is finished. For this reason, the VM can safely continue to run after its running state has been transferred to the destination host\ \cite{vm-live-migration}. Live migration for containers, on the other hand, is not as mature as for VMs. Unlike VMs, containers require OS-level process migration, which is less reliable. However, several projects aim to provide live migration for containers, such as Checkpoint-restore in Userspace (CRUI). CRUI operates by checkpointing the state of a container, such as its memory and file descriptors, and then restoring the state on another host. While this is considered as container live migration, it is more comparable to snapshot and restore operations than a live migration operation, as CRUI freezes the container during the checkpointing, and converts the state of a container to a collection of files on disk, that can be used to restore the container on another host\ \cite{container-criu-overview, container-criu-linux, container-criu-web}. Another approach for live migrating containers involves terminating them on one host and restarting them on another. This strategy assumes the container is stateless or utilizes shared storage for its state, as is often used in Kubernetes since it does not provide container live migration. Instead, Kubernetes facilitates persistent storage through Persistent Volume Claims (PVCs), essentially allowing a user to map portions of the container's state to shared storage, and assumes the rest can be discarded when a container is terminated\ \cite{k8s-overview}.

\subsection{Decision Factors}
The choice between containers and VMs depends on the specific use case, and there are several factors to consider when deciding which technology to use. Some of the core factors are security, scalability, and legacy support\ \cite{container-vm-comparison}. 

Security is a key factor when deciding between containers and VMs. Using containers means sharing the OS kernel, which implies that a vulnerability in the OS kernel could affect all containers running on the same host. Containers are thus, from an attack-surface perspective, more prone to some vulnerabilities than VMs, making VMs more suitable in scenarios that require a higher level of isolation. For instance when an application manages sensitive or confidential data\ \cite{container-vm-decision-redhat}. Moreover, sandboxing is a security mechanism that is used to restrict an execution environment such that it can safely run untrusted code without affecting the rest of the system. For instance, when analyzing the effect of malware a sandbox environment ensures that the malware is not able to escape the restricted environment. Sandboxing is done in multiple layers, including limiting the resources a process can access, such as memory and CPU, and limiting the system calls a process can make. As such, a sandbox environment requires a high level of isolation, which makes VMs a natural fit\ \cite{vm-decision-kong, sandboxing}.

Configuration drift is another aspect to consider regarding security. It is a phenomenon that arises when the configuration of a system changes over time due to several factors including manual intervention and software updates, and can lead to an undefined state in a system that could create security holes\ \cite{configuration-drift-tripwire, configuration-drift-wiz}. Containers, as described in\ \ref{container-vm-live-migration}, are often run in a stateless manner using a single configuration file to describe the container, such as a Dockerfile. By being able to discard the state and recreate a container from a configuration file, the risk of configuration drift is reduced. As such, containers can provide a more consistent environment than VMs. Moreover, containers represent a more standardized and portable environment, using layers of dependencies, see Fig. By standardizing the environment, external tools are more easily able to provide security features such as vulnerability scanning\ \cite{container-vm-decision}.  

When deciding between containers and VMs scalability is an important factor. One of the main benefits of containers is their ability to share the underlying OS. However, as described in\ \ref{container-vm-isolation}, this also means that containers are more prone to the noisy neighbor problem, where co-located applications can cause performance interference. While this issue is related to isolation and security, it is more prominent in larger-scale environments, especially in multi-tenant environments\ \cite{container-vm-scale, container-vm-decision}.

Scalability is also related to cloud-native applications, which are applications that are designed to run in the cloud\ \cite{cloud-native}. A cloud-native application is characterized by being horizontally scalable, stateless, and resilient, and is often built using microservices, see\ \ref{containers-kubernetes}. Cloud-native applications thus require small and lightweight environments, making containers with an orchestration platform such as Kubernetes a natural fit.

Legacy support is another factor to consider when deciding between containers and VMs. Both VMs and containers can provide environments for legacy applications, such as applications that require a specific OS version or a specific set of dependencies. However, certain legacy applications are not able to run in a container due to the way they are designed, such as applications that require a specific OS kernel or a specific set of hardware. In such cases, a VM is often a better fit, as it enables virtualized hardware and a complete OS to be run\ \cite{container-vm-decision,vm-decision-microsoft}.

\subsection{Towards Stronger Container Isolation}
Containers are often used in conjunction with VMs due to the lack of strong isolation properties out of the box, see\ \ref{container-vm-isolation}. However, several projects aim to provide the isolation properties of a VM and the lightweight properties of a container, such as Firecracker, Kata Containers, and gVisor.

Firecracker is a KVM-based hypervisor that creates lightweight VMs, called microVMs. Compared to a traditional VM, a microVM can start in a fraction of a second using only a fraction of the memory, comparable to that of containers. This is achieved through heavy optimization both in the hypervisor and in the guest functionality, such as reduced overhead through lightweight I/O and a stripped-down guest OS\ \cite{container-runtimes-overview, container-firecracker}.

Kata Containers is a project that, similar to Firecracker, aims to provide lightweight VMs that perform like containers while also providing strong isolation properties. Unlike Firecracker, Kata Containers uses both VMs and containers and replaces the container runtime, such as containerd (see\ \ref{containers}) with the Kata runtime. The Kata runtime implements the OCI runtime specification to be interoperable with container ecosystems. However, instead of running the container directly on the host, the Kata runtime creates a VM, in which the container runs. This provides the isolation properties of a VM and the lightweight properties of a container. Kata Containers can achieve this by heavily optimizing the VM and its creation process towards running containers, as this is the only purpose of the VM\ \cite{container-runtimes-overview}. 

Another approach to strong container isolation is gVisor, a user-space kernel that provides a secure environment for the container. Unlike both Firecracker and Kata Containers, gVisor does not use VMs to enhance the isolation properties but instead operates as an extension around the container. gVisor consists of two core components, \textit{Sentry} and \textit{Gofer}. Sentry is a user-space kernel that intercepts and handles system calls made by the container and can provide a secure environment for the container. Gofer is a file access component that provides the container with file access. Separating the file access from the syscall access allows for a more secure environment. For instance, a compromised Sentry does not imply read or write access to any file\ \cite{container-gvisor-firecracker}.

In summary, Firecracker stands out among the three since it does not use container runtimes. As such, Kata Containers and gVisor are more comparable. Figure \ref{fig:secure-containers-overview} visualizes the differences between how Kata Containers and gVisor wrap a container to provide a more secure execution environment. However, due to the varying added overhead, they are yet to be widely adopted\ \cite{container-runtimes-overview}.

\section{Extending Kubernetes with KubeVirt}
KubeVirt is a novel project that aims to provide a unified platform for managing both containers and VMs using Kubernetes. It is designed to naturally blend in with the Kubernetes ecosystem and provide a consistent interface to manage the life cycle of VMs, such as deployment, scaling, and monitoring. KubeVirt being built on top of Kubernetes means that it does not implement all the components found in a traditional VM management system, such as the scheduler, but instead uses the mature and feature-rich services the Kubernetes ecosystem provides\ \cite{k8s-scheduler}. By using CRDs (see\ \ref{containers-kubernetes}), KubeVirt can extend the Kubernetes API to provide a specification for VMs to support features commonly found in traditional VM management systems, such as live migration, scaling and snapshot management. Moreover, CRDs allow KubeVirt to enforce validation on the resources to prevent invalid configurations, such as a VM that references a non-existing storage disk\ \cite{kubevirt-architecture, kubevirt-validation}.

The design pattern chosen for KubeVirt's VM management is heavily inspired by other Kubernetes resources. A VM in KubeVirt is split into two resources called \textit{VirtualMachine} and \textit{VirtualMachineInstance}. The VirtualMachine resource is comparable to a StatefulSet and is used to describe the desired state of a VM, such as the VM's desired running state, CPU and RAM requirements, as well as cloud-init configuration. The VirtualMachineInstance, on the other hand, is comparable to a Pod and is created by the VirtualMachine resource to represent the running state of the VM. As such, a non-existing VirtualMachineInstance represents a non-running VM, and an existing VirtualMachineInstance represents a running VM. The VirtualMachineInstance contains runtime information about the VM such as the VM's IP address, running state and resource usage. 

Similar to other Kubernetes resources, such as Deployments and StatefulSets, KubeVirt uses controllers to manage the life cycle of the VMs. These controllers are together responsible for creating the environment in which a VM can be run, but also that the desired state of VirtualMachine resource matches the current state of its VirtualMachineInstance resource. For instance, when VirtualMachine is updated with requirements for CPU cores, the KubeVirt controllers are responsible for updating the VirtualMachineInstance to match the new specification\ \cite{k8s-docs-kubevirt, kubevirt-architecture}.

Live migration and snapshots are implemented as CRDs, allowing for a declarative interface to manage these features. This is in stark contrast to traditional VM management systems, where live migration and snapshot management are commonly implemented as imperative operations. For instance, by creating a LiveMigration resource, KubeVirt initiates a live migration operation for the referenced VirtualMachine resource\ \cite{kubevirt-snapshot,kubevirt-live-migration}. This is comparable to the binding operation process for a PersistentVolume and its corresponding PersistentVolumeClaim, where the binding takes place when a PVC is created\ \cite{k8s-persistent-volumes}

An additional benefit of KubeVirt being built on top of Kubernetes is that KubeVirt-related resources can be managed using the kubectl CLI tool. This allows for the declarative creation, modification and deletion of VMs using the same interface as for other Kubernetes resources. However, certain operations are not possible to perform using kubectl. For instance, accessing serial or graphical consoles of a VM is not possible using kubectl. To enable these operations, KubeVirt provides a complementary CLI tool called \textit{virtctl}. While virtctl primarily enables console access, it also provides convenience features such as live migration and snapshot management\ \cite{kubevirt-virtctl}.

\subsection{KubeVirt Technical Overview}
\label{sec:kubevirt-technical-overview}
On a technical level, KubeVirt's VMs are hosted inside Kubernetes Pods. As described in \ref{sec:kubernetes-orchestration}, a Pod is the smallest deployable unit in Kubernetes. KubeVirt does not create a new abstraction for an execution environment and instead encapsulates the VM within a Pod. This brings many benefits as it allows for seamless communication between the VM and the Kubernetes ecosystem since it complies with Container Network Interface (CNI). For instance, a VM can be exposed to the network using a Service or use PersistentVolumeClaims to provide persistent storage. 

KubeVirt consists of four main components to enable the running of VMs in Kubernetes: \textit{virt-controller}, \textit{virt-handler}, \textit{virt-launcher} and the \textit{libvirt daemon}. The virt-controller and virt-handler are responsible for managing the life cycle of the VMs, where the virt-controller is responsible for creating the Pod in which the VM will be hosted, and the the virt-handler is responsible for creating the VM itself. The virt-launcher, on the other hand, manages the runtime environment for a VM, including starting, stopping and monitoring the VM. Finally, the libvirt daemon (libvirtd) provides an interface to manage the VMs hosted by the underlying KVM hypervisor. The relationship between the three components is visualized in figure \ref{fig:kubevirt-components}, where the virt-controller operates closest to Kubernetes API-server, virt-handler operates as a DaemonSet on every node to communicate with the underlying hypervisor, and virt-launcher operates as a sidecar container in every Pod to manage the runtime environment of the VM\ \cite{kubevirt-architecture, k8s-docs-kubevirt}.

\subsection{KubeVirt and GitOps}
GitOps is a set of practices that enables using Git as a single source of truth for the desired state of the system, and it thus is heavily tied to the declarative nature of Kubernetes, as the desired state of the system is described in configuration files. By managing the infrastructure in Git, resources can be version controlled, and any changes to the system can be tracked and audited. This creates a developer-centric workflow, where developers can manage the infrastructure using the same tools and processes they use for managing application code\ \cite{gitops-evolution}. Moreover, since GitOps is used for systems to manage their resources declaratively, it is a natural fit for KubeVirt. As such, KubeVirt opens up the possibility of managing VMs using GitOps, meaning that VM specification and life cycle management can be version-controlled and audited\ \cite{gitops-kubevirt}. Being easily integrated with GitOps further indicates that KubeVirt can unify the management of both containers and VMs using Kubernetes.

\subsection{Comparison with Traditional Approaches}
KubeVirt does not represent a new virtualization or hypervisor technology but instead leverages QEMU and KVM to run the VMs, which means KubeVirt is fundamentally similar to traditional VM management systems, such as OpenStack, CloudStack, and OpenNebula\ \cite{kubevirt-white-paper-platform9}. Therefore, KubeVirt can provide the same level of isolation and security as traditional VM management systems, as it uses the same underlying hypervisor technology. However, KubeVirt is not as feature-rich, as it is not designed to be a complete replacement for traditional VM management systems. For instance, OpenStack not only manages VMs and their life cycle but also provides additional services, including host provisioning and data warehousing\ \cite{kubevirt-tech-comparison}. KubeVirt, on the other hand, is primarily designed to manage VMs and their life cycle such that it can provide a unified platform for managing both containers and VMs. As such, KubeVirt integrates well with Kubernetes to provide a seamless experience for VM management. For instance, VMs created with KubeVirt support CNI plugins and integrate with PersistentVolumeClaims to provide storage\ \cite{kubevirt-architecture, kubevirt-network-plugin, kubevirt-disks}.

\section{Related Work}
\label{literature-study}
\subsection{Current challenges}
The Cloud Native Computing Foundation (CNCF) presents in a report some of the challenges of managing VMs using a traditional IaaS platform and uses OpenStack as an example\ \cite{kubevirt-cncf}. While OpenStack has been a popular choice for managing VMs, it has been criticized for being complex and difficult to manage, as well as having a steep learning curve. They argue that the complexity of OpenStack is not always needed and that it is possible to manage VMs using a simpler and more efficient platform, such as Kubernetes with KubeVirt. 

Additionally, the CNCF report highlights the importance of having a unified platform for managing both VMs and containers, as it could reduce the complexity of the underlying technology stack and management costs. These claims are further supported by a report by Kubacki\ \cite{kubevirt-storware}. He argues that managing VMs using traditional VM management systems alongside Kubernetes leads to unnecessary costs, such as maintaining personnel and skill sets for two technologies. Moreover, by allowing the coexistence of VMs and containers in a single platform, KubeVirt could potentially reduce any barrier of entry around managing VMs for organizations that are already using Kubernetes for container orchestration, or act as a catalyst for organizations considering adopting Kubernetes. 

\subsection{KubeVirt challenges}
While KubeVirt is presented as a solution to the challenges of managing VMs using traditional VM management systems, it is important to consider the challenges of using KubeVirt itself. A study by Trakulmaiphol and Piromsopa presents some shortcomings when migrating from a traditional VM management system to KubeVirt\ \cite{kubevirt-migrate-issues}. One of the issues identified is the management of IP addresses, particularly when maintaining static IP addresses in a dynamic IP environment, such as Kubernetes. In the study, they developed a framework to solve some of the issues on the application level, such as DNS queries, and provided a set of guidelines related to KubeVirt's unique features, such as the Multus Container Network Interface. 

A common practice when managing Kubernetes clusters is to deploy Kubernetes nodes on VMs created by a VM management system. This means that when using KubeVirt, VMs are created inside other VMs leading to nested virtualization. Ye et al.\ argue in a study that nested virtualization raises security concerns due to the introduction of a larger trusted computing base (TCB)\ \cite{free-turtles}. By using a larger TCB, the size of the system's attack vector increases leading to an increased risk for vulnerabilities. For this reason, nested virtualization might not be suitable when strict security is required, such as applications managing sensitive or confidential data. This study is particularly relevant as it highlights a potential limitation if KubeVirt is simply seen as another Kubernetes resource, and not as a VM management system. 

\subsection{Usability of Kubernetes and KubeVirt}
Using Kubernetes as a VM management system is a relatively new concept, and there is limited research on the usability of KubeVirt. However, several studies have evaluated the usability of Kubernetes, which is relevant as KubeVirt is an extension built on top of Kubernetes.

Kubernetes API is inherently declarative, meaning an approach that promotes the usage of documents that describe a desired state of a system. A study by Sonninen investigates many of the possible benefits of using a declarative approach in system architecture\ \cite{declarative-benefits}. The study found that a declarative approach facilitates GitOps, a way to manage infrastructure using files in Git, which simplifies system and resource management and enables easier rollbacks. Using Git as a source of truth for the desired state of the system has the added benefits of version control, audit and collaboration. These benefits were exemplified when CERN migrated the internal CMS project from a VM cluster to Kubernetes, as described in the two studies they published. The first study conducted by Imran et al.\ describes how the Kubernetes migration enhanced the overall system management compared to using a VM cluster\ \cite{k8s-cern}. While many of their experienced benefits stem from adopting containers instead of VMs, the study still suggests Kubernetes provided many advantages, such as reduced operational costs. The other study conducted by Badran presents the practical benefits they experienced by using Kubernetes and is more focused on the adoption of a declarative approach for system management\ \cite{k8s-cern-techreport}. It describes how using Kubernetes instead of a VM cluster allowed them to craft reusable and composable configurations that enabled their entire infrastructure to be managed as code. 

The benefits of moving into Kubernetes are further exemplified in a study by Gelley\ \cite{k8s-cloud-foundry}. It presents a case study where an organization moved its business application from a Platform as a Service (PaaS) provided by Cloud Foundry into Kubernetes. The study not only describes how Kubernetes provided tremendous flexibility in terms of configuration and deployment compared to the previous PaaS but also how the organization was able to leverage the Kubernetes ecosystem to provide additional features such as auto-scaling and monitoring. Moreover, by moving into Kubernetes, possible future migration to other platforms was made simpler due to how widely adopted Kubernetes is. 

User experience in Kubernetes was investigated in a study by Packard et al.\ that gathered several scenarios in which projects were built and managed using Kubernetes\ \cite{k8s-user-experience}. Many aspects were considered in the study, including administration such as managing persistent storage and limiting user resources with quotas, but also developer experience such as building and deploying applications. The study found that while system administrators face a learning curve when using Kubernetes, such as understanding the Kubernetes API and the ecosystem, users without prior knowledge of Kubernetes were able to successfully deploy applications. This finding is important to consider when evaluating KubeVirt, as it suggests that KubeVirt is a viable option even for users without prior knowledge of Kubernetes. Another study that investigated the user experience of Kubernetes was conducted by Boronin\ \cite{k8s-migration-king}. The study presents a case study where an organization faced a migration from a traditionally monolithic system architecture to a more cloud-native design. The study is relevant as it not only describes the challenges or benefits of Kubernetes itself but also considers the organizational impact of migrating to Kubernetes. This involves transferring skill sets among personnel, and the impact on the organization's culture.

In summary, these studies suggest that the benefits of using Kubernetes as a platform are not directly tied to what kind of resources are managed, but rather to the platform itself. Thus, when evaluating KubeVirt, it is important to not only consider benefits of KubeVirt itself but also the benefits that follow by adopting Kubernetes. 

\subsection{Performance Evaluations}
Regarding performance, several studies have evaluated popular systems that can manage VMs. For instance, open-source alternatives such as OpenStack and CloudStack, or proprietary systems such as Amazon EC2. 

A study by Kang et al.\ compared the performance of OpenStack and Amazon EC2 using a wide range of configurations and analyzed diverse aspects based on different hypervisor, storage and networking configurations\ \cite{openstack-ec2-comparison}. Apart from the performance comparisons, the study also investigated performance impact when using more advanced features, such as Infiniband networking instead of Ethernet. The study showed mixed results, where OpenStack outperforms EC2 in some scenarios such as when using Infiniband networking, but EC2 tends to outperform OpenStack in parallel processing. As such, the study concluded that component-wise, each system has strengths. This study brings with it relevant perspectives as it highlights many of the aspects that are important to consider when evaluating the performance of VM management systems.

Another study by Mullerikkal and Sastri compared OpenStack and CloudStack regarding implementation complexity, overall stability and performance\ \cite{cs-os-comparison-2009}. The performance tests were designed to measure disk- and network I/O, and not CPU or memory usage. The study found that while OpenStack had a more complex implementation, it was more stable. Furthermore, by using a set of benchmarks it found that OpenStack had a better overall performance compared to CloudStack. However, a few years later, a complementary comparison between OpenStack and CloudStack was conducted by Paradowski et al.\ \cite{cs-os-comparison-2013}. The comparison focused on performance and used several benchmarks such as deployment time, deletion time, and CPU utilization during deployment and deletion, and revealed significant and quantifiable differences between the two platforms. These tests were designed to represent user interactions with the system. However, the scale of the tests used in the comparison was relatively small and did not explore how the systems manage VMs under higher load. Analyzing the performance of a VM management system under higher load is important, as it can reveal how well a system scales when the ratio between VMs and nodes increases. 

In contrast, a later study by Amaral benchmarked KubeVirt uses a setup with a large number of VMs relative to the size of the nodes, aiming to explore the limits of KubeVirt's control plane (scheduler etc.)\ \cite{ibm-kubevirt-scale}. The study found that, while KubeVirt was able to create all VMs, the control plane suffered a heavy load, indicating a possible large overhead when KubeVirt provisions a large number of VMs. This finding is important to consider when evaluating KubeVirt, as it indicates that KubeVirt may not yet be suitable for managing a large number of VMs.

In terms of network performance, a study conducted by Kolahdouzan investigated the networking traffic pattern in KubeVirt and OpenStack by performing networking performance tests to evaluate potential VM overhead\ \cite{kubevirt-openstack-network}. The study found that the network overhead was significantly larger for scenarios in which VMs reside on separate nodes. However, while the study found overhead for both KubeVirt and OpenStack, it found that OpenStack had a considerably larger overhead. While this project will not focus on networking performance, it is important to consider the findings of this study when evaluating KubeVirt.

Hypervisors are a key component of VM management systems, and several studies have evaluated the performance of hypervisors. A study by Hwang et al.\ evaluated the performance of a range of hypervisors including KVM, Hyper-V and Xen\ \cite{hypervisor-comparion-2010}. The study found that applications and assigned resources had a significant impact on the overhead of the hypervisors. However, the study concluded that no single hypervisor was the silver bullet and that the choice of hypervisor should be based on the specific use case. Reviewing the performance of hypervisors is important when evaluating KubeVirt, as KubeVirt uses KVM as its hypervisor.

\section{KubeVirt}
The following is a brief introduction to KubeVirt, its most relevant features and how it compares to previous VM management systems. If you are already familiar with KubeVirt and its machinery, you can skip this section.
\\~\\
KubeVirt is an extension to Kubernetes that allows for the creation of virtual machines (VMs) and is designed to provide a similar experience to managing any other resource in Kubernetes, such as Pods or Deployments. This means that VMs are defined in YAML files and integrate with the command line tool \textit{kubectl}. KubeVirt defines its resources as a \textit{Custom Resource Definition} (CRD) in Kubernetes, where the most prominent resources are \textit{VirtualMachine} and \textit{VirtualMachineInstance}. The relationship between them is comparable to that of a Deployment and a Pod, where the Deployment defines a desired state and a Pod represents the running state. See References below for more information. 

Certain VM operations are not possible using kubectl, such as accessing the VM using SSH or VNC. For this reason, KubeVirt provides a complementary CLI tool called \textit{virtctl}. Virtctl is thus required to complete the tasks, and you should have already been provided with a link to install it. 

Being built on top of Kubernetes allows the VMs to integrate with other resources, including the Service and PersistentVolume resources. For instance, a VM can be exposed to the network using a LoadBalancer Service or manage storage dynamically using a persistent storage provisioner. However, since a VM requires a disk image (such as a QCOW2 image), it means that a PersistentVolume must point to a folder with a disk image in it. For this reason, manually creating a PersistentVolume for a VM can be tedious. Instead, KubeVirt primarily provides two alternatives for managing disks that are presented below. See the References section for more information.
\begin{itemize}
    \item \textbf{containerDisk}: A disk that is created at every boot. It is not persistent and is similar to a container.
    \item \textbf{DataVolume}: A disk that is persistent and initialized when the VM is created. It uses PersistentVolumeClaim and a PersistentVolume under the hood where it injects a disk image. 
\end{itemize}

\section{Basic usage}
The first task is about creating a VM using KubeVirt. You will need to create a VM and then access it using SSH or VNC. You can use a containerDisk for simplicity. 

\subsection*{Manifests}
Below is a minimal manifest for creating the VM using a containerDisk. The credentials for the Cirros image are user \textbf{cirros} and password \textbf{gocubsgo}.
\begin{lstlisting}[language=yaml]
    apiVersion: kubevirt.io/v1
    kind: VirtualMachine
    metadata:
      name: my-vm
    spec:
      running: true  
      template:
        spec:
          domain:
          cpu:
            cores: 1
          devices:
            disks:
              - name: containerdisk
                disk:
                  bus: virtio
          resources:
            requests:
            memory: 128M
          volumes:
          - name: containerdisk
            containerDisk:
              image: quay.io/kubevirt/cirros-container-disk-demo
\end{lstlisting}

\subsubsection*{Steps}
\begin{enumerate}
    \item Create a VM
\begin{lstlisting}
    kubectl apply -f my-vm.yaml
\end{lstlisting}
    \item Access the VM through SSH or VNC (credentials vary depending on the image since no CloudInit is used) 
\begin{lstlisting}
    virtctl ssh cirros@my-vm
    virtctl vnc my-vm
\end{lstlisting}
    \item Stop, start and restart the VM
\begin{lstlisting}
    virtctl stop my-vm
    virtctl start my-vm
    virtctl restart my-vm
\end{lstlisting}
    \item Delete the VM
\begin{lstlisting}
    kubectl delete vm my-vm
\end{lstlisting}
\end{enumerate}

\section{Maintenance}
The second task is related to maintenance, where you will create a snapshot of a VM and then restore it. You will also perform a live migration on a VM to move it from one node to another without shutting it down. For this task, you can use a VM with a containerDisk since the task is not related to persistent storage.  

Both snapshots and live migrations are treated as a resource by KubeVirt, meaning that the operation will take place in the background after a VirtualMachineSnapshot resource or a LiveMigration resource is created. See the References section for more information.

A live migration resource in KubeVirt does not move resources itself and instead uses the Kubernetes scheduler to decide where to place the VM. This means that a LiveMigration resource only tells the scheduler that the VM can be moved but not where to move it. The simplest way to tell the scheduler to move a VM to a specific host is to edit \textit{spec.template.spec.nodeSelector} in the VM manifest. See the References section for more information. 

\subsection*{Manifests}
The following is a minimal manifest for creating a VM that runs on a node with label \textbf{name: worker-1}. The credentials for the Cirros image are user \textbf{cirros} and password \textbf{gocubsgo}.
\begin{lstlisting}[language=yaml]
  apiVersion: kubevirt.io/v1
  kind: VirtualMachine
  metadata:
    name: my-vm
  spec:
    running: true
    template:
      spec:
        nodeSelector:
          name: worker1
        domain:
        cpu:
          cores: 1
        devices:
          disks:
          - name: containerdisk
            disk:
              bus: virtio
        resources:
          requests:
          memory: 128M
        volumes:
        - name: containerdisk
          containerDisk:
            image: quay.io/kubevirt/cirros-container-disk-demo
\end{lstlisting}

\noindent The following is a minimal manifest for creating a VirtualMachineSnapshot resource.
\begin{lstlisting}[language=yaml]
    apiVersion: snapshot.kubevirt.io/v1alpha1
    kind: VirtualMachineSnapshot
    metadata:
        name: my-vm-snapshot
    spec:
        source:
        apiGroup: kubevirt.io
        kind: VirtualMachine
        name: my-vm
\end{lstlisting}

\noindent The following is a minimal manifest for creating a LiveMigration resource.
\begin{lstlisting}[language=yaml]
    apiVersion: kubevirt.io/v1
    kind: VirtualMachineInstanceMigration
    metadata:
        name: my-vm-migration
    spec:
        vmiName: my-vm
        targetNode: worker2
\end{lstlisting}

\noindent The following is a minimal manifest for restoring a VM from a snapshot.
\begin{lstlisting}[language=yaml]
    apiVersion: snapshot.kubevirt.io/v1alpha1
    kind: VirtualMachineRestore
    metadata:
        name: my-vm-restore
    spec:
    target:
        apiGroup: kubevirt.io
        kind: VirtualMachine
        name: my-vm
    virtualMachineSnapshotName: my-vm-snapshot
\end{lstlisting}

\subsection*{Snapshot Steps}
\begin{enumerate}
    \item Create a VM
\begin{lstlisting}
    kubectl apply -f my-vm.yaml
\end{lstlisting}
    \item Create a snapshot of the VM
\begin{lstlisting}
    kubectl apply -f my-vm-snapshot.yaml
\end{lstlisting}
    \item Wait for the snapshot to be ready:
\begin{lstlisting}
    kubectl wait vmsnapshot my-vm-snapshot --for condition=Ready
\end{lstlisting} 
    \item Restore the VM from the snapshot
\begin{lstlisting}
    kubectl apply -f my-vm-restore.yaml
\end{lstlisting}
    \item Use the following kubectl command to wait for the restore to be ready:
\begin{lstlisting}
    kubectl wait vmrestore my-vm-restore --for condition=Ready
\end{lstlisting}
\end{enumerate}

\section{Debugging}
The third task is about debugging a VM that is not accessible through SSH. You will be presented with a VM manifest which, if deployed, will result in a VM that is running but cannot be accessed through SSH. Since the VM is running, you can access it through VNC using virtctl. You can use a \textit{containerDisk} since the task is not related to persistent storage. 

\subsection*{Manifests}
The following is a manifest for creating a Ubuntu VM that is not accessible through SSH. Credentials for the Ubuntu image are user \textbf{cloud} and password \textbf{cloud}.
\begin{lstlisting}[language=yaml]
  apiVersion: kubevirt.io/v1
  kind: VirtualMachine
  metadata:
    name: my-vm
  spec:
    running: true
    template:
      spec:
        domain:
          devices:
            disks:
            - name: containerdisk
              disk:
                bus: virtio
            - name: cloudinit
              disk:
                bus: virtio
            rng: {}
          resources:
            requests:
              cpu: 2
              memory: 2Gi
        volumes:
        - name: containerdisk
          containerDisk:
            image: quay.io/containerdisks/ubuntu:22.04
        - name: cloudinit 
          cloudInitNoCloud:
            userData: |-
            #cloud-config
            users:
            - name: cloud
              passwd: $6$rounds=4096$abc
              shell: /bin/bash
              lock-passwd: false
              ssh_pwauth: True
              chpasswd: { expire: False }
              sudo: ALL=(ALL) NOPASSWD:ALL
              ssh_authorized_keys:
              - ssh-ed25519 abc
\end{lstlisting}

\subsubsection*{Steps}
\begin{enumerate}
    \item Create a VM
\begin{lstlisting}
    kubectl apply -f my-vm.yaml
\end{lstlisting}
    \item Access the VM through VNC
\begin{lstlisting}
    virtctl vnc my-vm
\end{lstlisting}
    \item Debug the issue to fix the misconfiguration
    \item Edit the VM manifest to fix the issue
\begin{lstlisting}
    kubectl edit vm my-vm
\end{lstlisting}
    \item Ensure the VM is accessible through SSH
\begin{lstlisting}
    virtctl ssh cirros@my-vm
\end{lstlisting}
    \item Delete the VM
\begin{lstlisting}
    kubectl delete vm my-vm
\end{lstlisting}
\end{enumerate}
