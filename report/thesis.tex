\section{Virtualization Fundamentals}
Virtualization is a technology \cite{virtualization-basics} that enables partitioning or grouping computing resources into operating environments using methods such as partitioning, emulation, and time-sharing. Virtualization could occur in various forms, such as network virtualization, storage virtualization, and server virtualization. More commonly used, though, is hypervisor based virtualization \cite{virtualization-history}, which is a software layer that creates virtual operating environments, or more commonly referred to as virtual machines (VMs).

A VM is a emulation of a physical computer \cite{virtualization-trends} and is perceived as \textit{real} computer by the application running inside it even though underlying resources are, more often than not, shared with other VMs. For this reason, virtualization is often used to improve resource utilization and to isolate applications from each other. Such a setup improves flexibility of the computing environment, but it implies that the underlying mechanism is formally different from that of a physical machine. For instance, hardware abstraction is used to decouple the VM's perceived hardware capabilities from the physical hardware. Abstracting the underlying hardware allows for \textit{virtual} computing resources, such as virtual CPUs and virtual memory, either by grouping or partitioning the physical resources.

Furthermore, the abstraction layer can exist in multiple different layers depending on the virtualization technology used, ranging from the hardware level to the application level, some of which are listed below:
\begin{itemize}
    \item Full virtualization (FV): Provides a complete emulation of the underlying physical hardware and creates a complete virtual environment. FV enables fully decoupling the VM from the underlying resource, and does not require any modification \cite{virtualization-basics}\cite{xen-virtualization}\cite{hypervisor-xen}.
    \item Paravirtualization (PV): A subset of FV aiming to improve the VM performance by reducing the amount of resource that are emulated. The VM is presented with an image that is similar, but not identical, to the underlying hardware. As such, the guest OS is aware it is running inside a VM \cite{xen-virtualization}\cite{hypervisor-xen}.
    \item Application virtualization (AV): A method where an application is encapsulated in a virtualized lightweight environment, such it perceived it is being run on a specific platform. Application virtualized using AV act as a portable and self-sufficient unit, thus often used to package and distribute applications, such as the JVM \cite{virtualization-basics}\cite{application-virtualization}.
\end{itemize}

Define virtualization and discuss its significance in modern IT infrastructures, including the evolution from physical to virtual resources.
\subsection*{Benefits and Challenges}
Many of benefits with virtualization stems from the fact that there exists an abstraction layer above the physical hardware. The abstraction layer decouples software from hardware, e.g. vendors with different I/O devices can be used in the same environment, allowing for a more flexible and scalable computing environment as whole where hardware is turned into simple pool of computing resources \cite{virtualization-trends}. As such, there are numerous benefits with virtualization, a few of which are listed below:
\begin{itemize}
    \item Resource utilization: Virtualization allows for better utilization of hardware resources, as multiple VMs can run on the same physical hardware. This is achieved by sharing the underlying resources, such as CPU and memory, between the VMs \cite{virtualization-basics}. 
    \item Isolation: VMs are able to isolate their entire state from one another, including operating system (OS), filesystem and application stack. Isolation is also beneficial in scenarios where reproducibility is key, such as in testing and development environments \cite{vm-docker-reproducibility}.
    \item Flexibility: VMs not tied to a specific hardware, and can be moved between different physical resources.
    \item Management: VMs can be created and destroyed on demand, and can be managed using software tools \cite{virtualization-basics}.
    \item Virtual hardware: A VM can be configured with hardware that does not exist in the physical environment, such as a virtual network card or a virtual hard drive \cite{virtualization-basics}.
\end{itemize}

However, the portability made possible by virtualization comes at a cost of performance. Since, at the core, a VM is a complete but emulated system, it means the every instructions issued must be interpreted in software. As such, despite what level of abstraction layer is used, there is an inescapable added overhead when using virtualization. A common hypervisor such as KVM (see \ref{hypervisors}) introduces an overhead of 5-10\% \cite{kvm-overhead}, and the overhead can be even larger when using a full virtualization approach.

\section{Understanding Hypervisors}
\label{hypervisors}
A hypervisor is a software layer that enables managing multiple VMs on a single physical machine \cite{virtualization-history} by allowing multiple operating systems (termed as \textit{guests}) to share the same hardware \cite{hypervisor-survey}. Hypervisors, often referred to as virtual machine monitors (VMMs), are responsible for creating and managing VMs, and for providing the necessary abstractions to the VMs.

Hypervisors can be categorized into two types, \textit{type 1} and \textit{Type 2}, based on their operational context and how they interact with the underlying hardware \cite{hypervisor-survey}\cite{hypervisor-types-1-2}. While the goals and purpose of type 1 and Type 2 hypervisors is the same, there are some core differences. type 1 hypervisors, commonly referred to as \textit{bare-metal} hypervisors, run directly on top of the physical hardware and do not require an OS underneath. A type 1 hypervisor is therefore scalable to terabytes of RAM and hundreds of CPU cores \cite{hypervisor-types-1-2}. However, without any other additional software, it is also itself responsible to schedule and allocate required system resources to the VMs. Type 2 hypervisors, also known as \textit{hosted hypervisors}, relies on an underlying OS to manage the calls to the CPU and memory resources. The presence of an underlying OS inevitably introduces latency, which is why Type 2 hypervisors are often used for development and testing purposes \cite{hypervisor-survey}.

Furthermore, hypervisors can be further categorized into \textit{monolithic} and \textit{microkernelized} hypervisors \cite{hypervisor-comparion-2013}. Monolithic hypervisors are built as a single large software stack, managing all the VMs and their resources, and hosting drivings for all hardware access. Having all drivers reside within the hypervisor means that the such as hypervisor will only run on selected hardware. However, it also means no operating system is required at all, as the hypervisor acts as a operating system by supporting all the operations of its VMs. Microkernelized hypervisors, on the other hand, does not require devices drivers to be present in the hypervisor. Instead, the drivers are installed in a form of managing VM, often referred to as parent VM. Each VM created by such a hypervisor is thus referred to as a child VM, as it uses the parent VM's drivers to access the hardware. This approach is more flexible, as it can allow the hypervisor to run on any hardware that supports the parent VM's operating system. 

% security rings definition
In the realm of hypervisors, the concept of \textit{security rings} is often used to describe the level of privilege different part of a hypervisor has. A security ring represents a hierarchical protection domain, where the innermost ring (ring 0), usually reserved for the OS kernel, has the highest privilege, and the outermost ring (ring 3), usually reserved for user applications, has the lowest privilege. The hypervisor itself is often placed in ring 0, its VM kernel space in ring 1, and the VM user space in ring 3. Security rings are thus used to define execution level to provide a system to limit execution capabilities of VMs in order to regulate access to shared memory. This enabled a strong isolation layer between the VMs, and between the VMs and the hypervisor, thus a key part of a hypervisor's security model \cite{hypervisor-comparion-2013}\cite{microkernel-monolithic}.

\subsection*{Key Hypervisors}
\label{hypervisors}
There are several hypervisors available, each with its own set of features and capabilities, some of which are listed below:

\begin{itemize}
    \item VMware ESXi: A lightweight type 1 monolithic hypervisor often used in enterprise environments. ESXi is known for its ultra lightweight footprint stretching merely a few hundred megabytes, and its ability to run directly on the hardware without an underlying OS \cite{hypervisor-comparion-2013}.
    \item Microsoft Hyper-V: A type 1 microkernelized hypervisor that is part of the Windows Server operating system. As a microkernelized hypervisor, Hyper-V differentiates between parent and child VMs, and uses the parent VM to manage hardware access for the child VMs \cite{hypervisor-comparion-2013}\cite{hypervisor-hyper-v-esxi}.
    \item Citrix Hypervisor: Formerly known as XenServer, is a type 1 microkernelized hypervisor based on the Xen Project. It runs using a modified Linux kernel in a VM, called Dom0, that manges hardware I/O and interacts with the other virtual machines, called DomU guests \cite{hypervisor-xen}\cite{hypervisor-comparion-2013}. 
    \item KVM: A type 1 hypervisor that is part of the Linux kernel. As such, KVM uses most of the functionalities from the Linux kernel, such as the scheduler and memory management. KVM has become a popular alternatives to proprietary hypervisors, such as VMware ESXi and Microsoft Hyper-V, as it is open-source and free to use \cite{hypervisor-comparion-2013}\cite{hypervisor-kvm}. 
\end{itemize}

\section{VM Management Essentials}
A management system for VMs involves a set of tools and processes that are used to create, deploy, and manage VMs \cite{openstack-opennebula-comparison}. The management system is responsible for managing the entire lifecycle of a VM, including provisioning, deployment, monitoring, and scaling. Not be confused with a hypervisor, a VM management system lives on a higher-level and aims to provide an abstraction layer in the form of an interface to allow interaction with the VMs, often enabling the support of a set of hypervisors \cite{cloudframeworks-and-hypervisors}. As such, a VM management system unifies hypervisors, typically through an higher-level API, such as a CLI or a web-based user interface.

\subsection*{Concepts and Features}
With a higher abstraction layer, a VM management system brings a set of concepts to simplify managing VMs, such as clustering, availability, and scalability \cite{cloudframeworks-and-hypervisors}. Clustering includes grouping a set of resources, such as VMs or storage nodes, to provide a image that is perceived as a single resource. Availability refers to the ability to keep a resource active or running, which is, more often than not, clustered resources. Scalability involves technologies to ensure resources are able to manage an increase or decrease in computational load, such as increasing the amount of virtual cores in a VM if it's overloaded.  

\subsection*{Infrastructure as a Service}
A VM management system is often part of a larger cloud computing infrastructure, such as Infrastructure as a Service (IaaS). IaaS is a lower-level cloud computing model that aims to eliminate the need to install and manage physical hardware by providing virtualized computing resources, such as VMs, virtual network and storage.

\subsection{Comparison of VM Management Systems}
Managing VMs can be done in many ways, and there are many VM management systems available, each with its own set of features and capabilities. Many such systems are a part of a larger platform, such as an IaaS or even as large as a cloud platform. Some of the most prevalent platforms that has been used in the industry are OpenStack, CloudStack, and OpenNebula, and some of the most popular public cloud platforms are Amazon EC2, Azure Virtual Machines, and Google Compute Engine.

\subsubsection*{OpenStack}
An open-source IaaS platform that offers a large set of features to manage VMs, network, and storage \cite{openstack-opennebula-comparison}. OpenStack's architecture is inherently modular, and it is designed to be deployed in a distributed manner \cite{openstack-overview}. The architecture includes three major parts:
\begin{itemize}
    \item \textit{Nova}: The compute service that manages VMs and provides an interface to manage VMs. Nova support all operation regarding the life cycle of a VM, such as creating, deleting, and migrating VMs. Nova is itself divided into multiple copmonents, each employing a \textit{shared-nothing} and \textit{messaging-based} policy, which means every component is independent and communicates with other components using a message queue. 
    \item \textit{Swift}: The object storage service that provides a scalable and redundant storage system for VMs. It is a long-term storage for static data, such as images and backups, and is comparable to Amazon S3.
    \item \textit{Glance}: The image service that provides a catalog of VM images that can be used to create new VMs. 
\end {itemize}

OpenStack supports features such as live migration, snapshot management, and scalability, and is often used in large-scale cloud environments \cite{openstack-opennebula-comparison}. However, these features are often hypervisor-specific, and requires both support from the hypervisor and specific configuration to work \cite{openstack-live-migration}.

\subsubsection*{CloudStack}
An open-source IaaS platform and cloud computing software that manages computing, networking and storage resources, and is used to build and manage public and private cloud environments \cite{cs-overview}. On a high level, CloudStack is built in a monolithic fashion, encapsulating all the management features into two software packages called the management server and agent. The management server is responsible for managing the lifecycle of VMs, including provisioning, deployment, and scaling, and is comparable to a control node. The agent, however, is installed on each node in the cloud and is responsible for low-level management task of the VMs on that node, such as interacting with the underlying hypervisor. While it is a monolithic system, CloudStack is designed to be horizontally scalable, and can be deployed in a distributed manner with multiple management servers \cite{cs-overview}. 

CloudStack supports a large variety of hypervisors, such as VMware, KVM, and XenServer, and storage services, such as Amazon S3, allowing it to function as an abridgment layer for compute, network, and storage to unify a wide range of existing infrastructure \cite{cs-overview}. Features such as live migration, snapshot management, and scalability are supported, but are often hypervisor-specific and requires specific configuration to work \cite{cs-live-migration}.

\subsubsection*{OpenNebula}
An open-source IaaS platform and toolkit used to provide open, scalable, and comprehensive management of virtualized data centers \cite{openstack-opennebula-comparison}. OpenStack employs a modular architecture divided in three layers \cite{openenbula-tele-teaching}\cite{iaas-comparisons}: 
\begin{itemize}
    \item \textit{Tools layer}: Provides a set of interfaces such as a CLI and web-based user interface to interact with the system, but also high-level management tools such as scheduling.
    \item \textit{Core layer}: Provides the core functionalities of the system, such as host management, VM management, and virtual network management.
    \item \textit{Driver layer}: Provides an abstraction on top of the underlying infrastructure to enable the system to support a wide range of hardware, including networking, storage, and hypervisors. As such, the driver layer is responsible for allocation, deallocation and monitoring of resources, such as VMs and storage. 
\end{itemize}

By dividing the system into these layers, OpenNebula provides modularity that enables the system to be extended and customized to fit specific needs, such as integrating with other systems or using OpenNebula to build a private cloud.

\subsubsection*{Amazon EC2}
Amazon Elastic Compute Cloud (EC2) is a web service that provides on-demand compute resources over the cloud \cite{ec2-overview}\cite{ec2-overview-web}. EC2 is part of Amazon Web Services (AWS), and is designed be scalable and flexible, and is often used to host general-purpose applications, including web applications, databases and enterprise applications. While EC2 platform uses Xen-based hypervisors (see \ref{hypervisors}) under the hood \cite{public-cloud-comparison}, it acts as a complete abstraction layer over the underlying hardware and hypervisors, and instead provides a rich set of tools to manage the VMs. 

EC2 provides a large set of preconfigured templates called Amazon Machine Images (AMIs) that can be used to create new VMs, as well as a configurations in terms of specifications. This enables EC2 to provide a wide range of VMs, such as general-purpose, compute-optimized, memory-optimized, and storage-optimized VMs \cite{ec2-overview}. \subsubsection*{Azure Virtual Machines}
Azure Virtual Machines is a service provided by Azure to manage VMs in the cloud \cite{azure-overview}, and is designed to be a cost-effective alternative to on-premises hardware \cite{azure-cost-effective}. Azure Virtual Machine provides a set of preconfigured VMs that can be used to create VMs for specific purposes, such as general-purpose or high-performance computing (HPC).

Behind the scenes Azure Virtual Machines runs on Hyper-V (see \ref{hypervisors}) and provides a set of features such as live migration, snapshot management, and auto-scaling \cite{azure-overview-web}.

\subsubsection*{Google Compute Engine}
Google Compute Engine is a service provided by Google Cloud to manage computing resources in the cloud \cite{gce-overview-web}. Compute Engine consists of three major parts: VMs, storage, and networking, where VMs is the core component. VMs are offered in a wide range of configurations, such as standard, high-memory, and high-CPU, making them suitable for different computational workloads \cite{gce-overview}. 

Compute Engine uses KVM as its hypervisor (see \ref{hypervisors}), and provides many features such as live migration, snapshot management, and auto-scaling \cite{gce-overview-web}\cite{public-cloud-comparison}.


\subsubsection{Comparison}
\label{vm-management-comparison}

OpenStack, OpenNebula, and CloudStack are all examples of software that can be used to build private cloud platforms \cite{public-private-cloud}\cite{hybrid-cloud}. A private cloud is an on-premise cloud that hosted within an organization, more often than not for that organization's internal use. A common denominator for these systems is that they are open-source, and that they are highly customizable and extendable. As such, much of the configuration and management of the system is done by the user, and the system can be integrated with other systems to fit specific needs. This means many of the conveniences, such as preconfigured VMs, and features, such as live migration, often requires specific configuration and setup. 

On the other hand, VM management systems part of a public clouds, such as Amazon EC2, Azure Virtual Machines and Google Compute Engine, are designed to be more user-friendly and to provide a more complete abstraction layer over the underlying hardware and hypervisors \cite{public-private-cloud}. As such, many of the conveniences and features are provided out of the box, and are perceived as hypervisor- and hardware-agnostic. This means that the user does not need to worry about the underlying hardware and hypervisors, and can instead focus on managing the VMs.

Both private and public cloud solutions for VM management are designed to be scalable and flexible, and support a wide range of use cases (although private cloud solutions scales worse in practice \cite{hybrid-cloud}). Which solution to use depends on the specific needs of the organization, and the level of control, customization needs, and the required security level of the system.

\section{Containers and Kubernetes: A Paradigm Shift}
Besides the common cloud service models (IaaS, PaaS, and SaaS), another service models has emerged, called \textit{Container as a Service (CaaS)} \cite{caas}. CaaS is a container based cloud service model that has gained significant traction due to the lightweight and portable nature of a container, allowing for simple application bundling that is able to run in a vast range of environments. 

Containers are often used in conjunction with a container orchestration platform, such as Kubernetes, to manage the lifecycle of the containers, including deployment, scaling, and monitoring. As such, they are often used to build, test, and deploy applications in a consistent manner. ThiA common use case for containers is to run microservices, which is a software architecture that employs a set of small, independent and loosely coupled applications \cite{microservices-overview}. 

\subsection{Containers Explained}
Containers are a form of virtualization that offers a horizontally scalable and lightweight alternative to VMs \cite{container-k8s-benchmark}\cite{container-vm-overview}. Unlike VMs, containers do not require a hypervisor to run, and instead run on top of an OS and shares the OS kernel with other containers. By not requiring a hypervisor, containers are able to run with less overhead than VMs, making their lifecycle relatively fast and efficient compared to VMs. 

The Open Container Initiative (OCI) is a standardization effort that was created to ensure that containers are portable and interoperable across different platforms \cite{oci}. The OCI standard defines a runtime specification, image specification, and distribution specification, allowing for a container to be consistent in terms of its runtime environment, image format, and distribution method. 

Creating, deploying, and managing containers is using a container runtime, such as containerd \cite{container-runtimes}. Most runtimes, including containerd, implement the OCI runtime specification, and are able to run containers in a consistent manner. This also suggests that the OCI can be viewed as an abstraction layers, where, for instance, the OCI runtime specification acts as an intermediary between the container runtime and the OS kernel.

\subsection{Orchestration Using Kubernetes}
Kubernetes is an open-source container orchestration platform that manages containers. It maintains a set of abstraction layers to manage the life cycle of containers and to provide a consistent environment for the containers to run in, such as scheduling, scaling, and networking \cite{container-vm-overview}.

Kubernetes is inherently declarative, meaning that instead of issuing commands to perform a specific task, such as starting a container, Kubernetes is given a desired state for a set of resources, and it is responsible for ensuring that the current state of the resources matches a desired state \cite{container-vm-overview}\cite{k8s-overview}. As such, Kubernetes provides auto-restart, re-schedule, and replication control to ensure a resource stays in the desired state. By specifying the desired state, an entire application is able to be contained in a single configuration file, including its container, networking, storage, and security requirements. 

Some of the abstractions used in Kubernetes includes \textit{Pods}, \textit{Services}, and \textit{Deployments} \cite{k8s-overview}. A Pod is the smallest deployable unit in Kubernetes, and represents a group of one or more containers that are scheduled to run on a node. A Service represents a single-point-of-entry communication layer for a set of Pods, and is used to provide a consistent way to access the Pods, such as load balancing. A Deployment is a higher-level abstraction on top of Pods that is used to manage the life cycle of the Pods, such as scaling and rolling updates. A Deployment does not represent an executable resource, but instead describes a desired state in which a set of Pods should be in. There are many other abstractions in Kubernetes, each with its own set of features and capabilities that together creates a feature complete platform for container orchestration.

A Custom Resource Definition (CRDs) is a form of extension for Kubernetes resources that allows for the creation of custom resources \cite{k8s-overview}\cite{k8s-crd-webiste}. CRDs are used to extend the Kubernetes existing abstractions to provide additional features and capabilities to allow for a more feature complete platform when custom behavior is needed. For instance, a CRD can be used to enforce certain fields to be specified in order to create the resource, or to provide a custom validation logic \cite{k8s-crds}.

Kubernetes uses a client-server architecture, where the server, often referred to as the control plane, is responsible for managing the state of the cluster, and the client, often referred to as the worker node, is responsible for running the containers \cite{k8s-overview}. The control plane runs system-related services, such as the API server, scheduler, and controller manager, and is thus responsible for managing the state of the cluster. The API server can be viewed as the primary service for a Kubernetes cluster as it is the main management point for the cluster, receiving and processing request from the client. The worker node, on the other hand, is responsible for running the containers containers. The worker node consist of two primary components: the kubelet and the kube-proxy. The kubelet is a service that includes a container runtime, which runs containers in according to requests from the control plane. Meanwhile, the kube-proxy service manages the network communication for the containers. To interact with Kubernetes form a user perspective, a client is used to send requests to the API server. The client can be a CLI, such as kubectl, a web-based user interface, or through a programmatic interface such as a REST API \cite{k8s-overview}. The CLI tool kubectl is the most common way to interact with Kubernetes, and is used to create, delete, and manage resources in the cluster.
