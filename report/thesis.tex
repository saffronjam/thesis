\begin{abstract}
    \Glspl{vm} and containers both offer a computing environment for a wide range of workloads, providing their respective benefits such as strong isolation or being lightweight. However, they have traditionally been managed in separate management environments which has led to a fragmented infrastructure for organizations that use both \Glspl{vm} and containers. KubeVirt was created to unify the management of \Glspl{vm} and containers by integrating \Glspl{vm} into Kubernetes, a widely adopted platform originally designed for container orchestration. This project aims to evaluate Kubernetes with KubeVirt as a \Gls{vm} management system both by comparing it with traditional approaches and by investigating the implications of moving into the Kubernetes ecosystem. The project uses a mixed-methods approach including a literature study, a qualitative comparison in the form of a usability analysis and a quantitative comparison in the form of a performance analysis. The results show that managing \Glspl{vm} in Kubernetes using KubeVirt comes with a steeper learning curve than traditional approaches and that there is additional overhead. However, the results also show that KubeVirt can provide performance and operational efficiencies that are not possible with traditional approaches.
\end{abstract}

\chapter{Introduction}
Virtualization is a technique that either splits or combines computing resources into one or more operating environments. This is achieved by creating a software layer that surrounds or underlies existing hardware or software, which enables resources to be decoupled from the underlying hardware. Virtualization comes in various forms including the virtualization of storage, networks and servers. Virtualizing networks and storage is achieved by creating an abstraction layer on top of the physical network and storage resources, respectively, to provide a more flexible interface to interact with. Server virtualization, on the other hand, is used to create virtual computing environments that can be used for various workloads. By combining virtualized resources into a single operating environment, virtualization allows heterogeneous hardware resources to be used homogeneously\ \cite{virtualization-history, virtualization-basics, virtualization-trends}.

Server virtualization is achieved by using a hypervisor, a software layer that operates between the hardware and the \Gls{os} and is responsible for creating virtual computing environments called virtual machines (\Glspl{vm})\ \cite{virtualization-trends, virtualization-ibm, hypervisor-survey}. \Glspl{vm} are emulations of physical computers that are perceived as real computers by applications running inside them and are therefore used as complete and independent environments. Such environments are particularly useful when running legacy applications and applications that require a high level of isolation. For this reason, \Glspl{vm} have long been the core part of IT infrastructure\ \cite{virtualization-history, virtualization-basics}. 

In parallel with the evolution of \Glspl{vm}, containers have emerged as an alternative to \Glspl{vm}. Container virtualization is not based on a hypervisor and instead operators higher level allowing multiple computing environments on a single \Gls{os} instance. As such, containers are more lightweight, have less overhead and have faster startup times than \Glspl{vm}\ \cite{container-vm-overview}. For this reason, containers have become a popular choice for scenarios in which scaling and portability are important, such as in a microservice architecture, which are core components in modern cloud-native applications\ \cite{cloud-native}. Figure\ \ref{fig:container-vm-architecture} visualizes the differences between how \Glspl{vm} and containers are virtualized, and how the virtualization layer is placed on a lower level for \Glspl{vm} than for containers. 

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{img/vm_container_virtualization.pdf}
    \caption{Overview of \Glspl{vm} and containers. The virtualization layer is placed on a lower level for \Glspl{vm} than for containers, illustrating the difference in their respective virtualization techniques.}
    \label{fig:container-vm-architecture}
\end{figure}

However, while containers provide a lightweight alternative to \Glspl{vm}, they lack the strong isolation properties \Glspl{vm} have. Sharing the \Gls{os} kernel with the host and other containers opens up a larger attack surface, which implies that containers are more prone to security vulnerabilities. While there exist technologies that strive for the benefits of both \Glspl{vm} and containers, such as Kata Containers, gVisor and Firecracker, it remains a fact that both containers and \Glspl{vm} have their separate use cases\ \cite{container-runtimes-overview, container-as-vm-comparison}. 

Due to their different use cases, \Glspl{vm} and containers have traditionally been managed separately. \Glspl{vm} are often managed by a \Gls{vm} management system that provides tools and processes for life-cycle management. This makes a \Gls{vm} management system act as a higher-level abstraction layer that unifies hypervisors and provides interfaces to interact with the underlying VMs, typically through a \Gls{cli} or a web-based \Gls{gui}. Figure \ref{fig:vm-management-system} illustrates this in an overview of a \Gls{vm} management system's role where clients interact with the system to manage \Glspl{vm} distributed across several hosts. Moreover, a \Gls{vm} management system is often part of a larger infrastructure, such as an \Gls{iaas}, that extends its role to also include resources such as networking and storage.  Examples of such systems include OpenStack\ \cite{openstack-website}, CloudStack\ \cite{cloudstack-website}, and OpenNebula\ \cite{opennebula-website}. Containers, on the other hand, are often managed by a container orchestration platform. These platforms provide similar tools and processes as a \Gls{vm} management system but for containers, with some additional tools for container-centric operation such as scaling. Examples of such platforms include Kubernetes\ \cite{k8s-website}, Nomad\ \cite{nomad-website} and Docker Swarm\ \cite{docker-swarm-website}, with the former being the most widely adopted \cite{k8s-it-administration}. As such, while VMs and containers have their respective use, they also share aspects regarding how they provide virtual computing environments. For this reason, there has been an ongoing convergence in how \Glspl{vm} and containers are managed\ \cite{vm-cloud}. 

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{img/vm_management_system.pdf}
    \caption{Brief overview of a monolithic \Gls{vm} management system where clients communicate with a management node through a set of interfaces to manage \Glspl{vm} distributed across several hosts. The indirect communication allows a client to manage \Glspl{vm} without having to interact with the hosts directly.} 
    \label{fig:vm-management-system}
\end{figure}

Kubernetes, originally designed to manage containerized applications at scale, was recently extended to accommodate \Glspl{vm} through the KubeVirt project. KubeVirt is a Kubernetes extension that enables \Glspl{vm} to run inside a Kubernetes environment alongside containers, creating a common environment for both virtualization technologies\ \cite{kubevirt}. Figure \ref{fig:container-vm-kubevirt-architecture} depicts a high-level overview of how \Glspl{vm} and containers coincide in a Kubernetes environment using KubeVirt, where traffic flows natively between the two types of resources. How this is achieved on a technical level is discussed later in \ref{sec:kubevirt-technical-overview}. 

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{img/container_vm_with_kubevirt.pdf}
    \caption{Overview of \Glspl{vm} and containers managed by using KubeVirt. The figure illustrates how \Glspl{vm} and containers communicate natively in a single environment.}
    \label{fig:container-vm-kubevirt-architecture}
\end{figure}

The approach of including \Glspl{vm} in the Kubernetes ecosystem not only benefits from the mature Kubernetes infrastructure but also from tools and processes that are designed to integrate with the life cycle of Kubernetes resources. This means by using KubeVirt, \Glspl{vm} can be managed more similarly to containers, which creates a more cloud-native approach to managing \Gls{vm}\ \cite{cloud-native}. Additionally, consolidating the management of \Glspl{vm} and containers into a single platform could offer more benefits since the system architecture could be simplified and the required skill set in an organization could be reduced\ \cite{kubevirt-cncf, kubevirt-storware}. 

The integration of \Glspl{vm} into Kubernetes using KubeVirt represents a significant milestone toward a more agile, efficient and unified IT infrastructure\ \cite{kubevirt-storware}. However, KubeVirt's practicality in a real-world scenario is not yet fully explored. As such, this project aims to evaluate Kubernetes with KubeVirt as a \Gls{vm} management system and compare it with traditional approaches to understand its practicality and limitations.

\section{Problem}
\Glspl{vm} and containers are similar technologies but with different use cases. This has led to separate tools and processes for managing them, and entire systems built solely for one of them. However, both \Glspl{vm} and containers are necessary parts of the modern IT infrastructure that many organizations depend on. This has led to a fragmented infrastructure and a larger skillset among operators to manage both types. Furthermore, many technologies today integrate well with Kubernetes's declarative approach to resource management, and thus, by managing a resource outside of Kubernetes, these benefits are not fully utilized.

\section{Purpose}
The purpose of this project is to evaluate whether Kubernetes with KubeVirt is a viable option for \Gls{vm} management. This evaluation is important in the field of cloud computing since it can provide insight into how an existing system could benefit from integrating \Glspl{vm} management in Kubernetes, as well as consolidating the management of \Glspl{vm} and containers to a single platform.

\section{Goals}
The goal of this project is to provide a transparent evaluation with quantifiable results regarding the applicability of managing \Glspl{vm} inside the Kubernetes ecosystem. The results could be used as a fundamental argument for or against Kubernetes as a management platform, and potentially result in a more efficient management system. Potential results that argue for the use of KubeVirt in terms of performance could be used as an argument for future research regarding sustainability.

\section{Research Method}
This project will use a mixed-methods approach to investigate the research questions. A literature study will initially be conducted to thoroughly explore the reasons that led to KubeVirt being created as well as the current challenges regarding \Gls{vm} management. To evaluate KubeVirt itself both a quantitative and qualitative analysis will be conducted. The quantitative analysis will be based on a performance analysis that compares KubeVirt against a traditional approach and benchmarks the performance of the system when performing tasks such as creating \Glspl{vm}. The qualitative aspects will be based on a usability analysis, which involves having system administrators perform certain common tasks using KubeVirt. Together, these three methods capture an image of using KubeVirt for managing \Glspl{vm}.  

\section{Limitations}

\section{Virtualization Fundamentals}
\label{virtualization}
Virtualization is the technology that enables hardware to be split or grouped into logical units. Each unit runs with its own allocated resources and is separated from other units running on the same host. Virtualization can thus occur in various forms including virtualizing networks, storage and servers\ \cite{virtualization-basics}. The two former are a simpler type of virtualization that involves creating an abstraction layer on top of existing physical resources to provide a logical interface defined in software. By defining storage and network in software, a more flexible environment is created, where an application interacts with interfaces that indirectly point to a group of hardware, instead of directly communicating with the hardware. Server virtualization, on the other hand, is the more common form of virtualization and is also known as hypervisor-based virtualization. It involves splitting a server into multiple virtual computing environments\ \cite{virtualization-history}. Hypervisors will be discussed in more detail in the next section, see\ \ref{hypervisor}

A virtual computing environment, more commonly referred to as a \Gls{vm}, is a complete system that operates as an isolated environment. The environment is perceived as if it were an ordinary computer from the perspective of the application running inside it, even though the physical resources are shared with other \Glspl{vm} running on the same host. Being a complete system is particularly useful when running legacy applications that are dependent on a certain \Gls{os} or driver version. This could be taken a step further where a piece of hardware could be emulated to suit the needs of a legacy application. Additionally, the isolated environments are suitable for scenarios where there are strict security requirements for the application running inside it\ \cite{virtualization-basics, virtualization-trends}.

Virtualization can be achieved in different layers depending on the required level of abstraction. The three major levels are hardware level, \Gls{os}-level and application level. For instance, storage and network virtualization both fall under the hardware level category since they create an abstraction layer on top of the physical resources. However, in other scenarios, virtualization is not required to be as low-level. Application virtualization is concerned with encapsulating an application in a more lightweight environment, such that the application thinks it is executed on a specific platform. A common example is \gls{jvm} that enables programs written in the programming language Java to be run on multiple operating systems\ \cite{application-virtualization}. Figure \ref{fig:hardware-os-virtualization} illustrates this as an overview of the differences between the two more common layers of virtualization; hardware and \Gls{os} virtualization. It is important to note that the virtualization layers are not mutually exclusive and that using them in combination leads to a unique use case. For instance, a guest process (\Gls{os} level) can run in a guest \Gls{os} (hardware level). This specific use case is referred to as nested virtualization\ \cite{virtualization-basics}.

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/hardware_os_virtualization.pdf}
    \caption{Overview of the different layers of virtualization for hardware-level and \Gls{os}-level virtualization. The figure illustrates how hardware virtualization inhibits several internal \Glspl{os} from running on the same hardware, while \Gls{os} virtualization keeps a single \Gls{os} for each process.}
    \label{fig:hardware-os-virtualization}
\end{figure}

Virtualization can be divided into two categories, full virtualization and paravirtualization. Full virtualization is the older and traditional approach where the entire underlying hardware is virtualized to its core. This creates an image of the virtual system that is indistinguishable from a real one from the perspective of the application running inside it. As such, full virtualization allows the \Gls{vm} to be completely decoupled from the underlying hardware and for the guest \Gls{os} to run without any modifications.\ \cite{virtualization-basics, xen-virtualization, hypervisor-xen}. However, while virtualizing on such as low level is practical due to the flexibility it provides, it comes with a cost of performance. For this reason, paravirtualization was created to reduce the number of resources that are virtualized, effectively raising the level of abstraction slightly. With paravirtualization, a \Gls{vm} is presented with an image that is similar, but not identical, to the underlying hardware, and thus the guest \Gls{os} is aware it is running inside a \Gls{vm}\ \cite{xen-virtualization, hypervisor-xen}.

\subsection*{Benefits and Challenges}
Many of the benefits of virtualization stem from the fact that there exists an abstraction layer above the physical hardware. The abstraction layer decouples software from hardware, allowing for the use of vendors with different I/O in the same environment. This creates a more flexible and scalable computing environment as a whole where hardware is turned into a simple pool of computing resources\ \cite{virtualization-trends}. As such, there are numerous benefits of virtualization, a few of which are listed below.
\begin{itemize}
    \item \textbf{Isolation}\\ \Glspl{vm} are complete and independent systems and are thus able to run in isolation from one another. This includes isolating the \Gls{os}, filesystem and application stack.  Isolation is particularly useful for applications with strict security requirements, such as applications managing sensitive or confidential data. Isolation is also beneficial in scenarios that require reproducibility including testing and development environments\ \cite{vm-docker-reproducibility}.
    \item \textbf{Flexibility}\\ \Glspl{vm} are not tied to specific hardware* and can be moved between different physical resources. Flexibility is an important factor for many practical tasks when managing computing environments, such as moving workloads to a different machine when performing maintenance.\\ 
    \\
    * As described previously, this is not always true for paravirtualization since it does not virtualize every resource that makes up a \Gls{vm}.     
    \item \textbf{Resource utilization}\\ Virtualization allows for better utilization of hardware resources since workloads can be consolidated into fewer physical resources that might not have been fully utilized, and reduce the need to acquire additional hardware. The flip side of this is larger points of failure since multiple computing environments depend on the same physical resource. However, this risk is minimized considering the benefits described in \textit{Flexibility}\ \cite{virtualization-basics}.
    \item \textbf{Management}\\ \Glspl{vm} can be created and destroyed on demand and can be managed using software tools\ \cite{virtualization-basics}.
    \item \textbf{Virtual hardware}\\ A \Gls{vm} can be configured to use hardware that does not exist in the physical environment. This allows for better support for legacy applications or certain applications that are dependent on a specific hardware configuration. For instance, a \Gls{vm} can be configured with a virtual network card that mimics a physical network card that could not be acquired\ \cite{virtualization-basics}.
\end{itemize}

However, the portability made possible by virtualization comes at a cost of performance. Since a \Gls{vm} is a complete but virtualized system, many or all instructions issued are interpreted in software. This means that despite the level of abstraction layer used, there is an inescapable added overhead when using virtualization. A common hypervisor such as \Gls{kvm} (see\ \ref{hypervisors}) introduces an overhead of 5 to 10\%\ \cite{kvm-overhead}, and the overhead can be even larger when using a full virtualization approach.

\section{Understanding Hypervisors}
\label{hypervisors}
A hypervisor is a software layer that enables running multiple \Glspl{vm} on a single physical machine. It achieves this by allowing multiple operating systems (called \textit{guests}) to co-exist and share hardware\ \cite{virtualization-history, hypervisor-survey}. Hypervisors, often referred to as \Gls{vmm} are responsible for creating and managing \Glspl{vm} and for providing the necessary abstractions to the \Glspl{vm}.

Hypervisors can be categorized into two types, \textit{type 1} and \textit{type 2}, based on their operational context and how they interact with the underlying hardware. While both type 1 and type 2 hypervisors create virtual computing environments, they operate on different abstraction levels and have different use cases. Type 1 hypervisors, commonly referred to as \textit{bare-metal} hypervisors, run directly on top of the physical hardware and do not require an \Gls{os} underneath. This enables type 1 hypervisors to run with little to no overhead and scale up to terabytes of RAM and hundreds of CPU cores\ \cite{hypervisor-types-1-2}. However, it also means the hypervisor is entirely responsible for scheduling and ensuring that a \Gls{vm} is allocated the required system resources. For this reason, type 2 hypervisors, also known as \textit{hosted hypervisors}, were created. A type 2 hypervisor is installed on top of an \Gls{os} and uses \Gls{os}-level features to manage the resources for a \Gls{VM}. This makes the hypervisor more portable and simpler to use for an end-user since it can be installed on virtually any system. However, it also introduces an inevitable latency and overhead, which is why type 2 hypervisors are often used for development and testing purposes\ \cite{hypervisor-survey}. As such, type 1 and type 2 are inherently different kinds of hypervisors due to their respective trade-off with portability and performance. Figure \ref{fig:hypervisor-types} illustrates a high-level overview of the difference between a type 1 and type 2 hypervisor\ \cite{hypervisor-survey, hypervisor-types-1-2}.

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/type1_type2_hypervisor.pdf}
    \caption{Overview of type 1 and type 2 hypervisors. The figure illustrates how a type 1 hypervisor does not use an underlying \Gls{os} and instead runs directly on the hardware, while a type 2 hypervisor relies on an intermediary \Gls{os} to manage the underlying hardware.}
    \label{fig:hypervisor-types}
\end{figure}

Hypervisors can be further categorized into \textit{monolithic} and \textit{microkernelized} hypervisors\ \cite{hypervisor-comparion-2013}. Monolithic hypervisors are built as a single large software stack, managing all the \Glspl{vm} and their resources, and hosting drivers for all hardware access. Having all drivers reside within the hypervisor means that the hypervisor will only run on certain hardware. However, it also means no operating system is required at all, as the hypervisor acts as an operating system by supporting all the operations of its \Glspl{vm}. Microkernelized hypervisors, on the other hand, do not require device drivers to be present in the hypervisor. Instead, the drivers are installed in the form of a manager \Gls{vm}, often referred to as parent \Gls{vm}. Each \Gls{vm} created by such a hypervisor is thus referred to as a child \Gls{vm} since it uses the parent \Gls{vm}'s drivers to access the hardware. A microkernelized approach is more flexible than a monolithic, as it can allow the hypervisor to run on any hardware that supports the parent \Gls{vm}'s operating system. This distinction is similar to type 1 vs type 2 hypervisor, with one being more self-sufficient and the other more modular. However, it is important to note that monolithic and micro kernelized hypervisors are more concerned with the inner technical aspects of a hypervisor rather than the usability of the end-user.

% security rings definition
The concept of \textit{security rings} is often used to describe the level of privilege different part of a hypervisor has. A security ring represents a hierarchical protection domain, where the innermost ring (ring 0), usually reserved for the \Gls{os} kernel, has the highest privilege, and the outermost ring (ring 3), usually reserved for user applications, has the lowest privilege. The hypervisor itself is often placed in ring 0, its \Gls{vm} kernel space in ring 1, and the \Gls{vm} user space in ring 3. Security rings are thus used to define execution level to provide a system to limit the execution capabilities of \Glspl{vm} to regulate access to shared memory. The abstraction leads to more transparency regarding the execution level of certain applications and is key for ensuring that a \Gls{vm} maintains its isolation\ \cite{hypervisor-comparion-2013, microkernel-monolithic}. Figure \ref{fig:hypervisor-security-rings} illustrates this concept as a layered group of rings.

\begin{figure}
    \centering
    \includegraphics[width=0.7\textwidth]{img/hypervisor_security_rings.pdf}
    \caption{Overview of security rings in a hypervisor. The figure illustrates how the inner rings have higher privileges than the outer rings.}
    \label{fig:hypervisor-security-rings}
\end{figure}

\subsection*{Key Hypervisors}
\label{key-hypervisors}
There are several hypervisors available, each with its own set of features and capabilities, some of which are listed below.

\begin{itemize}
    \item \textbf{VMware ESXi}\\A lightweight type 1 monolithic hypervisor often used in enterprise environments. ESXi is known for its ultra-lightweight footprint utilizing just a few hundred megabytes and its ability to run directly on the hardware without an underlying \Gls{os}\ \cite{hypervisor-comparion-2013}.
    \item \textbf{Microsoft Hyper-V}\\A type 1 microkernelized hypervisor that is part of the Windows Server operating system. As a microkernelized hypervisor, Hyper-V differentiates between parent and child \Glspl{vm} and uses the parent \Gls{vm} to manage hardware access for the child \Glspl{vm}\ \cite{hypervisor-comparion-2013, hypervisor-hyper-v-esxi}.
    \item \textbf{Citrix Hypervisor}\\Formerly known as XenServer, is a type 1 microkernelized hypervisor based on the Xen Project. It runs using a modified Linux kernel in a \Gls{vm}, called Dom0, that manages hardware I/O and interacts with the other \Glspl{vm}, called DomU guests\ \cite{hypervisor-xen, hypervisor-comparion-2013}. 
    \item \textbf{\Gls{kvm}}\\A type 1 hypervisor that is part of the Linux kernel inheriting many of its functionalities from the Linux kernel such as the scheduler and memory management. \Gls{kvm} is free and open-source and has thus become a popular alternative to proprietary hypervisors, such as VMware ESXi and Microsoft Hyper-V\ \cite{hypervisor-comparion-2013, hypervisor-kvm}. 
\end{itemize}

\section{VM Management Essentials}
A management system for \Glspl{vm} has tools and processes for creating, deploying and managing \Glspl{vm}. The management system is responsible for managing the entire life cycle of a \Gls{vm}, including provisioning, deployment, monitoring and scaling. This is comparable to the role of a hypervisor that also is responsible for creating and managing \Glspl{vm}. However, a \Gls{vm} management system operates on a higher level to provide further abstraction on top of hypervisors, often supporting a range of hypervisors. A \Gls{vm} management system can thus be seen as a control point for the underlying hypervisors\ \cite{cloudframeworks-and-hypervisors}. However, certain hypervisor features, such as live migration, could be have unique implementations for different hypervisors. This creates a possible trade-off for a \Gls{vm} management system where it can choose to implement fewer but homogeneous operations, or more tunable operations that implement hypervisor-specific features\ \cite{openstack-opennebula-comparison}.

\subsection*{Concepts and Features}
With a higher abstraction layer, a \Gls{vm} management system brings a set of concepts to simplify the management of \Glspl{vm}. Many such features and concepts exist and can be broadly grouped as clustering, availability and scalability\ \cite{cloudframeworks-and-hypervisors}. 

Clustering involves grouping a set of resources, such as \Glspl{vm} or storage nodes, to provide an image that is perceived as a single much larger resource. By allowing multiple nodes to cooperate and share resources, clustering can allow for a system that can grow horizontally, meaning increasing the number rather than the size of each hardware resource. \Gls{hpc} is a common example of compute clustering where computing resources are grouped to gain performance greater than that of a single computer. This enables \Gls{hpc} clusters to process large datasets and complex calculations that would be infeasible for a single computer. As such, the computational limit is significantly lower when expanding the size of a single server rather than the number of servers. Moreover, \Gls{hpc} clusters are designed such that they allow for the distribution of computational tasks where larger tasks are divided into smaller subtasks. This might not be possible for all computational tasks, and \Gls{hpc} clusters often require a specific format on input to fully utilize the capabilities of the system\ \cite{google-hpc-overview,oracle-hpc-overview}. 

Availability refers to the ability to ensure a service is kept active or running. This is commonly measured by ensuring the end user can access certain applications without disruption. Availability is typically implemented using a set of \Glspl{az}, which represent a logical grouping of resources such that each \Gls{az} maintains a copy of the service. If the underlying hardware fails in one \Gls{az}, another \Gls{az} can continue to serve requests. This redundancy allows for outages to be isolated and maintenance to be performed without affecting the uptime of service\ \cite{cloudframeworks-and-hypervisors, azure-availability-zones}. Figure\ \ref{fig:availability-zones} illustrates the concepts of an \Gls{az} with an overview of how 3 \Glspl{az} provide redundant access to the same service. 

Scalability involves technologies to ensure resources can manage an increase or decrease in computational loads. A \Gls{vm} management system often facilitates both scaling up and scaling out. Scaling up, also referred to as vertical scaling, involves increasing or decreasing the performance specification of a resource. For instance, a \Gls{vm} can be scaled up by increasing the number of CPUs or the amount of memory. Scaling out, also referred to as horizontal scaling, involves adding or removing entire resources, such as \Glspl{vm}. This means, that instead of upgrading a single \Gls{vm}, additional \Glspl{vm} are added to the system to handle the increased load. This approach is often regarded as more scalable than scaling up since it does not involve any downtime. Scaling up is also in practice usually quite limited since it requires the application running inside the resource to fully utilize its resources, while scaling out expects normal operational load on each instance. The common argument against scaling out though, is that the application might not be designed to operate in several instances, such as a single-node database\ \cite{scale-up-out-case-study, scale-up-out-ibm}.

For example, an \Gls{iaas} platform typically provides tools to create and manage \Glspl{vm}, their storage and virtual networks for them to reside in. As such, an \Gls{iaas} platform is a foundation for building and managing cloud environments\ \cite{iaas-overview}.

Together, the three cloud service models form a hierarchical relationship with each other, which means a \Gls{saas} service can be built on top of a \Gls{paas} service, which in turn can be built on top of an \Gls{iaas} service. \subsection{Comparison of VM Management Systems}
Managing \Glspl{vm} can be done in several ways depending on the required level of availability and scalability, as well as economic factors. There are numerous \Gls{vm} management systems available, each with its own set of features, capabilities and limitations, and they can be divided into two main groups depending on which type of system it is part of: public and private cloud. Public clouds refer to the commercial alternative maintained and operated by a third-party company. The cost of hosting resources on these platforms is usually tied to the usage and not the hardware itself. Private clouds on the other hand refer to software that can be installed on a group of servers hosted by the organization using them. For this reason, private clouds are commonly open-source and free to use. Some of the most prevalent private cloud platforms are OpenStack\ \cite{openstack-overview}, CloudStack\ \cite{cs-overview}, and OpenNebula\ \cite{openstack-opennebula-comparison}, and some of the most popular public cloud platforms are Amazon, Azure, and Google Cloud. The public clouds offer far more than just \Gls{vm} management. However, to keep the focus of the comparison, only the \Gls{vm} service of each public cloud will be brought up. These are Amazon EC2\ \cite{ec2-overview}, Azure Virtual Machines\ \cite{azure-overview} and Google Compute Engine\ \cite{gce-overview}.

\subsubsection*{OpenStack}
OpenStack is an open-source \Gls{iaas} platform that provides a large set of tools for managing virtualized resources. Its architecture is built to be modular using a component-based approach with each of the services it provides running as a separate component. The components then communicate in a distributed manner using technologies such as message queues\ \cite{openstack-opennebula-comparison}\ \cite{openstack-overview}.

While OpenStack includes a large set of components, only the three major and most relevant to this project are brought up below.
\begin{itemize}
    \item \textbf{Nova}\\ The \Gls{vm} service that provides a set of tools for managing the life-cycle of \Glspl{vm}, such as creating, deleting and migrating \Glspl{vm}. Nova is a relatively large component that is split into multiple parts each with a \textit{shared-nothing} and \textit{messaging-based} policy, meaning they are independent and communicate with other components using message queues. 
    \item \textbf{Swift}\\ The service that manages static data in a scalable and redundant manner. This is used by \textit{Nova} to store \Gls{OS} images and backups.
    \item \textbf{Glance}\\ The image service that provides a catalog of \Gls{vm} images that can be used when creating new \Glspl{vm}.  
\end{itemize}

In summary, OpenStack supports many features regarding \Gls{vm} management as well as a wide range of hypervisors. It is built to be efficient and scalable and has thus gained enormous traction in the industry\ \cite{openstack-opennebula-comparison}.

\subsubsection*{CloudStack}
CloudStack is an open-source \Gls{iaas} platform and cloud computing software that manages virtualized resources including \Glspl{vm}, virtual networks and storage resources. It is used primarily to build and manage private cloud environments\ \cite{cs-overview}. Architecture-wise, CloudStack is built in a monolithic fashion, encapsulating all the management features into two software packages called the management server and agent. The management server can be viewed as a controller node and is responsible for sending out instructions regarding the life-cycle of \Glspl{vm}, such as creating, scaling and deleting them. The agent, on the other hand, is installed on each node designated to running \Glspl{vm} and is responsible for low-level management tasks of the \Glspl{vm} on that node, such as interacting with the underlying hypervisor. The agent can thus be viewed as a intermediator between the hypervisor and the management server that enables CloudStack to be hypervisor agnostic. While CloudStack is a monolithic system, it is designed to be horizontally scalable and can be deployed in a distributed manner with multiple management servers\ \cite{cs-overview}\ \cite{cs-live-migration}.

CloudStack supports a large variety of hypervisors, such as VMware, \Gls{kvm}, and XenServer, as well as storage services\ \cite{cs-overview}. In summary, CloudStack provides an efficient public cloud environment similar to OpenStack. However, CloudStack does not scale to the same size an OpenStack due to its simple design\ \cite{openstack-opennebula-comparison}.

\subsubsection*{OpenNebula}
OpenNebula is an open-source \Gls{iaas} platform that is mainly used to build private clouds. Contrary to OpenStack, OpenNebula is built to be simpler with as little overhead as possible, which has made OpenNebula a common choice for larger-scale datacenters\ \cite{openstack-opennebula-comparison}. Its architecture is designed to be modular and is split into three parts as described below. \ \cite{openenbula-tele-teaching, iaas-comparisons}.
\begin{itemize}
    \item \textbf{Tools layer}\\The high-level layer that provides a set of interfaces such as a \Gls{cli} and web-based user interface to interact with the system.
    \item \textbf{Core layer}\\Provides core functionalities to the system, such as host management, \Gls{vm} management and virtual network management.
    \item \textbf{Driver layer}\\The low-level layer that provides an abstraction on top of the underlying infrastructure, including networks, storage and hypervisors. Having this layer allows OpenNebula to support a wide range of providers, as well as adapt quickly to new technologies.
\end{itemize}

In summary, OpenNebula provides a simple yet efficient approach to \Gls{vm} management. Its simplicity and low overhead have made it a popular choice in the industry\ \cite{opennebula-website}.

\subsubsection*{Amazon EC2}
Amazon Elastic Compute Cloud (EC2) is a web service that provides on-demand compute resources over the cloud\ \cite{ec2-overview, ec2-overview-web}. EC2 is part of the cloud platform called Amazon Web Services (AWS) and is used to host general-purpose applications. Since AWS is a public cloud platform, the technical specifications regarding hypervisor technology and such are not visible to the end user. 

AWS provides a large set of templates that can be used when creating \Glspl{vm}, called  Amazon Machine Images (AMIs). Which images is used depends on the use case and how much a user is willing to pay\ \cite{ec2-overview}.

\subsubsection*{Azure Virtual Machines}
Azure Virtual Machines is a service provided by the public cloud platform Azure to manage \Glspl{vm}\ \cite{azure-overview}\ \cite{azure-cost-effective}. Similar to AWS, Azure provides a set of preconfigured templates that can be used when creating \Glspl{vm} are chosen depending on the use case, such as general-purpose or \Gls{hpc}. Any technical specifications on the architecture of Azure Virtual Machines are not visible to the end user \cite{azure-overview-web}.

\subsubsection*{Google Compute Engine}
Google Compute Engine is a service provided by Google Cloud to manage computing resources in the cloud\ \cite{gce-overview-web}. Similar to both AWS and Azure, Google Cloud also provides a set of templates when creating \Glspl{vm} depending on the specific workload\ \cite{gce-overview}. 

\subsubsection{Comparison}
\label{vm-management-comparison}

OpenStack, OpenNebula, and CloudStack are all examples of software that can be used to build both private and public cloud platforms. A private cloud is an on-premise cloud that is hosted within an organization and is, more often than not, dedicated to internal use. These systems are also often free and open-source, making them a go-to tool for many organization that seeks to host internal services. The systems are also highly configurable to work on most existing infrastructure. However, being highly configurable could be seen as a double-edged sword since it leaves much of the configuration of the system to the user, such as preconfigured \Gls{vm} templates.\ \cite{public-private-cloud, hybrid-cloud}. 

Amazon EC2, Azure Virtual Machines and Google Compute Engine are examples of \Gls{vm} management systems in public cloud platforms. They are designed with the end user in mind by providing a complete abstraction layer over the underlying hardware and hypervisors. This means that any configuration regarding the hypervisor and its settings is taken out of the equation. All three cloud platforms can be viewed as complete implementations of the previously mentioned private cloud platforms, including hardware. For this reason, the public cloud platform has seen enormous traction in the industry.
However, this also creates a trade-off where the user has less control and customization but instead gains a more user-friendly and complete system\ \cite{public-private-cloud}.

Private and public clouds both provide solutions for managing virtual resources and have their respective advantages. Public clouds scale better performance-wise since they make use of large data centers. However, it inevitably involves a third party which might not be suitable for certain use cases, such as for applications managing sensitive data. Private cloud, on the other hand, comes with a larger initial cost, both in terms of purchasing hardware but also to acquire the required skillset among the personnel managing the system. Which solution to choose depends on the specific needs of the organization, such as the level of control required, customization needs and security requirements\ \cite{hybrid-cloud}. 

\section{Containers and Kubernetes: A Paradigm Shift}
\label{containers-kubernetes}
Besides the common cloud service models (\Gls{iaas}, \Gls{paas}, and \Gls{saas}), another service model has emerged, called \gls{caas} \cite{caas}. \gls{caas} is a container-based cloud service model that has gained significant traction due to the lightweight and portable nature of a container, allowing for simple application bundling that can run in a vast range of environments\ \cite{microservices-overview}. 

Similar to \Glspl{vm}, containers are often used together with a management platform, such as Kubernetes. More specifically, Kubernetes is a container orchestration platform that manages the life cycle of containers and any communication between them. Kubernetes will be described in technical detail in a later section, see \ref{sec:kubernetes-orchestration}

A common use case for containers is to form the base for a microservice architecture. Microservices is a software design pattern that uses a set of small, independent and loosely coupled applications to provide a scalable service with minimal downtime\ \cite{microservices-overview}. 

\subsection{Containers Explained}
\label{containers}
Containers are a form of virtualization that offers a lightweight alternative to \Glspl{vm}. Unlike \Glspl{vm}, containers do not require a hypervisor to run and instead run on top of an \Gls{os} and share the \Gls{os} kernel with other containers. By not requiring a hypervisor, containers can run with less overhead than \Glspl{vm}, making their life cycle relatively fast and efficient compared to \Glspl{vm}\ \cite{container-vm-overview, container-k8s-benchmark}. The differences between container and hypervisor virtualization are visualized in figure \ref{fig:container-vm-architecture-2}.

    \centering
    \includegraphics[width=0.9\textwidth]{img/vm_container_virtualization.pdf}
    \caption{Overview of \Glspl{vm} and containers. The virtualization layer is placed on a lower level for \Glspl{vm} than for containers, illustrating the difference in their respective virtualization techniques.}
    \label{fig:container-vm-architecture-2}
\end{figure}

\label{dockerfile}
A container is created from a container image, which is a lightweight package that is used when configuring the virtual environment for the container. It encapsulates the application together with any dependencies such that it runs on any platform, which makes the container itself portable and consistent. The image itself is created using a configuration file called \textit{Dockerfile}*, which describes a parent image, such as an \Gls{os} image, together with several steps required to produce the container image. Once the container image is created, it can be used as a parent image in any other configuration files, creating a chain of dependencies.

To enable caching, a container image can be divided into a series of layers. Each layer represents a change to the parent image, such as adding a file or installing a package, and thus acts as a checkpoint. This enables container images to be built incrementally, where layers are only rebuilt when a change is made\ \cite{container-docker-layers, container-dockerfile, container-docker-reproducibility}. Figure\ \ref{fig:container-image-layers} illustrates the concept of container image layers, where each operation represents a change to the parent image.

    \label{fig:container-image-layers}
\end{figure}

\label{container-registry}
Moreover, container images are often stored in container registries, which provide central storage and are used to distribute the images. A container registry often comes with tools to manage its images, such as tools for version control and scanning for malicious images. Another common event-based feature is webhook. It acts as a message broker for predefined events, such as notifying an external system that a new image was pushed. Together these features enable a container registry to integrate well in a cloud-native environment and development pipelines\ \cite{container-registries, container-registries-redhat}.

Originally, containers were designed to run on top of a technology developed by the company that invented containers called the \textit{Docker Engine}. Docker Engine is a container runtime that creates and manages containers similar to a hypervisor. However, to standardize an interface for container communication, the Open Container Initiative (OCI) was created, which defines a group of standards that aims to make containers portable across different platforms\ \cite{oci}. It is split into three standards called the runtime specification, image specification and distribution specification. From these specifications, the \textit{containerd} runtime was created, which aims to provide a more transparent and open-source alternative to the Docker Engine. Since it adheres to the standards, it allows for simpler integration with tools and services built on top of it. Today, many large systems use containerd under the hood, such as Kubernetes\ \cite{k8s-overview}

\subsection{Orchestration Using Kubernetes}
\label{sec:kubernetes-orchestration}
Kubernetes is an open-source container orchestration platform that manages containers. It uses a set of abstraction layers to manage the life cycle of containers and all the communication between them. It provides a complete environment for the containers to run in, including scheduling, scaling and networking\ \cite{container-vm-overview}.

Kubernetes defines many abstractions to simplify container orchestration and communication, where the most basic ones are \textit{Pods} and \textit{Services}. A Pod is the smallest deployable unit in Kubernetes, which groups one or more containers into a single unit that can be run on nodes. This means that Kubernetes ensures all containers in a Pod run on the same host. The encapsulation allows coupled containers to reside on the same host and enables efficient communication between them. A common design pattern for containers in a Pod is the sidecar design pattern. It involves designating one or more containers to provide some additional functionality to a main container, such as logging or monitoring\ \cite{k8s-docs-pods, k8s-docs-sidecars}. A Service represents a communication entry point that is placed in front of one or more Pods. This means that a Service targetting a group of nodes acts as a load balancer, while a service that targets a single Pod is comparable to port-forwarding. Separating the communication into a separate layer is especially useful in scenarios where the Pods can serve client requests independently, and that the requesting client does not care which Pod it targets. Such Pods are referred to as \textit{fungible} Pods and are more common for cloud-native applications. Additionally, using Services in front of a group of Pods can lead to simpler deployment of updates, as the Pods can be updated one at a time using a rolling update strategy\ \cite{k8s-docs-services}. 

To enable a more consistent deployment of Pods, Kubernetes provides an abstraction on top of the Pods called \textit{Deployments}. A Deployment is given a desired state in which Pods should be and actively operates towards it. This means that if a Pod fails, the number of running Pods no longer matches the desired number of Pods, and the Deployment should create a new Pod. By following a desired state, Deployments are commonly used when the load of an application is irregular. For instance, a website might face heavier traffic during the day than during the night. In this scenario, the Deployment could be adjusted to scale down the number of Pods\ \cite{k8s-docs-deployments, k8s-overview}. 

So far, none of the scenarios have included a state that should be kept between or within Pods. Containers are inherently stateless by design, meaning that any data inside a container is lost upon deletion. Kubernetes primarily provides three abstractions to manage the state of an application: \textit{PersistentVolume}, \textit{PersistentVolumeClaim} and \textit{StatefulSet}. The former two are used in conjunction, where a PersistentVolume defines a storage target, such as the file path in a network storage server or on the local node, while a PersistentVolumeClaim acts as the intermediator between a PersistentVolume and a Pod. There is a one-to-one relationship between a PersistentVolume and PersistentVolumeClaim to ensure consistent access to the underlying storage, but also to provide a mechanism to notify Kubernetes that storage is no longer used\ \cite{k8s-docs-peristent-volumes}. A StatefulSet, on the other hand, is more comparable to a Deployment with a specific order for the Pods to start in. As previously described, a Deployment ensures that a desired number of Pods are running, but it does not guarantee the order in which they start or stop. A StatefulSet solves this by providing a more predictable strategy where each Pod has its own identity that is preserved between restarts. For this reason, a StatefulSet is often used for applications that define roles for a group of nodes, such as a database that must start a primary node before any secondary nodes\ \cite{k8s-docs-statefulsets, k8s-overview}.

Figure \ref{fig:k8s-overview} presents a brief overview of the Kubernetes abstractions and how they interact with one another. 

\begin{figure}
    \centering
    \includegraphics[width=1\textwidth]{img/kubernetes_overview.pdf}
    \caption{Brief overview of Kubernetes abstractions including many of the core components. The figure illustrates the communication between the different components and how storage is managed in Kubernetes.}
    \label{fig:k8s-overview}
\end{figure}
 
In some more sophisticated scenarios, these built-in abstractions might not suffice since they might not be able to model more complex types. For such scenarios, Kubernetes provides \Glspl{crd}, a form of extension, that allows user-defined types with logic and validation. For instance, a \Gls{crd} can define an abstraction similar to a Deployment, but it enforces certain sidecar containers to exist using custom validation rules. With \Glspl{crd}, the Kubernetes ecosystem can be tailored for specific use cases forming a more feature-rich environment\ \cite{k8s-crds, k8s-overview, k8s-crd-webiste}. Kubernetes includes many more abstractions, each with its use case. However, presenting an exhaustive list falls out of the scope of this project.

The design and workflow of Kubernetes resources are inherently declarative, meaning that instead of providing a set of instructions to move a resource into a specific state, a complete state is given in the form of configuration files called manifests. The manifest defines metadata, relationships and dependencies for a resource, and Kubernetes controllers are responsible for ensuring this state is applied and up to date\ \cite{k8s-docs-controllers}. Depending on the kind of abstraction, different logic is applied to ensure the resource specification matches the manifest. For instance, a Pod might be rescheduled to ensure it matches the manifest\ \cite{container-vm-overview, k8s-overview}. 

Architecture-wise, Kubernetes uses a client-server approach. The server, often referred to as the control plane, manages the state of the cluster, including scheduling and scaling, as well as networking. Since it consists of the API server, it can be viewed as the gateway to cluster, processing and acting upon the request of clients\ \cite{k8s-docs-components}. The client, often referred to as the worker node, receives Pod-related instructions from the control plane. Contrary to the control plane, a worker node consists of a container runtime, which allows it to run containers\ \cite{k8s-docs-kubelet, k8s-overview, k8s-overview, k8s-docs-kubeproxy}. 

From an end user's perspective, Kubernetes resources are managed through the API server, which in turn notifies its workers. An end user can send requests using a \Gls{cli} tool such as \textit{kubectl} or using a programmatic interface through a REST \Gls{api}\ \cite{k8s-overview}. 

Through its abstractions, life cycle management and declarative client-server architecture, Kubernetes provides a complete and feature-rich platform for container orchestration. This has led to enormous traction in the industry and has Kubernetes a popular choice for container infrastructure\ \cite{k8s-it-administration}. 

\section{Containers vs. Virtual Machines}
Virtualization has brought many benefits within the realm of cloud computing by allowing hardware to be split or grouped into virtual hardware. It has been the key in enabling what is today referred to as cloud computing and cloud platforms. There are mainly two technologies that were made possible due to virtualization: \Gls{vm} and containers. \Glspl{vm} are complete and isolated systems with their own \Gls{os} and containers are a lightweight alternative to \Glspl{vm}. This creates a trade-off where depending on the use case, a \Gls{vm} might be more suitable or vice versa\ \cite{container-vm-overview, container-vm-comparison}. 

\subsection{Comparative Overview}
\label{container-vm-isolation}
The main difference between \Glspl{vm} and containers revolves around isolation and how independently they can run. As previously described \Glspl{vm} provides a complete system while containers share the \Gls{os} kernel with other containers. This can lead to more efficient resource sharing among the containers, but it also increases the attack surface of containers. A vulnerability found in the underlying \Gls{os} might compromise the containers running on the host. A \Gls{vm} reduces this risk by removing the \Gls{os} as a dependency\ \cite{container-kata-security}.

Isolation is also important when considering the scenario of co-location, meaning containers and \Glspl{vm} running on the same host. Since containers are allocated resources from the \Gls{os}, they are allocated as if they were ordinary applications. In many scenarios, this leads to efficient resource utilization, but sharing resources this way can be a double-edged sword. For instance, containers are more vulnerable than \Glspl{vm} to the noisy neighbor problem, since a co-located container could consume an unproportionate amount of the host's resources, which could lead to resource starvation among the other containers running on the host. Some solutions exist for this, such as the \Gls{os}-level feature \textit{cgroups}. A cgroup is a mechanism that limits, prioritizes and isolates resource usage for a process\ \cite{cgroups, vm-noisy-neighbor, container-vm-scale}. \Glspl{vm}, on the other hand, are not as susceptible to the noisy neighbor problem since they are allocated a static amount of resources from the host. However, it is important to note that a \Gls{vm} does not reduce the noisy neighbor problem for all resource types. For instance, unproportionate consumption of network traffic is still an issue for both \Glspl{vm} and containers\ \cite{container-vm-comparison}. 

Overcommitment refers to a mechanism that allocates more virtual resources than what is physically available. This is a common practice when using virtualization, as hardware resources are often not fully utilized. Containers use mechanisms built into the \Gls{os}to achieve resource overcommitment, which involves multiplexing and virtual memory. However, resource overcommitment is more challenging for \Glspl{vm}. While CPU overcommitment is achieved using \Gls{vcpu}, memory overcommitment is solved by more sophisticated methods since a hypervisor allocates a static amount of memory to each \Gls{vm}. One such method is called memory ballooning and it involves sharing a subset of each \Gls{vm}'s memory with other \Glspl{vm}. It works by reclaiming memory that is not used inside a \Gls{vm} and allocating it to another \Gls{vm}. Then, when the first \Gls{vm} requires the now reclaimed memory, the hypervisor ensures the \Gls{vm} is either given back its memory or allocated a new chunk. As such, memory ballooning operates under the assumption that certain parts of the allocated memory are not used\ \cite{vm-ballooning}. However, it is important to note that a technique like memory ballooning does not directly solve memory overcommitment, but instead reduces the need for it\ \cite{container-vm-comparison, vm-overcommitment}. Figure \ref{fig:memory-ballooning} illustrates the processes of deflating the memory of a \Gls{vm} and how it is used in another \Gls{vm}.

\label{container-vm-live-migration}
Live migration is a technique used to move a running resource to a new host without stopping and starting it. The technique is often used when balancing load over hosts or when performing maintenance. For \Glspl{vm}, live migration is a mature and widely used technique and works by copying memory pages from the source host to the destination host and then transferring the \Gls{vm} from the source hypervisor to the destination hypervisor. Since \Gls{vm} disks are often stored on shared storage, such as a \Gls{san}, the \Gls{vm} disk is not transferred but instead reattached after the live migration is finished. This usually requires some form of time-freezing to reduce the risk of data corruption. However, a live migration operation is usually fast enough to even notice\ \cite{vm-live-migration}. Live migration for containers, however, is not as mature as for \Glspl{vm}. A container requires \Gls{os}-level process migration, which is less reliable. However, several projects aim to provide live migration for containers, such as \Gls{crui}. \Gls{crui} works by checkpointing the state of a container, including its memory and file descriptors, and then restoring the state on another host. While this is considered as the defacto container live migration, it is more comparable to a snapshot operation than a live migration operation\ \cite{container-criu-overview, container-criu-linux, container-criu-web}. Another approach for live migrating containers involves terminating them on one host and restarting them on another. This strategy assumes the container is stateless or utilizes shared storage for its state, as is often used in Kubernetes since it does not provide container live migration. Instead, Kubernetes facilitates persistent storage through its persistent storage abstractions, such as PersistentVolumes and PersistentVolumeClaims. This means a user maps portions of the container's state to shared storage and assumes the rest can be discarded when a container is terminated\ \cite{k8s-overview}.

\subsection{Decision Factors}
The choice between containers and \Glspl{vm} depends on the specific use case, and there are several factors to consider when deciding which technology to use. Some of the core factors are security, scalability and legacy support\ \cite{container-vm-comparison}. 

As previously described, using a container implies a larger attack surface than for a \Gls{vm} due to sharing the \Gls{os} kernel. For this reason, \Gls{vm} have commonly been associated with stronger security since they are less prone to security vulnerabilities. This makes \Glspl{vm} an excellent candidate for application with stricter security requirements. Some examples include applications managing sensitive or confidential data\ \cite{container-vm-decision-redhat}. Another use case that is made possible due to a \Gls{vm}'s strong isolation is running unsafe or untrusted code. This use case is often referred to as sandbox and is useful when evaluating potentially malicious code that could otherwise risk harming the host system. Untrusted code could also involve user-submitted code, such as in GitHub pipelines that use \Glspl{vm} to execute actions based on some predefined event. This could be tests or some other development-related tasks created by a user\ \cite{gh-selfhost-runner-sec}. Using a \Gls{vm} in these scenarios ensures that the untrusted code does not affect the hosting system\ \cite{vm-decision-kong, sandboxing}

Configuration drift is another aspect to consider when deciding between using a \Gls{vm} or a container. It is a phenomenon that occurs when the configuration of a system changes over time, such that it could potentially lead to security risks. This is usually caused by continuous manual intervention in a system, where the summarized effects of all the changes made could be an unstable system\ \cite{configuration-drift-tripwire, configuration-drift-wiz}. This is more common for \Glspl{vm} than for containers, as a \Gls{vm}'s state is saved indefinitely. A container mitigates this problem by having its entire state defined in the Dockerfile (see\ \ref{dockerfile}), where its dependencies and configuration are more transparent to the person managing it. A container state is thus discarded once it is stopped, which allows containers to have a more consistent environment than \Glspl{vm}. This is further supported by the fact container registries (a central location for container images, see\ \ref{container-registry}) usually include image-scanning tools to find vulnerabilities in a container image itself\ \cite{container-vm-decision}. 

Scalability is a factor that is concerned with how well a system adapts to an increase or decrease in load. This can be measured in many ways where one being the overhead each additional computational unit has. Containers being lightweight and sharing hardware resources more efficiently implies they can scale up and down both quicker but also with a smaller resource footprint. However, as previously described, they are more susceptible to the noisy neighbor problem, where co-located containers could interfere\ \cite{container-vm-scale, container-vm-decision}. Moreover, scalability is also related to cloud-native applications, which are applications that are designed to run in the cloud\ \cite{cloud-native}. A cloud-native application is characterized by being horizontally scalable, stateless, and resilient, and is often built using microservices, see\ \ref{containers-kubernetes}. Cloud-native applications thus require small and lightweight environments making containers a natural fit.

Legacy support is another relevant factor to consider when comparing \Glspl{vm} and containers. It can be measured by how well a system can keep running old software or software depending on older hardware. In both cases, \Glspl{vm} is a good fit since they are a complete system. A \Gls{vm} can simply replicate the state of a legacy system, even including the hardware that was used. This allows an organization to acquire new infrastructure while still maintaining its legacy applications. Containers are also able to support legacy applications, but not to the same extent, as they are inevitably using a modern kernel\ \cite{container-vm-decision,vm-decision-microsoft}.

\subsection{Towards Stronger Container Isolation}
So far, containers have been described as having weaker isolation than \Glspl{vm} due to it sharing the underlying \Gls{os} kernel. However. several projects have been created to tackle this issue without completely resorting to a \Gls{vm}. These include Firecracker, Kata Containers and gVisor.

Firecracker is a \Gls{kvm}-based hypervisor that creates lightweight \Glspl{vm}, called microVMs. Compared to an ordinary \Gls{vm}, a microVM is much more lightweight and can start in a fraction of a second. Firecracker achieves such speeds through optimizations and a heavy stripped-down guest\ \cite{container-runtimes-overview, container-firecracker}.

Kata Container is another \Gls{vm} alternative that aims to provide a lightweight environment, while still having strong isolation properties. Unlike Firecracker, a Kata Container still uses a container, but wraps it using a lightweight version of a \Gls{vm}, which enables Kata Container to act as a runtime, such as containerd. This means Kata Container is simpler to implement than Firecracker since it can run ordinary containers\ \cite{container-runtimes-overview}. 

Another approach to strong container isolation is gVisor, an extension to a container that enforces stricter isolation by restricting system calls. This approach differs from both Firecracker and Kata Container in that it does not use a \Gls{vm} at all, and instead relies on intercepting system calls and file access requests\ \cite{container-gvisor-firecracker}.

In summary, these three technologies provide different approaches to the same problem. Firecracker is more \Gls{vm}-like, while gVisor is most similar to a container. However, due to the varying added overhead, they are yet to be widely adopted\ \cite{container-runtimes-overview}. Figure \ref{fig:secure-containers-overview} illustrates the differences between how Kata Containers and gVisor wrap a container to provide a more secure execution environment.

\section{Extending Kubernetes with KubeVirt}
\label{sec:kubevirt}
Kubernetes was recently extended to accommodate \Glspl{vm} as a native resource through the KubeVirt project. KubeVirt is designed to integrate well with the Kubernetes ecosystem and provide a similar interface to \Glspl{vm} as any other Kubernetes resource. \Glspl{vm} created with KubeVirt are scheduled and communicate as ordinary Pods, allowing them to blend into the environment\ \cite{k8s-scheduler}. This means that a Kubernetes resource that integrates with a Pod can also integrate with a KubeVirt \Gls{vm}. For instance, Services can expose a \Gls{vm} externally and PersistentVolumeClaims can maintain its persistent storage. While KubeVirt uses Pods to wrap a \Gls{vm}, a \Gls{vm} abstraction is implemented as a \Gls{crd}. This allows KubeVirt to perform validation that is specific to a \Gls{vm}, such as referencing a non-existing disk\ \cite{kubevirt-architecture, kubevirt-validation}.

KubeVirt's design pattern follows Kubernetes conventions, using a declarative approach to managing the \Glspl{vm} and any other resources revolving around it, such as snapshots and live migration. A \Gls{vm} is split into a \textit{VirtualMachine} and \textit{VirtualMachineInstance}, where the VirtualMachine represents a desired state and the VirtualMachineInstance describes a running state. This relationship is similar to that of a StatefulSet and its Pods\ \cite{kubevirt-vmi}. Moreover, KubeVirt implements a \Gls{vm} controller to ensure the desired state is met, which is responsible for requesting an environment in which the \Gls{vm} can run, but also update the running \Gls{vm} when its desired configuration changes.\ \cite{k8s-docs-kubevirt, kubevirt-architecture}.

KubeVirt's declarative approach stretches to aspects that have been imperative operations in traditional VM management systems. The most prominent examples of this are live migrations and snapshots that are both requested using configuration files. As such, they are implemented as \Gls{crd} that triggers live migration or snapshot operations once created\ \cite{kubevirt-snapshot,kubevirt-live-migration}. This is a novel take on operations that traditionally have been imperative. However, the processes of triggering an operation upon resource creation have existed in Kubernetes for a long time. For instance, a storage device is connected to a Pod when a PersistentVolumeClaim is created\ \cite{k8s-persistent-volumes}.

A consequence of being built on top of Kubernetes is that KubeVirt-related resources can be managed using the same tools used for other resources, such as the \Gls{cli} tool kubectl. However, certain operations are still unique to \Glspl{vm} and are not yet supported by kubectl. This includes accessing a \Gls{vm} over \Gls{ssh} or \Gls{vnc}, which are inherently imperative operations. For this reason, KubeVirt provides a complementary \Gls{cli} tool that implements these missing features\ \cite{kubevirt-virtctl}.

\subsection{KubeVirt Technical Overview}
\label{sec:kubevirt-technical-overview}
On a technical level, KubeVirt does not implement a new deployable unit for their \Glspl{vm} and was instead designed to encapsulate them using Pods (see \ref{sec:kubernetes-orchestration}). This design has allowed KubeVirt \Glspl{vm} to integrate well with other Kubernetes resources such as Sevices and PersistentVolumeClaims. 

KubeVirt can be split into four core components: \textit{virt-controller}, \textit{virt-handler}, \textit{virt-launcher} and the \textit{libvirt daemon}. The virt-controller and virt-handler are both concerned with managing the life cycle of the \Glspl{vm}, where the virt-controller is responsible for creating the Pod for the \Gls{vm} and the virt-handler is responsible for creating the \Gls{vm} itself. The virt-launcher manages the runtime environment for a \Gls{vm} making it responsible for requesting start and stop operations, as well as monitoring. Finally, the libvirt daemon (libvirtd) provides the interface to manage the \Glspl{vm} hosted by the underlying \Gls{kvm} hypervisor. The relationship between the three components is illustrated in figure \ref{fig:kubevirt-components}, where the virt-controller operates closest to Kubernetes \Gls{api}-server, virt-handler operates on every node to communicate with the underlying hypervisor, and virt-launcher operates as a sidecar container in every Pod to manage the runtime environment of the \Gls{vm}\ \cite{kubevirt-architecture, k8s-docs-kubevirt}.

\subsection{KubeVirt and GitOps}
GitOps is a set of practices that enables using Git as a storage for the desired state of a system. Using a storage managed by Git allows resources to be version-controlled and audited. This approach keeps developers and maintainers in focus since it allows their resources and codebase to exist in the same system. For some scenarios, the resources managed in Git might even be the same resources that host their application\ \cite{gitops-evolution}.
For this reason, GitOps becomes a natural fit for KubeVirt since \Glspl{vm} are managed in files\ \cite{gitops-kubevirt}.

\subsection{Comparison with Traditional Approaches}
So far, KubeVirt has been presented as a silver bullet for \Gls{vm} management. However, it is important to note that KubeVirt does not represent a new virtualization technology, but rather a unifying technology. Behind the hood, KubeVirt uses \Gls{qemu} and \Gls{kvm} to create the underlying \Gls{vm}, which is similar to any other traditional \Gls{vm} management system. This means that KubeVirt should in theory provide the same level of isolation and performance as any other management system. However, KubeVirt does fall short in that the \Gls{vm} is encapsulated inside a Pod, as this could lead to a performance overhead. Another aspect to highlight is that KubeVirt itself is not directly comparable to existing \Gls{vm} management systems, but rather forms a fully-fledged system when evaluated together with Kubernetes. For example, KubeVirt's implementation does not implement its scheduler or storage solution and instead relies on Kubernetes implementation and integrations, while OpenStack has its own scheduler as well wide support for different storage solutions including data warehousing\ \cite{kubevirt-tech-comparison, kubevirt-architecture, kubevirt-network-plugin, kubevirt-disks}.

\section{Related Work}
\label{literature-study}
\subsection{Current challenges}
The \Gls{cncf} presents in a report some of the current challenges of managing \Gls{vm} using traditional approaches, such as an \Gls{iaas} platform\ \cite{kubevirt-cncf}. They argue that a platform such as OpenStack is overly complex and difficult to manage for an organization that only wants to manage \Glspl{vm}. They conclude by suggesting KubeVirt with Kubernetes as a simpler alternative to \Gls{vm} management.

Moreover, the \Gls{cncf} report also mentions the potential benefits of a unified platform for both \Glspl{vm} and containers. They argue that a unified platform reduces the required skillset and management burden for personnel, as well as reducing the architectural complexity of the system. These benefits are supported in a report by Kubacki\ \cite{kubevirt-storware}. He argues that managing both a traditional \Gls{vm} management system and a Kubernetes could lead to unnecessary costs for an organization. Additionally, he argues that by allowing \Glspl{vm} and containers to exist in a single platform \Gls{vm}-only organizations could potentially be driven toward adopting containers for suitable workloads. 

\subsection{KubeVirt challenges}
While KubeVirt is presented as a solution to many challenges related to managing \Glspl{vm}, it is important to consider the challenges that come with KubeVirt. A study by Trakulmaiphol and Piromsopa presents some shortcomings when moving into a KubeVirt-based system and found some issues related to how networking\ \cite{kubevirt-migrate-issues}. One of the issues was related to how IP address management differs in Kubernetes compared to traditional \Gls{vm} management systems. For instance, static IP addresses were difficult to assign and manage. In the study, they developed a framework to solve some of the issues on the application level, such as DNS queries, and provided a set of guidelines related to KubeVirt's unique features including the Multus Container Network Interface. 

A common practice when managing Kubernetes clusters is to deploy Kubernetes nodes on \Glspl{vm} created by a \Gls{vm} management system. This means that when using KubeVirt, \Glspl{vm} are created inside other \Glspl{vm} leading to nested virtualization. Ye et al.\ argue in a study that nested virtualization raises security concerns since it increases the size of the \Gls{tcb}\ \cite{free-turtles}. A larger \Gls{tcb} means that the size of the system's attack vector increases, which in turn leads to an increased risk for vulnerabilities. This study is particularly relevant as it highlights a potential limitation if KubeVirt is simply seen as another Kubernetes resource and not as a \Gls{vm} management system. 

\subsection{Usability of Kubernetes and KubeVirt}
Using Kubernetes as a \Gls{vm} management system is a relatively new concept and there is limited research on the usability of KubeVirt. However, there exist several studies that have evaluated the usability of Kubernetes, which is equally relevant since KubeVirt is built on top of Kubernetes.

Kubernetes \Gls{api} is inherently declarative, meaning it uses a configuration to describe a desired state it should be in. A study by Sonninen investigates many of the possible benefits of such an approach\ \cite{declarative-benefits}. The study found that a declarative approach enables a system to use GitOps, which is a way to manage infrastructure using files in storage backed by Git. This simplifies resource management through its built-in version control and audit, as well as rollbacks (given that it is supported by the resource). Two studies exemplify these benefits when presenting their case study on when CERN migrated an internal project from a cluster of \Gls{vm} to Kubernetes. The first study conducted by Imran et al.\ describes how migrating to Kubernetes improved the overall system management compared to using a \Gls{vm} cluster\ \cite{k8s-cern}. While many of their experienced benefits came from adopting containers instead of \Glspl{vm}, the study still suggests Kubernetes itself provided many benefits, such as reduced operational costs. The other study conducted by Badran presents the practical benefits they experienced by using Kubernetes and is more focused on the adoption of a declarative approach for system management\ \cite{k8s-cern-techreport}. It describes how using Kubernetes instead of a \Gls{vm} cluster allowed them to create reusable configurations that enabled their entire infrastructure to be managed as code. 

The benefits of moving into Kubernetes are further exemplified in a study by Gelley\ \cite{k8s-cloud-foundry}. It presents a case study where an organization moved its application from a \Gls{paas} provided by Cloud Foundry into Kubernetes. The study not only describes how Kubernetes provides more flexible configuration and deployment compared to the previous \Gls{paas} but also how the organization was able to make use of the tools Kubernetes provides to enable auto-scaling and monitoring. Moreover, the study concludes that by moving into Kubernetes, possible future migration to other platforms was made simpler due to how widely adopted Kubernetes is. 

User experience in Kubernetes was further investigated in a study by Packard et al.\ that collected several projects that moved into Kubernetes\ \cite{k8s-user-experience}. The study covered many aspects, including administration such as managing persistent storage and quotas, but also developer experience such as deploying applications. The study found that system administrators in general face a relatively steep learning curve when using Kubernetes since they need to learn many of the concepts that make up an application. Interestingly, it also found that many users without prior knowledge of Kubernetes could perform basic tasks without any major issues, such as deploying an application. This finding is especially useful when evaluating KubeVirt since the ability to deploy a built-in resource in Kubernetes is similar, if not identical, to deploying a KubeVirt-based resource. Another study that investigated the user experience of Kubernetes was conducted by Boronin\ \cite{k8s-migration-king}. The study presents a case study where an organization migrated from a monolithic system design to a more cloud-native design. The study is relevant as it is not only concerned with the technical aspects of Kubernetes but also considers the organizational impact of migrating to Kubernetes. For instance, transferring skill sets among personnel and the impact on the organization's culture.

\subsection{Performance Evaluations}
Regarding performance, several studies have evaluated popular systems that can manage \Glspl{vm}, including open-source alternatives such as OpenStack and CloudStack, or proprietary systems such as Amazon EC2. 

A study by Kang et al.\ compared the performance of OpenStack and Amazon EC2 using a wide range of configurations and analyzed diverse aspects based on different hypervisor, storage and networking configurations\ \cite{openstack-ec2-comparison}. Apart from the performance comparisons, the study also investigated performance impact when using more advanced features, such as Infiniband networking instead of Ethernet. The study showed mixed results, where OpenStack outperforms EC2 in some scenarios such as when using Infiniband networking, but EC2 tends to outperform OpenStack in parallel processing. As such, the study concluded that component-wise, each system has strengths. This study brings with it relevant perspectives as it highlights many of the aspects that are important to consider when evaluating the performance of \Gls{vm} management systems.

Another study by Mullerikkal and Sastri compared OpenStack and CloudStack regarding implementation complexity, stability and performance\ \cite{cs-os-comparison-2009}. Compared to the previous study, these performance tests were designed to measure disk and network I/O, and not CPU or memory usage. The study found that OpenStack was significantly more complex than CloudStack, but that it was overall more stable. The study also found that OpenStack had a slightly but overall better performance than CloudStack. These claims are further supported by another study conducted by Paradowski et al.\ \cite{cs-os-comparison-2013} a few years later that also compared CloudStack and OpenStack. The study benchmarked the two systems in terms of practical tasks such as time to deploy a \Gls{vm} and resource utilization during the deployment. The study found that OpenStack was slightly faster in most operations and had a lower resource utilization. While the tests were designed to represent user interactions, their scale was relatively small and did not explore any limitations of either system. 

In contrast, a later study by Amaral benchmarked KubeVirt uses a setup with a large number of \Glspl{vm} relative to the size of the nodes, aiming to explore the limits of KubeVirt's control plane (scheduler etc.)\ \cite{ibm-kubevirt-scale}. The study found that, while KubeVirt was able to create all \Glspl{vm}, the control plane suffered a heavy load, indicating a possible large overhead when KubeVirt provisions a large number of \Glspl{vm}. This finding is important to consider when evaluating KubeVirt, as it indicates that KubeVirt may not yet be suitable for managing a large number of \Glspl{vm}.

In terms of network performance, a study conducted by Kolahdouzan investigated the networking traffic pattern in KubeVirt and OpenStack by performing networking performance tests to evaluate potential \Gls{vm} overhead\ \cite{kubevirt-openstack-network}. The study found that the network overhead was significantly larger for scenarios in which \Glspl{vm} reside on separate nodes. However, while the study found overhead for both KubeVirt and OpenStack, it found that OpenStack had a considerably larger overhead. While this project will not focus on networking performance, it is important to consider the findings of this study when evaluating KubeVirt.

So far, only related work that evaluated higher-level systems has been presented. However, several studies have evaluated the performance of hypervisors. A study by Hwang et al.\ evaluated the performance of a range of hypervisors including \Gls{kvm}, Hyper-V and Xen\ \cite{hypervisor-comparion-2010}. The study did not manage to draw any conclusions on which hypervisor was most performant and instead concluded that it depended on the specific use case. This conclusion is interesting as it suggests that the performance of a hypervisor could be an obsolete factor when deciding which one to use. Either way, reviewing hypervisors is important when evaluating KubeVirt, as KubeVirt uses \Gls{kvm} as its hypervisor. 

\chapter{Performance Analysis}
As described in the literature study, Paradowski et al.\ presented a set of benchmarks used to evaluate CloudStack and OpenStack\ \cite{cs-os-comparison-2013}. This comparison was used as a foundational basis for the performance analysis in this project, employing many of its approaches for evaluating the performance of a \Gls{vm} management system. In this project, however, OpenNebula was compared with KubeVirt. Choosing OpenNebula as the second system was based on its popularity and the fact it is open-source, but also to ensure that the tests were relevant to Net Insight's current infrastructure. 

\section{Setup}
The testing process was conducted on Azure using a group of \Glspl{vm} that acted as hosting nodes. This approach was chosen to ensure that the tests are not affected by the underlying hardware and to facilitate reproducibility. 

A set of scripts was created to automate the entire testing process. The scripts automated node creation in Azure, as well as installation and configuration of the \Gls{vm} management system on the nodes. Once the tests were finished, the results were collected by the scripts and the nodes were deleted in Azure. The scripts were thus used to create a complete setup for each system in a consistent manner using a single command.

Payload-wise, this project used a similar approach to the study by Paradowski et al.\ with predefined types for \Glspl{vm} with CPU ranging from 1 to 4 CPU cores, and RAM ranging from 128 MB to 4 GB (see Table\ \ref{table:vm-specs} for a detailed overview). Moreover, \Glspl{vm} used the lightweight Cirros Linux image, since a lightweight \Gls{os} image is less likely to introduce noise in measurements\ \cite{linux-boot-time}. This also meant a \Gls{vm} disk could be smaller, which allowed for more \Glspl{vm} to be created per node and for tests to run quicker since a lightweight image is usually faster to boot than a larger image.

Each test case used a setup of three Azure nodes, one management node and two worker nodes. Having more than one worker node was crucial for certain tests to be able to run. For instance, a live migration test requires at least two worker nodes since it involves moving \Glspl{vm} between nodes. The Azure nodes were configured large enough to facilitate a broad range of tests, including tests with many small \Glspl{vm} and tests with a few large \Glspl{vm}. Azure offered many different \Gls{vm} types, many of which were suitable in terms of CPU and memory specification. However, only a few offered virtualization support, which was required when running nested \Glspl{vm}. This left the Azure Dv3-series as the most suitable \Gls{vm} type and, specifically, the D4v3 \Gls{vm} type that has 4 CPU cores and 16 GB of memory\ \cite{azure-dv3-series}. The price of the D4v3 \Gls{vm} type was 0.2 USD per hour, which was deemed acceptable for the project\ \cite{azure-pricing}. \section{Building a Framework}
Evaluating the performance of a \Gls{vm} management system is a complex task that requires a structured approach due to its many components and multi-node setup. For this reason, a framework to form standardized testing environments was created. The framework was designed to be \Gls{vm} management system agnostic, meaning that it can be used to evaluate any \Gls{vm} management system as long as the system supports the set of features defined in the framework. Consequently, this means that the framework can be extended to evaluate any other \Gls{vm} management system, such as the previously mentioned OpenStack and CloudStack, and not only OpenNebula and KubeVirt. However, it is important to note that the framework was designed to evaluate \Gls{vm} management systems that can be hosted on-premise since it quantifies the performance of a \Gls{vm} management system by measuring the utilization of nodes hosting the system. As such, the framework does not consider cloud-based \Gls{vm} management systems, such as Amazon EC2, Google Compute Engine or Azure Virtual Machines.

\subsection{Design}
\label{sec:framework-design}
The framework was designed to be modular and loosely coupled with external libraries. This design choice was made to ensure that the framework can be easily extended and modified and that it does not rely on external dependencies. To interact with a \Gls{vm} management system an interface was created that defines the methods that a \Gls{vm} management system must implement to be compatible. It defines methods for creating and deleting \Glspl{vm}, migrating \Glspl{vm} between nodes and scaling up and down the number of worker nodes, as well as methods for installing control and worker nodes. Each \Gls{vm} management system implements these methods in their respective packages. The full definition of the interface can be found in the appendix\ \ref{appendix:vmm-interface}. To host each \Gls{vm} management system, the framework created a set of nodes in Azure, which were completely installed and configured before any performance analysis was conducted to form a fair starting point for every \Gls{vm} management system. However, any dependencies required by the \Gls{vm} management system were intentionally left to the installation, as unnecessary dependencies could introduce noise in the measurements. 

Metrics were collected by running a daemon on each Azure node that collects utilization metrics, including CPU, memory and disk utilization. The daemon operated by fetching metrics every second and storing them in a JSON file locally. However, to only capture metrics when a test was running, the daemon only started collecting metrics when a test was initiated and stopped when the test was finished. This was controlled by placing an empty file called \textit{run} where the daemon was running. The daemon checked for the file's existence periodically and started collecting metrics when it noticed the file was present. As such, each node timestamped its metrics locally, which meant that the timestamp of the data points could vary between the nodes. However, this distributed approach was chosen to reduce the number of network requests to a minimum and to ensure that no skew was introduced in the measurements due to network latency. 

The utilization metrics were collected by a set of tools that were installed on each Azure node. Since the tools were required to be installed on every node, they were installed directly after creation. Table \ref{table:metric-tools} shows an overview of the tools used by the daemon. Another important aspect considered was asynchronous execution. Each of the tools used by the daemon was executed in parallel to minimize the time it took to collect the metrics for a given timestamp, essentially allowing shorter intervals between each data point. The structure of the output varied between the tools and was parsed using built-in tools in Linux such as \textit{awk} and \textit{sed} before being stored in a JSON file. The final output file was fetched over SSH using \textit{scp} and parsed by the framework to extract the metrics. The daemon was written in Bash to ensure that it could run on any Linux distribution and did not require any external dependencies. The code for the daemon can be found in the appendix\ \ref{appendix:metrics-collector} and the framework design is visualized in Figure\ \ref{fig:framework-design}.

The workflow of the framework was split into four phases to form a structured approach that allows for modularity and caching. Each phase was responsible for a specific part of the workflow and, in most cases, independent of each other. For instance, the installation phase can be omitted if the nodes are already installed and configured from a previous run. However, if the teardown phase was executed in one run, the next run after it must start with the setup phase. The design choice of dividing the workflow into phases was made to allow for more efficient testing, both when developing the scripts and when executing them. This was especially important considering that each test came with a cost in Azure. The entire workflow is illustrated in Figure \ref{fig:framework-workflow} and is described below:
\begin{enumerate}
    \item \textbf{Setup}: The setup phase is responsible for creating the nodes and networking in Azure. It is also responsible for installing base packages that every node requires, such as \textit{sysstat} to collect utilization metrics and \textit{jq} to parse JSON output.
    \item \textbf{Installation}: The installation phase is responsible for installing and configuring the \Gls{vm} management system on the Azure nodes. Each \Gls{vm} management system implements an installation method that is responsible for installing and configuring the system with one control node and two worker nodes such that it can run a \Gls{vm}. The installation method also installs several standby workers that will be used when scaling up and down the number of worker nodes. By having standby workers, the framework can independently measure the cost of scaling up and down without the cost of downloading dependencies. While larger dependencies can be seen as a metric on their own, it is important to note that production environments usually generate \Gls{os} images with prepared dependencies, rendering the cost of downloading dependencies irrelevant.
    \item \textbf{Benchmark}: The benchmark phase is responsible for running the performance tests. The tests are described in\ \ref{table:performance-tests} and are designed to evaluate the general performance of the \Gls{vm} management system. Each test is a combination of the methods the framework interface offers, such as creating \Glspl{vm} in different sizes, migrating \Glspl{vm} between nodes and scaling up and down the number of worker nodes. This phase generates and saves the results of the tests in a JSON file.
    \item \textbf{Teardown}: The teardown phase is responsible for deleting the Azure nodes and ensuring that no resources are left behind. Having this as a separate phase allows for tests to run sequentially without the time cost of creating and deleting nodes for each test. 
\end{enumerate}

\subsection{Implementation}
The framework was implemented in the programming language Go, a language popular for its performance and concurrency features. Concurrency was a particularly important factor as several tests involved running multiple operations in parallel, such as creating or migrating several \Glspl{vm} at the same time. As such, settling for Go reduced the risk of introducing noise in the measurements and could run each test more efficiently. 

The most important part of the framework was arguably to provide fair and reproducible comparisons. As described previously, each \Gls{vm} management system implemented an interface that defined methods for creating and deleting \Glspl{vm}. While this provided uniform operations from the framework's point of view, it also made the method's implementation crucial: KubeVirt's implementation of creating a \Gls{vm} should be as close to OpenNebula's implementation as possible to ensure that the comparisons are fair and that no unnecessary overhead are introduced. One example of this was that any network-related configuration had to be omitted since network-related metrics were not considered in the project. Another example was that the scheduling configuration had to be edited to ensure that \Glspl{vm} were created in as similar a manner as possible. However, all features that a \Gls{vm} management system offered and that were beneficial for the specific operation were considered. For instance, OpenNebula did not allow a live migration action to be performed on a \Gls{vm} that is not running, while KubeVirt allowed the operation to be requested despite the state of a \Gls{vm}.\ This allowed KubeVirt to omit the waiting time for a \Gls{vm} to be in the correct state before requesting a migration action.

\chapter{Usability Analysis}
To accurately measure the usability of KubeVirt, a set of tasks was crafted in cooperation with a system administrator at Net Insight. The tasks were designed to be representative of the tasks that a system administrator would perform when managing \Glspl{vm}. The tasks were designed to be performed using the Kubernetes \Gls{cli} tool \textit{kubectl} and KubeVirt's complementary \Gls{cli} tool \textit{virtctl}. This approach was chosen since KubeVirt did not directly offer a \Gls{gui}, and using any third-party solution could risk introducing biases in the results. 

\section{Setup}
To host an environment that was easily accessible to the participants, an AKS cluster was set up. Alternatives included installing a Kubernetes cluster locally using tools such as \textit{Minicube} or \textit{k3d}, but this approach was deemed less suitable since it could be time-consuming and require more resources from the participants. Moreover, a cloud-based solution ensured that the participants could access the environment anywhere reducing possible barriers to participation. AKS specifically was chosen since it included preinstalled storage classes that enabled plug-and-play persistent storage functionality. KubeVirt's snapshot-restore \Gls{api} required a CSI storage class, which made AKS a suitable choice. Similar to the performance analysis, the cluster included one control-plane node and two worker nodes to ensure that the participants could perform the tasks without any issues. Having two worker nodes was crucial for the live migration task, as it required moving a \Gls{vm} between two nodes. Separating the control node from the worker nodes was not necessary for the tests, but it is a common practice since it allows workers to be recreated without affecting the cluster's control components.

\section{Task Design}
The analysis was designed as a step-by-step guide that would take the participants through a set of tasks that were divided into three groups: basic usage, maintenance and debugging. The focus of the analysis was not to evaluate the participant's ability to perform the tasks or their Kubernetes knowledge but rather to expose them to KubeVirt and its workflows. As such, the tasks were designed to be as straightforward as possible, except for a debugging task that was designed to evaluate finding and resolving runtime issues. Moreover, the order of the tasks was designed to gradually increase in complexity to ensure a structured approach and allow the participants to familiarize themselves with KubeVirt. Below is a description of each group of tasks:

\begin{itemize}
    \item\textbf{Basic usage}\\
    This task was designed to act as a starting point for the usability analysis. The participants were asked to create a single \Gls{vm} configured with a \textit{containerDisk}, connect to it using SSH and VNC and then delete it. The purpose of this task was to evaluate how well KubeVirt supports the most basic operations and whether the participants deem the process intuitive. 

    \item\textbf{Maintenance}\\
    The second task was designed to evaluate how well KubeVirt supports recurring tasks that a system administrator could face when managing \Glspl{vm} and the platform hosting them. The analysis divided this task into two parts: live migration and snapshotting. The participants were first asked to create a single \Gls{vm} and start it on a specific host, then migrate the \Gls{vm} to another host without any downtime. A lightweight \Gls{vm} configured with a \textit{containerDisk} was chosen for this task to keep the focus on the live migration operation. The second part of the task was to create a snapshot of a \Gls{vm} and then restore it. A \Gls{vm} configured with a \textit{DataVolume} disk was chosen for this task since a container disk cannot be snapshotted. Both live migration and snapshotting were included in the usability analysis since they are inherently different in KubeVirt compared to traditional \Gls{vm} management systems due to the declarative nature of Kubernetes, see \ref{sec:kubevirt}. 

    \item\textbf{Debugging}\\
    The third task was designed to evaluate how well KubeVirt supports debugging and how intuitive the process of resolving runtime issues is. While KubeVirt does enforce correct specification upon creation with \Gls{api} validation, see \ref{sec:kubevirt}, it is still possible that a \Gls{vm} does not start if the misconfiguration affects the runtime. For this reason, the participants were presented with a \Gls{vm} that was not accessible due to a cloud-init misconfiguration that prevents SSH access to the \Gls{vm} by having invalid password syntax and missing authorized SSH public keys. This means that the participants could solve the issue either by resolving the password formatting issue or supplying a valid SSH public key. The participants were presented with a basic problem statement and were encouraged to use the official documentation if needed. For this task, a lightweight \Gls{vm} configured with a containerDisk was used to keep the focus on the debugging operation.
\end{itemize}

After each group of tasks was finished, the participants were asked to reflect on the experience and provide feedback on the tasks. The participants were encouraged to use a free-form approach but could use a set of open-ended questions as a reference. 

\section{Conducting the Usability Analysis}
The usability analysis was performed together with a group of system administrators at Net Insight. The participants were presented with a set of tasks as described in the previous section. Each participant was given a document containing all the tasks as well as an introduction to KubeVirt and each of the operations they would perform. For each task, the participant was encouraged to use the official KubeVirt documentation to aid them in completing the tasks. Together with the document containing the tasks, each participant was given a \textit{kubeconfig} and the name for their Kubernetes namespace in which they would perform their tasks. They were given separate namespaces to ensure that none of the tasks would interfere with the results of the other participants. Once finished, the reflection and feedback were collected from each participant. They were given two weeks to complete the tasks, and while this was a relatively long time, it was chosen to ensure that the participants were given enough time to potentially explore features in KubeVirt that were not part of the given tasks. The entire usability analysis can be found in the appendix\ \ref{appendix:usability-tasks}.

The participants found the process of creating a \Gls{vm} in KubeVirt to be mostly straightforward.

However, participants expressed some confusion when connecting to the \Gls{vm} using SSH, as it was not clear which SSH public key was used to connect to the \Gls{vm}, and that conflicts with previously created \Glspl{vm} could occur. \chapter{Discussion}
The research questions involved investigating potential efficiencies that could be gained by integrating \Glspl{vm} in Kubernetes using KubeVirt. This project has evaluated KubeVirt using two different approaches: a performance analysis and a usability analysis. The discussion aims to explore deeper into the results of the project to provide insight into how feasible KubeVirt is as a \Gls{vm} management system.

\section{Limited Features and Result Intepretation}
The goal of the performance analysis was to find factual and provable claims regarding the performance of a \Gls{vm} management system. This included creating a framework that could be used to craft environments that were both reusable and fair among the systems being evaluated. The framework also provided a common interface for which an implementation was created per \Gls{vm} management system. By definition, such a system is only as good as the implementations are, which is why the implementations were designed to be as similar as possible to ensure a fair comparison. However, certain features and settings were not able to be tuned in the same way, such as the scheduling configuration where OpenNebula provides only a single option for scheduling: the number of \Glspl{vm} that can be dispatched concurrently. Not being able to accurately configure the systems identically in every aspect leaves the interpretation of the results open for discussion. This project has chosen the approach where if the system does not provide a specific feature to be tuned, the system is considered to be as optimized as possible. This approach was chosen as it favors the \Gls{vm} management system that provides the most features and flexibility, and as such, the results should be interpreted with this in mind. Having this aspect in mind is particularly important when interpreting the \textit{Create Many Type-1 \Glspl{vm}} tests, see figures\ \ref{fig:create-many-cpu},\ \ref{fig:create-many-memory},\ \ref{fig:create-many-disk} and\ \ref{fig:create-many-duration}. KubeVirt, using the scheduler built into Kubernetes, was able to create 20 type-1 \Glspl{vm} in 67 seconds, while it took OpenNebula 192 seconds on average. However, the results also suggest that KubeVirt is more resource-intensive, as the utilization of CPU, memory and disk was significantly higher than OpenNebula, which can viewed as a trade-off between the duration to complete a task and the resources required to complete it. This project has chosen the duration to be the more important metric, as it indicates that the system is capable of handling a burst of requests efficiently and using its allocated resources to its fullest potential. 

It is not always the case that KubeVirt's larger resource utilization is short-lived compared to OpenNebula, where some results suggest that KubeVirt has a significantly larger disk utilization than OpenNeubla for a similar period. This is particularly evident in figure\ \ref{fig:scale-cluster-disk} where KubeVirt and OpenNebula follow a similar characteristic when scaling the cluster but KubeVirt disk utilization is significantly more intense. For this specific test, the result suggests that KubeVirt does not provide a trade-off between the duration to complete the task and the resources required to complete it. This could be due to the way KubeVirt implements adding additional nodes to the cluster, which could be a resource-intensive operation. Another explanation could be that the specific Kubernetes distribution used in the tests is not optimized for scaling the cluster. However, exploring solutions for this is outside the scope of this project.

The final test used in this project included scaling the cluster with \Glspl{vm} running on each node. It involved scaling the cluster from 2 worker nodes up to 10 worker nodes, then creating a type-1 \Gls{vm} on each worker node before scaling it down to 2 worker nodes. This caused the \Gls{vm} management system to migrate the \Glspl{vm} to available nodes. This test posed a challenge for the scheduler of both systems since a \Gls{vm} might be scheduled to the wrong host if the scale-down operation was not handled immediately. The test was also an extension to the \textit{Scale Cluster} test that involved the same steps but without the \Glspl{vm}. The result shows that OpenNebula was not able to complete the test due to some \Glspl{vm} not being migrated to the correct host, and instead remained on the host that was requested to be removed. Since the test was designed to disconnect the worker and then wait for each host to be empty, the OpenNebula implementation hung indefinitely. A possible explanation for this could be that OpenNebula attempted to move a \Gls{vm} from a host that disconnected to another host that was just about to disconnect, but hadn't yet, and could cause the receiving host to end up in a bad state. Another explanation could be that OpenNebula requires additional control plane nodes to handle such bursts for migration operation since the test environment only provided a single control plane node to reduce the cost of the test since the nodes were hosted in Azure. However, this is purely speculative and exploring the cause of the issue is outside the scope of this project. The issue of \Glspl{vm} stuck on hosts could be resolved by manual intervention, but since this was deemed unfair to the other system, OpenNebula was disqualified from the scale-with-\Glspl{vm} test. As such, the result for scaling with \Glspl{vm} was omitted for OpenNebula and only KubeVirt was evaluated for this test.

\section{Kubernetes Overhead and Induced Load}
The results of the performance analysis indicate that both KubeVirt and OpenNebula have a consistent overhead and that it differs between the systems. Having a larger overhead implies the system has more ongoing computational work in the background while not performing a requested operation. This is particularly evident in the tests creating different \Gls{vm}-types, see figures\ \ref{fig:create-tiny-kubevirt},\ \ref{fig:create-tiny-opennebula},\ \ref{fig:create-large-kubevirt} and\ \ref{fig:create-large-opennebula}. KubeVirt CPU utilization hovers around 4\% with a memory utilization of just below 12\%. OpenNebula, on the other hand, has a CPU utilization just above 2\% and a RAM utilization of 8\%. This suggests that OpenNebula has a lower required overhead to maintain the system compared to KubeVirt. The overhead is also evident in the tests including live migration, see\ \ref{fig:migrate-many-memory}. OpenNebula starts the test utilizing just above 8\% of the available memory while KubeVirt uses over 13\%. However, while this would suggest that KubeVirt is less resource-efficient than OpenNebula, it is important to first consider the surrounding factors. Firstly, since
Kubernetes was originally designed to manage containers, it has a different set of requirements compared to a traditional \Gls{vm} management system such as OpenNebula. Certain operations enabling container orchestration might not be necessary for a \Gls{vm} management system, which could manifest as an overhead. Secondly, the overhead shown in the results is not necessarily connected to KubeVirts implementation but rather Kubernetes itself, which could in turn be tied to the specific Kubernetes distribution used in the tests. However, exploring possible optimizations to reduce such overhead or the source of the overhead is outside the scope of this project. Lastly, as described in\ \ref{sec:kubevirt-technical-overview}, KubeVirt achieves \Gls{vm} hosting in Kubernetes by using custom resource and controllers that wraps a \Gls{vm} instance inside a Pod. While this approach allows for seamless integration of \Glspl{vm} in Kubernetes, it also introduces an unavoidable overhead. In conclusion, the overhead is a trade-off for the flexibility and features that KubeVirt provides, and the results suggest that KubeVirt is more resource-intensive than OpenNebula when not performing operations.

The flip side of this is that many of the results suggest the opposite regarding induced load. Induced load is the added load on the system when performing a specific operation and is calculated as the difference between the total utilization and the base utilization. This can be difficult to interpret since a large induced load over a short period is not necessarily worse than a small induced load over a longer period. However, this project has chosen a short-lived burst of utilization to be favorable over a longer period of low to medium utilization since it suggests that the system is capable of using its allocated resources to its fullest extent. The results suggest that KubeVirt has a lower induced load in general than OpenNebula, where in many cases the induced load is more than double for OpenNebula compared to KubeVirt. This is particularly evident in figure \ref{fig:create-many-cpu} where OpenNebula both take twice as long to complete the test and have twice as large induced load from the operation, around 6\% compared to KubeVirt's 3\%. The difference in induced load is further illustrated in figure\ \ref{fig:migrate-many-cpu}. Not only does it show that OpenNebula reaches an induced load of 76\% while KubeVirt reaches just above 50\%, but also that induced load is more prominent in the total utilization for both systems. This suggests that induced load is a more significant factor when evaluating the performance of \Gls{vm} management systems and, as such, should be considered when interpreting the results. Figure\ \ref{fig:migrate-many-memory-induced} further illustrates this point where for OpenNebula, the induced load is a significant part of the total utilization. Having a smaller induced load suggests that the system is better at scale while having a smaller base overhead suggests that the system is more efficient when the system is idle or performing few operations. However, whether this is true for all operations a \Gls{vm} management system offers remains to be investigated.

\section{Jagged Disk Utilization}
Throughout the performance analysis results, there is a recurring and notable pattern specifically tied to disk utilization. The disk utilization graphs are jagged and, in many cases, do not follow the expected pattern of the specific operation performed in the test. This is particularly evident in the \textit{Create \Gls{vm} Type 4} tests, see figures\ \ref{fig:create-large-kubevirt} and\ \ref{fig:create-large-opennebula}. The disk utilization for both Kubevirt and OpenNebula is initially larger than expected and does not return to the same state after the test is finished. Since the tests were run multiple times in no particular order, this likely means that noise was introduced in the measurement that caused each test to generate data points in different ranges. When the induced disk utilization was low for a particular test, background disk utilization from sources such as \Gls{os}-level operations, was more prominent in the results. By then applying a rolling mean smoothing function to the dataset, the noise was reduced, but due to the variability, some jagged features remained. This is further supported by the fact that the disk utilization is less jagged for tests with a larger expected disk utilization, such as the \textit{Create Many Type-1 \Glspl{vm}} tests since these tests generate a larger induced load. While the jagged disk utilization is not a problem in itself, it is important to consider when interpreting the results, especially those where disk utilization is small, as it could introduce bias unrelated to the operation performed.

\section{Mixed Opinions Using KubeVirt}
The results of the usability analysis broadly indicate that the participants experienced KubeVirt differently. Some participants found KubeVirt intuitive and acknowledged many of the benefits discussed throughout this report, while other participants made strong claims against the technology as a whole. To understand this discrepancy, it is important to consider what the positive or negative feedback targeted specifically to understand if the opinions were contradictory or if they were based on different aspects of the technology. The positive feedback was mostly tied to the idea of using a well-known \Gls{cli} tool, \textit{kubectl}, and using declarative configuration files to define the \Glspl{vm}. This formed a foundation where the participants were comfortable and appeared to find similarities with their existing knowledge of Kubernetes and, as such, basic tasks received well. Mixed opinions appeared once the tasks became more complex. For instance, the live migration task was found, contradictorily, to be both intuitive and confusing. This could be due to the different workflow KubeVirt presents compared to OpenNebula, where KubeVirt uses a declarative approach to define the live migration operation, while OpenNebula uses an imperative action-based approach. This is supported by the feedback from participants 1 and 2, where Participant 2 found the process of creating a resource for the live migration to be straightforward, while Participant 1 expressed confusion and argued that the process was unclear regarding clean-up and when the operation was completed. Structuring a live migration operation as a resource, where the action is performed once the resource is created is a common pattern in Kubernetes, but the results of the usability analysis suggest this approach could be confusing.

Other negative feedback was more concerned with exploring KubeVirt outside the scope of the given tasks. This is a particularly prevalent issue for one participant who deemed KubeVirt's official documentation to be lacking and missing practical examples of how to get started. Since exploring documentation was not part of the usability analysis, and the participants were not given any guidance on how to find the necessary information, it is difficult to determine if the participants were unable to find the information due to the lack of documentation or guidance. Either way, having issues with finding correct documentation is a significant problem for a technology that involves many user interactions and operations.

The mixed responses in the usability analysis indicate both possible biases among the participants regarding previous knowledge and experience with Kubernetes, but also that KubeVirt could be a controversial approach to managing resources. Kubernetes is known for its steep learning curve due to its declarative approach to resource management and verbose configuration files, and since KubeVirt follows the same idea, it is not surprising that the participants found the technology complex. Biases could thus have been caused by previous \Gls{vm} management systems being more approachable than KubeVirt, which could have affected the participant's opinions on KubeVirt. Exploring whether previous knowledge of a \Gls{vm} management system affects the perceived usability of KubeVirt is a possible future research direction but it is outside the scope of this project. 

\section{Concluding Research Questions}
The results from the performance analysis indicate both strengths and weaknesses for KubeVirt. As discussed in previous sections, KubeVirt inhibits a larger base utilization without performing any operation requested by a user compared to OpenNebula. However, the induced load is in general significantly lower for KubeVirt than for OpenNebula. This suggests that KubeVirt is more efficient when performing operations and thus could be more scalable than OpenNebula. Furthermore, the usability analysis indicates that KubeVirt in many cases benefits from the declarative approach inherited by Kubernetes. This section aims to answer the research questions presented in the introduction.

\subsubsection*{What performance, scalability, and operational efficiencies can be gained by integrating \Glspl{vm} in Kubernetes?}
KubeVirt provides an efficient way to manage \Glspl{vm} in Kubernetes with a lower induced load than OpenNebula. KubeVirt appears to be more scalable than OpenNebula, as evidenced by the lower induced load and the shorter duration to complete tasks. However, KubeVirt has a larger base utilization than OpenNebula, which argues for the consideration of each specific use case, such as whether KubeVirt is a viable option when each hosting node is relatively small in terms of resources. Operational efficiencies were acknowledged by some participants, but argued against by others. This suggests that KubeVirt could provide operational efficiencies, but that it requires a strong familiarity with Kubernetes and its concepts to be fully utilized. It also suggests that KubeVirt might lack in certain aspects regarding usability and that it requires further development. In summary, KubeVirt provides performance and scalability efficiencies when managing many \Glspl{vm}, but might be a more resource-intensive solution when idle or performing few operations. Operational efficiencies are gained when the system administrators are familiar with KubeVirt, but it could lead to confusion regarding the workflow and the technology itself.

\subsubsection*{What are the current challenges and limitations in managing \Glspl{vm} and containerized applications in separate environments?}
The literature study suggests that \Glspl{vm} and containers share many aspects including what they provide for a user, such as an environment that allows for computational workloads, and are thus often used in the same context. However, the literature study also described how both \Glspl{vm} and containers have their own use cases due to inherent properties, including the level of isolation they provide. Having \Glspl{vm} and containers in separate environments not only implies that the system administrators are required to have a broader skillset to manage both environments but also that benefits from one environment cannot be utilized in the other. This is particularly evident in the performance analysis where KubeVirt appears to benefit from the mature and efficient resource scheduler built into Kubernetes. Furthermore, Kubernetes provides a vast ecosystem of tools and resources that could be utilized by KubeVirt, including monitoring, logging and networking, as well as third-party integrations for storage, security and observability. In summary, this project not only presents the benefits of integrating \Glspl{vm} and containers into a unified environment but also the benefits of integrating \Glspl{vm} into Kubernetes. 

\subsubsection*{How can system administration be simplified by running \Glspl{vm} in Kubernetes using KubeVirt?}
The usability analysis highlights several aspects of how a system administrator can harness the benefits of moving into a more declarative approach to managing resources. This is particularly evident by one participant who found that the configuration files were easy to debug and "googleable" due to how contained the information was. This suggests that error handling and debugging could be simplified by being less time-consuming. Furthermore, the literature study presents other benefits of managing \Glspl{vm} in KubeVirt. Since the resources are managed declaratively it opens up the possibility of managing the resources using GitOps, which could provide a more secure and reliable way of managing resources since it removes the burden of version control and auditing from the system administrator. However, it is important to note that the results of usability analysis suggest that KubeVirt comes with a steep learning curve and that the technology could be perceived as complex. As such, the benefits found in the usability analysis and literature study might not be fully utilized if the system administrators are not familiar with Kubernetes and its concepts.

\chapter{Conclusions and Future Work}
This project has successfully evaluated Kubernetes as a \Gls{vm} management system through a literature study, by benchmarking the performance of KubeVirt against a traditional \Gls{vm} management system and by performing a usability analysis with system administrators. The literature study presented the benefits of adopting a Kubernetes approach to managing resources, including how the ecosystem of tools and services could utilized for a more efficient environment. The findings suggest that moving into Kubernetes enables integration with existing cloud-native tools such as GitOps. The results of the performance analysis suggest that the performance of both \Gls{vm} management systems are similar, but that KubeVirt is more efficient in large-scale environments due to its lower induced load. However, the overhead is not negligible and could be a limiting factor for certain use cases, especially those where the hosting nodes are relatively weak in terms of resources. Limitations among the \Gls{vm} management system, including available configuration and tuning options and the performance analysis framework itself were discussed, and the results were interpreted with these in mind. The usability analysis suggests that KubeVirt can provide operational efficiencies such as simplified debugging and resource management. However, it also found that the technology could be perceived as complex and require familiarity with Kubernetes, its concepts and design philosophy. Potential biases among the participants as well as flaws in the technology itself were discussed, including whether the previous \Gls{vm} management system used affects the usability of KubeVirt. 

Further work could involve exploring the source of the overhead discrepancy found in the performance analysis, including why OpenNebula inhibits a significantly lower baseline overhead than KubeVirt. It could also involve exploring if KubeVirt's apparent lower induced load is a result of the Kubernetes scheduler, the Kubernetes distribution or KubeVirt's implementation of \Gls{vm} management. Additionally, further work could explore the source of the potential biases found in the usability analysis and answer if the adoption of KubeVirt differs depending on the previous \Gls{vm} management systems used, as this project has only concluded that previous Kubernetes experience affects the usability of KubeVirt positively. Lastly, further work could investigate KubeVirt's applicability in environments with strict security requirements and whether Kubernetes offers the necessary tools and resources to meet these requirements, as this could further support the claims made in this project regarding the benefits of Kubernetes integration.

In summary, the project has shown that while OpenNebula outperforms KubeVirt in some scenarios due to its simplicity and low overhead, KubeVirt appears more scalable in larger-scale deployments. KubeVirt also provides operational efficiencies given that the system administrators are familiar with both the core concepts of Kubernetes and KubeVirt itself.

\section{Metrics Collector}
\label{appendix:metrics-collector}
\begin{lstlisting}[language=bash]
#!/bin/bash

interval=1
OUTPUT_FILE="./output.json"

cpu_tmp_file=$(mktemp)
ram_tmp_file=$(mktemp)
disk_tmp_file=$(mktemp)

while true; do
    if [ ! -f "./run" ]; then
        sleep $interval
        continue
    fi

    (mpstat -P ALL "$interval" 1 | awk '/^[0-9]/ {if (NR>1) print $3/100}' | jq -R -s -c 'split("\n")[1:-1]' | jq '[.[] | gsub(","; ".") | tonumber]' > "$cpu_tmp_file") &

    (free | awk '/Mem:/ {print $3/$2}' | sed 's/,/./' > "$ram_tmp_file") &

    (iostat -dx "$disk" 1 2 | awk '/Device/{flag=1; next} flag && /^[^ ]/ {util=$NF} END{print util/100}' | sed 's/,/./' > "$disk_tmp_file") &
    timestamp=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

    wait

    name=$(hostname)
    cpu_core_utilization=$(cat "$cpu_tmp_file")
    cpu_utilization_avg=$(echo "$cpu_core_utilization" | jq 'add/length')
    ram_utilization=$(cat "$ram_tmp_file")
    disk_utilization=$(cat "$disk_tmp_file")

    # Create the JSON object for the current datapoint
    datapoint=$(jq -n \
                    --argjson cpuCores "$cpu_core_utilization" \
                    --argjson cpuAverage "$cpu_utilization_avg" \
                    --arg name "$name" \
                    --arg ramUtilization "$ram_utilization" \
                    --arg diskUtilization "$disk_utilization" \
                    --arg timestamp "$timestamp" \
                    '{name: $name, cpu: {cores: $cpuCores, average: $cpuAverage}, ram: ($ramUtilization | tonumber), disk: ($diskUtilization | tonumber), timestamp: $timestamp}')

    if [ ! -f "$OUTPUT_FILE" ]; then
        echo "[]" > "$OUTPUT_FILE"
    fi

    jq --argjson newData "$datapoint" '. += [$newData]' "$OUTPUT_FILE" > tmp.$$.json && mv tmp.$$.json "$OUTPUT_FILE"

    sleep $interval
done
\end{lstlisting}

\section*{Introduction}
This is a usability analysis of KubeVirt that involves performing a set of tasks and then reflecting on the experience, where the goal is to evaluate how well KubeVirt works in practice. By participating in the usability analysis you provide valuable feedback and insight that can be used to improve KubeVirt. Your answers will be included as-is in the appendix of the project report. Upon request, the answers can be anonymized.

The following is a brief introduction to KubeVirt, its most relevant features and how it compares to previous \Gls{vm} management systems. If you are already familiar with KubeVirt and its machinery, you can skip this section.
\\~\\
KubeVirt is an extension to Kubernetes that allows for the creation of virtual machines (\Glspl{vm}) and is designed to provide a similar experience to managing any other resource in Kubernetes, such as Pods or Deployments. This means that \Glspl{vm} are defined in YAML files and integrate with the command line tool \textit{kubectl}. KubeVirt defines its resources as a \textit{Custom Resource Definition} (CRD) in Kubernetes, where the most prominent resources are \textit{VirtualMachine} and \textit{VirtualMachineInstance}. The relationship between them is comparable to that of a Deployment and a Pod, where the Deployment defines a desired state and a Pod represents the running state. See References below for more information. 

Certain \Gls{vm} operations are not possible using kubectl, such as accessing the \Gls{vm} using SSH or VNC. For this reason, KubeVirt provides a complementary \Gls{cli} tool called \textit{virtctl}.

Being built on top of Kubernetes allows the \Glspl{vm} to integrate with other resources, including the Service and PersistentVolume resources. For instance, a \Gls{vm} can be exposed to the network using a LoadBalancer Service or manage storage dynamically using a persistent storage provisioner. However, since a \Gls{vm} requires a disk image (such as a QCOW2 image), it means that a PersistentVolume must point to a folder with a disk image in it. For this reason, manually creating a PersistentVolume for a \Gls{vm} can be tedious. Instead, KubeVirt primarily provides two alternatives for managing disks that are presented below. See the References section for more information.
\begin{itemize}
    \item \textbf{containerDisk}: A disk that is created at every boot. It is not persistent and is similar to a container.
    \item \textbf{DataVolume}: A disk that is persistent and initialized when the \Gls{vm} is created. It uses PersistentVolumeClaim and a PersistentVolume under the hood where it injects a disk image. 
\end{itemize}

\section*{Basic usage (Task 1/3)}
The first task is about creating a \Gls{vm} using KubeVirt. You will need to create a \Gls{vm} and then access it using SSH or VNC. You can use a containerDisk for simplicity.

\subsection*{Manifests}
Below is a minimal manifest for creating the \Gls{vm} using a containerDisk. The credentials for the Cirros image are user \textbf{cirros} and password \textbf{gocubsgo}.
\begin{lstlisting}[language=yaml]
    apiVersion: kubevirt.io/v1
    kind: VirtualMachine
    metadata:
      name: my-vm
    spec:
      running: true
      template:
        spec:
          domain:
            cpu:
              cores: 1
            devices:
              disks:
              - name: containerdisk
                disk:
                  bus: virtio
              interfaces:
              - name: default
                masquerade: {}
            resources:
              requests:
                memory: 128M
          networks:
          - name: default
            pod: {}
          volumes:
          - name: containerdisk
            containerDisk:
              image: quay.io/kubevirt/cirros-container-disk-demo
\end{lstlisting}

\subsubsection*{Steps}
\begin{enumerate}
    \item Create a \Gls{vm}
\begin{lstlisting}
    kubectl apply -f my-vm.yaml
\end{lstlisting}
    \item Access the \Gls{vm} through SSH or VNC (credentials vary depending on the image since no CloudInit is used)
\begin{lstlisting}
    virtctl ssh cirros@my-vm
    virtctl vnc my-vm
\end{lstlisting}
    \item Stop, start and restart the \Gls{vm}
\begin{lstlisting}
    virtctl stop my-vm
    virtctl start my-vm
    virtctl restart my-vm
\end{lstlisting}
    \item Delete the \Gls{vm}
\begin{lstlisting}
    kubectl delete vm my-vm
\end{lstlisting}
\end{enumerate}

After you have finished the tasks, reflect on the experience. You may use the following questions as a guide, but feel free to elaborate with your own thoughts.

    \item How intuitive was the process of creating and deleting a \Gls{vm}?
    \item How straightforward was it to access the \Gls{vm} using SSH and VNC?
    \item Describe, if any, the issues you encountered during the task. Were the issues related to KubeVirt, documentation, or other?
\end{itemize}

\section*{Maintenance (Task 2/3)}
The second task is related to maintenance, where you will create a snapshot of a \Gls{vm} and then restore it. You will also perform a live migration on a \Gls{vm} to move it from one node to another without shutting it down. 

Both snapshots and live migrations are treated as a resource by KubeVirt, meaning that the operation will take place in the background after a VirtualMachineSnapshot resource or a VirtalMachineInstanceMigration resource is created. See the References section for more information.

A live migration resource in KubeVirt does not move resources itself and instead uses the Kubernetes scheduler to decide where to place the \Gls{vm}. This means that a LiveMigration resource only tells the scheduler that the \Gls{vm} can be moved but not where to move it. See the References section for more information. The Kubernetes scheduler is sophisticated and supports a wide range of scheduling strategies, but for this task, we can force a move between two nodes by using the \textit{spec.template.spec.nodeSelector} in the \Gls{vm} and editing the labels of the worker nodes in the cluster. To not interfere with other participants, you should create a unique label for you. 

A snapshot in KubeVirt works similarly to a snapshot in other \Gls{vm} management systems, where it captures the state of the \Gls{vm} at a specific point in time. However, the snapshot uses Kubernetes VolumeSnapshot, which is a Kubernetes \Gls{api} for taking snapshots of PersistentVolumes. This means that a \Gls{vm} with a containerDisk cannot be snapshotted. Instead, you should use a \Gls{vm} with a DataVolume. See the References section for more information.

You will be presented with a \Gls{vm} manifest which, if deployed, will result in a \Gls{vm} that is running but cannot be accessed through SSH.
